## 二、已验证的 10 个核心问题（按严重度排序）

### 🔴 致命级

**B1: FTS5 对中文完全失效**
```typescript
// hybrid.ts:24-28 — 已验证
const tokens = raw.match(/[A-Za-z0-9_]+/g)  // 只匹配英文 token！
```
- 中文查询 `"琳娜喜欢什么"` → `tokens=[]` → `return null` → FTS 完全跳过
- 混合搜索退化为纯向量搜索，BM25 通道对中文**完全无效**

**B2: 无零依赖本地快速文本搜索**
- 100% 依赖嵌入向量做语义检索
- 嵌入 provider 不可用/超时/配额耗尽时 → **检索完全瘫痪**
- 精确匹配（"API key"、"端口 3000"）不如简单 grep 准确
- `memory-keyword-search.ts` 是个极简回退（仅搜 memory/ 顶层 .md，不递归，±1 行上下文）

**B3: 每次查询必须等待远程嵌入**
```typescript
// manager.ts:289 — 已验证
const queryVec = await this.embedQueryWithTimeout(cleaned);  // 60s 超时!
```
- 每次搜索都调远程 embedding API，延迟 200-2000ms
- 简单关键词查询完全不必要

### 🟡 高优级

**B4: 记忆文件发现范围过窄**
```typescript
// internal.ts:63-86 — 已验证
// listMemoryFiles() 只扫描 MEMORY.md + memory/ 目录（仅 .md 文件）
if (!entry.name.endsWith(".md")) continue;  // 行58：.txt 被忽略
```
- 不索引 `characters/lina/memory/`、`characters/lina/assets/`
- 不索引 `workspace/` 任务产出
- 不索引 `.kiro/lessons-learned/`（163 个经验文件！）
- `.txt` 文件被忽略

**B5: 关键词搜索极简且原始**
```typescript
// memory-keyword-search.ts:33 — 已验证
const files = await fs.readdir(params.memoryDir);  // 不递归！只读顶层
```
- 只搜 `memory/` 顶层目录
- 每次重新读取所有文件（无缓存）
- 上下文只取 ±1 行
- 无分词、无同义词、无模糊匹配

**B6: 子任务执行无记忆上下文**
- `retrieveMemoryContext()` 有 `isQueueTask` 守卫
- 子任务执行时 LLM 完全"失忆"

**B7: 写入后索引不同步**
- LLM 写入 → chokidar 检测 → debounce 延迟 → sync
- 紧接着调用 memory_search → 大概率拿到旧索引

### 🟢 中优级

**B8: 无专用 memory_write 工具**
- LLM 只能用通用 write/edit 工具
- P89 只在用户明确说"写入记忆"时注入路径提示
- 无格式校验、无自动索引触发

**B9: Factory 类型断言**
```typescript
// factory.ts:38 — 已验证
const memoryCfg = (agentCfg as any).memory;  // 类型缺失
```

**B10: filters.ts 关键词完全硬编码**
- 87 个技术关键词 + 41 个角色关键词，全部硬编码
- 无法根据实际记忆内容适配

## 三、优化方案（四阶段递进）

### 阶段 1: 急修致命缺陷（1-2 天，即时收益）

> 目标：让中文搜索可用 + 本地快速检索兜底 + 扩展索引源

#### M1: 修复中文 FTS（B1）
- **文件**: `src/memory/hybrid.ts` → `buildFtsQuery()`
- **改动**: 增加中文字符级 tokenizer
  ```typescript
  // 新逻辑：中文按字/词切分，英文保持原逻辑
  const cjkTokens = raw.match(/[\u4e00-\u9fff\u3400-\u4dbf]+/g) ?? [];
  const alphaTokens = raw.match(/[A-Za-z0-9_]+/g) ?? [];
  // CJK: 按 bigram 切分（2字一组），避免单字噪音太大
  // 合并后构建 FTS5 MATCH 查询
  ```
- **预期效果**: BM25 通道恢复对中文的有效性，混合搜索质量大幅提升
- **风险**: 低（纯函数修改，不影响其他逻辑）

#### M2: 本地快速文本搜索通道（B2 + B3 + B5）
- **新建文件**: `src/memory/local-search.ts`（~200-300行）
- **核心能力**:
  - `localGrepSearch(query, dirs[], options)` — 类 ripgrep 的本地快速检索
  - 递归遍历指定目录，支持 `.md` + `.txt` + `.json` + `.jsonl`
  - 中文友好：按空格/标点分词 + 完整子串匹配
  - 上下文窗口：默认 ±5 行（可配置）
  - 结果排序：关键词命中率 × 位置权重（标题 > 正文）
  - 文件缓存：内存缓存文件内容（hash 校验，TTL 5分钟）
  - **零外部依赖**：纯 Node.js fs + 字符串操作，不依赖任何 API
- **集成点**:
  - `MemoryIndexManager.search()` 中增加第三路搜索通道
  - 混合排序从二路变三路：`vector + FTS/BM25 + localGrep`
  - 嵌入不可用时自动降级为 `FTS + localGrep` 双通道
- **预期效果**: 精确匹配能力大幅提升 + 离线兜底 + 延迟降到 <50ms

#### M3: 扩展索引源（B4）
- **文件**: `src/memory/internal.ts` → `listMemoryFiles()`
- **改动**: 新增可配置的额外扫描目录
  ```typescript
  // 新增参数：extraDirs?: string[]
  // 默认扫描：MEMORY.md + memory/ + characters/*/memory/ + .kiro/lessons-learned/
  // 支持 .txt 文件（移除 .md 限制）
  ```
- **同步修改**: `manager.ts` 的 `ensureWatcher()` 增加对新目录的 chokidar 监听
- **预期效果**: 角色记忆、经验教训自动进入索引，可搜索性大幅提升

#### M4: keywordSearch 增强（B5 回退通道加固）
- **文件**: `src/agents/tools/memory-keyword-search.ts`
- **改动**:
  - 递归搜索（复用 `walkDir` 或调用 M2 的 `localGrepSearch`）
  - 上下文扩大到 ±5 行
  - 支持 `.txt` + `.md`
  - 增加简单缓存（避免每次重新读取所有文件）

### 阶段 2: 智能查询路由 + 三路融合（3-5 天）

> 目标：根据查询特征自动选择最优检索策略

#### M5: 智能查询路由器
- **新建文件**: `src/memory/query-router.ts`（~150行）
- **路由规则**:
  | 查询特征 | 路由策略 | 原因 |
  |----------|----------|------|
  | 短查询（<10字）+ 精确关键词 | localGrep only | 快速精确 |
  | 中文查询 + 嵌入不可用 | FTS(bigram) + localGrep | 中文兜底 |
  | 长查询（>30字）| vector + FTS + localGrep | 语义理解 |
  | 混合查询 | 三路并行 | 最大召回 |
  | 嵌入 provider 不可用 | FTS + localGrep | 离线降级 |
- **超时策略**: localGrep 和 FTS 立即返回；vector 有 5s 超时；三路结果按时间窗口融合

#### M6: 三路混合排序
- **文件**: `src/memory/hybrid.ts` → `mergeHybridResults()`
- **改动**: 从二路 `(vector, keyword)` 扩展为三路 `(vector, keyword, localGrep)`
- **新增参数**: `grepWeight` 加权系数
- **融合策略**:
  - vector 命中 + localGrep 命中 → 双重加权（极高相关性）
  - 仅 localGrep 命中 → 精确匹配但语义未知
  - 仅 vector 命中 → 语义相关但文本不匹配

#### M7: 写入后即时索引（B7）
- **文件**: `src/memory/manager.ts`
- **改动**: 新增 `notifyFileChanged(absPath)` 公共方法
  - 立即触发该文件的增量索引（跳过 debounce）
  - 用于 `memory_write` 工具写入后主动调用
- **同步修改**: localGrep 缓存失效机制

## 六、技术设计要点

### 6.1 local-search.ts 核心设计

```typescript
// 接口定义
interface LocalSearchOptions {
  dirs: string[];                    // 搜索目录列表
  extensions?: string[];             // 文件扩展名（默认 [".md", ".txt"]）
  recursive?: boolean;               // 是否递归（默认 true）
  contextLines?: number;             // 上下文行数（默认 5）
  maxResults?: number;               // 最大结果数（默认 20）
  maxFileSize?: number;              // 最大文件大小（默认 1MB，跳过超大文件）
  cacheEnabled?: boolean;            // 是否启用文件缓存
  cacheTTLMs?: number;               // 缓存 TTL（默认 5min）
}

interface LocalSearchResult {
  path: string;                      // 文件相对路径
  absPath: string;                   // 绝对路径
  startLine: number;
  endLine: number;
  score: number;                     // 0-1 归一化分数
  snippet: string;                   // 上下文片段
  matchType: "exact" | "fuzzy";      // 匹配类型
}

// 搜索策略
// 1. 将查询按空格/标点分词
// 2. 对每个文件：
//    a. 逐行扫描，计算每行的关键词命中数
//    b. 命中行 ±contextLines 组成上下文窗口
//    c. 相邻窗口合并（避免重叠片段）
//    d. 标题行（# 开头）命中额外加权
// 3. 全局按 score 排序，截断到 maxResults
```

### 6.2 中文 FTS bigram 切分

```typescript
function buildFtsQuery(raw: string): string | null {
  // 英文：保持原有逻辑
  const alphaTokens = raw.match(/[A-Za-z0-9_]+/g)?.filter(Boolean) ?? [];
  
  // 中文：提取 CJK 连续字符串，按 bigram 切分
  const cjkMatches = raw.match(/[\u4e00-\u9fff\u3400-\u4dbf]{2,}/g) ?? [];
  const cjkBigrams: string[] = [];
  for (const segment of cjkMatches) {
    for (let i = 0; i < segment.length - 1; i++) {
      cjkBigrams.push(segment.substring(i, i + 2));
    }
    // 单独保留完整词（如果 >= 2 字）
    if (segment.length >= 2) cjkBigrams.push(segment);
  }
  
  const allTokens = [...alphaTokens, ...cjkBigrams];
  if (allTokens.length === 0) return null;
  
  // 用 OR 连接（中文 bigram 可能部分命中也有价值）
  const quoted = allTokens.map(t => `"${t.replaceAll('"', "")}"`);
  return quoted.join(" OR ");
}
```

## 八、与 Fast Context (Cascade grep) 的对照

你提到欣赏 Cascade 的 Fast Context（code_search / grep_search）能力。对照分析：

| 能力 | Cascade Fast Context | Clawdbot 记忆系统（现状）| 优化后目标 |
|------|---------------------|------------------------|-----------|
| 精确文本搜索 | ✅ ripgrep 内核 | ❌ 无 | ✅ M2 localGrepSearch |
| 中文搜索 | ✅ Unicode 支持 | ❌ FTS 中文失效 | ✅ M1 bigram FTS |
| 语义搜索 | ❌ 无 | ✅ 向量搜索 | ✅ 保持 |
| 搜索速度 | <100ms | 200-2000ms | <50ms (localGrep) |
| 离线可用 | ✅ 零依赖 | ❌ 依赖 embedding API | ✅ M2 兜底 |
| 递归目录 | ✅ | ❌ 只搜 memory/ | ✅ M3 扩展 |
| 文件类型 | 所有文本 | 仅 .md | ✅ M3 扩展 |

**M2 (local-search.ts) 的目标就是把类似 ripgrep 的快速本地检索能力内嵌到 Clawdbot 的记忆系统中**，作为向量搜索的并行通道和离线兜底。
