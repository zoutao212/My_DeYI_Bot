# 任务分解、展示和总结机制

## 概述

任务分解、展示和总结机制是 Clawdbot 的一个核心功能，旨在将复杂任务的执行过程从黑盒变为透明可控的白盒。通过自动拆解任务、可视化进度、智能处理失败和自我改进，让用户清楚地了解系统正在做什么、已经做了什么、以及下一步要做什么。

## 核心功能

### 1. 任务自动拆解

当你提交一个复杂任务时，系统会自动判断是否需要拆解，并将其分解成 2-8 个可执行的子任务。

**触发条件**：
- 任务描述长度超过 200 字符
- 包含多个动词（"创建"、"修改"、"测试"等）
- 涉及多个文件或模块
- 用户明确要求拆解

**示例**：

```
用户输入：
"创建任务分解和跟踪机制，包括数据模型、持久化层、任务分解器、进度跟踪器和失败处理器"

系统拆解：
T1: 定义 TaskBoard 数据模型
T2: 实现 TaskBoard 持久化层
T3: 实现 TaskBoard 渲染器
T4: 实现 TaskDecomposer 接口
T5: 实现 ProgressTracker 接口
T6: 实现 TaskExecutor 接口
T7: 实现 FailureHandler 接口
T8: 实现 Agent Orchestrator
```

### 2. 任务看板展示

任务看板以清晰的格式展示主任务、所有子任务、当前焦点和进度。

**任务看板包含**：
- **主任务**：标题、目标、状态、进度
- **子任务列表**：ID、标题、描述、状态、进度、依赖关系、产出
- **当前焦点**：正在执行的子任务、推理摘要、下一步行动
- **检查点**：时间戳、摘要、关键决策、未决问题
- **风险和阻塞**：描述、缓解措施
- **上下文锚点**：代码位置、命令

**查看任务看板**：

任务看板会自动保存到 `~/.clawdbot/tasks/{session_id}/` 目录：
- `TASK_BOARD.json`：机器可读的 JSON 格式
- `TASK_BOARD.md`：人类可读的 Markdown 格式

你可以在 IDE 中打开 `TASK_BOARD.md` 查看任务进度。

### 3. 进度跟踪和持久化

系统会实时跟踪任务进度并自动保存到磁盘，即使会话中断也能恢复任务状态。

**状态类型**：
- `pending`：待执行
- `active`：正在执行
- `completed`：已完成
- `blocked`：已阻塞
- `skipped`：已跳过

**持久化机制**：
- 每次状态变化后自动保存
- 使用原子写入确保数据一致性
- 同时保存 JSON 和 Markdown 格式

### 4. 失败处理和重试

当子任务失败时，系统会分析失败原因并提供恢复选项。

**失败分析**：
- 识别错误类型（语法错误、类型错误、文件不存在等）
- 分析根本原因
- 生成建议的修复方案

**恢复选项**：
- **重试**：重新执行失败的子任务
- **跳过**：跳过失败的子任务，继续执行下一个
- **修改任务**：修改子任务描述后重新执行
- **中止**：中止整个任务

### 5. 检查点和总结

系统会在关键节点自动创建检查点并总结进展。

**检查点包含**：
- 时间戳
- 本阶段结论摘要
- 已确认的关键决策
- 未决问题

**任务总结**：
- 所有子任务的产出
- 所有检查点的关键决策
- 经验教训和改进建议

### 6. 自我改进机制

系统会从任务执行中学习，识别可复用的模式并建议固化为规则或技能。

**模式识别**：
- 重复的操作序列（出现 3 次以上）
- 常见的依赖关系模式
- 标准的工作流程

**经验固化**：
- 建议将模式固化为规则
- 建议将操作序列固化为技能
- 调用 maintain-rules Power 自动创建

## 使用指南

### 启用任务分解模式

任务分解模式会在满足触发条件时自动启用。你也可以明确要求拆解：

```
"请拆解这个任务：创建任务分解和跟踪机制"
```

### 查看任务看板

任务看板会自动保存到磁盘。你可以：

1. **在 IDE 中查看**：打开 `~/.clawdbot/tasks/{session_id}/TASK_BOARD.md`
2. **通过命令查看**：`cat ~/.clawdbot/tasks/{session_id}/TASK_BOARD.md`
3. **在对话中查看**：说"显示任务看板"或"查看任务进度"

### 处理失败

当子任务失败时，系统会展示失败原因和恢复选项。你可以：

1. **重试**：如果失败是临时性的（例如网络超时）
2. **跳过**：如果子任务不是必需的
3. **修改任务**：如果子任务描述不准确
4. **中止**：如果问题无法解决

### 恢复任务

如果会话中断，你可以恢复任务：

```
"继续任务"
"恢复任务"
"继续上次的任务"
```

系统会：
1. 从磁盘加载最新的任务看板
2. 展示主任务、当前焦点和下一步行动
3. 根据下一步行动的明确性决定是直接执行还是请求澄清

### 固化经验

任务完成后，系统会分析任务执行过程并建议固化经验：

```
💡 自我改进建议:
   类型: rule
   标题: 固化规则: 标准软件开发流程
   描述: 将 "标准软件开发流程" 固化为规则，以便在未来的任务中自动应用
   模式: 标准软件开发流程
   步骤: 分析 -> 设计 -> 实现 -> 测试

是否固化这个经验？(y/n)
```

如果你同意，系统会调用 maintain-rules Power 自动创建规则或技能。

## 配置选项

你可以通过配置文件自定义任务分解行为：

```json
{
  "taskDecomposition": {
    "enabled": true,
    "enableConcurrentExecution": false,
    "enableAutoRetry": false,
    "maxRetries": 3,
    "minSubTasks": 2,
    "maxSubTasks": 8
  }
}
```

**配置说明**：
- `enabled`：是否启用任务分解（默认：true）
- `enableConcurrentExecution`：是否启用并发执行（默认：false）
- `enableAutoRetry`：是否自动重试失败的子任务（默认：false）
- `maxRetries`：最大重试次数（默认：3）
- `minSubTasks`：最少子任务数量（默认：2）
- `maxSubTasks`：最多子任务数量（默认：8）

## 最佳实践

### 1. 明确任务目标

在提交任务时，尽量明确任务的目标和范围：

✅ **好的任务描述**：
```
"创建任务分解和跟踪机制，包括：
1. 数据模型定义
2. 持久化层实现
3. 任务分解器实现
4. 进度跟踪器实现
5. 失败处理器实现"
```

❌ **不好的任务描述**：
```
"做一个任务管理的东西"
```

### 2. 及时查看任务看板

定期查看任务看板，了解任务进度和当前焦点：

```bash
# 查看任务看板
cat ~/.clawdbot/tasks/session_123/TASK_BOARD.md

# 或在 IDE 中打开
code ~/.clawdbot/tasks/session_123/TASK_BOARD.md
```

### 3. 合理处理失败

当子任务失败时，根据失败原因选择合适的恢复选项：

- **临时性错误**（网络超时、资源暂时不可用）→ 重试
- **配置错误**（路径错误、权限不足）→ 修改任务
- **非必需任务**（可选的优化、文档更新）→ 跳过
- **严重错误**（无法解决的依赖问题）→ 中止

### 4. 固化有价值的经验

当系统建议固化经验时，评估其价值：

- **高价值**：重复出现 3 次以上的模式、标准工作流程
- **中价值**：特定场景下的最佳实践
- **低价值**：一次性的操作、特定项目的配置

只固化高价值和中价值的经验，避免规则膨胀。

## 故障排除

### 任务看板文件不存在

**问题**：找不到 `TASK_BOARD.json` 或 `TASK_BOARD.md` 文件

**解决方案**：
1. 检查会话 ID 是否正确
2. 检查文件路径：`~/.clawdbot/tasks/{session_id}/`
3. 确认任务分解模式已启用

### 任务恢复失败

**问题**：说"继续任务"后系统无法恢复任务

**解决方案**：
1. 检查任务看板文件是否存在
2. 检查任务看板文件是否损坏（尝试手动打开 JSON 文件）
3. 如果文件损坏，从备份恢复或重新开始任务

### 子任务一直失败

**问题**：子任务重试多次后仍然失败

**解决方案**：
1. 查看失败原因和建议的修复方案
2. 检查是否是环境问题（依赖缺失、权限不足等）
3. 尝试修改子任务描述，提供更多上下文
4. 如果无法解决，选择"中止"并手动处理

## 技术细节

### 数据模型

任务看板的数据结构定义在 `src/agents/task-board/types.ts` 中：

```typescript
interface TaskBoard {
  sessionId: string;
  mainTask: MainTask;
  subTasks: SubTask[];
  currentFocus: CurrentFocus;
  checkpoints: Checkpoint[];
  risksAndBlocks: Risk[];
  contextAnchors: ContextAnchors;
  lastUpdated: string;
  version: string;
}
```

### 持久化机制

任务看板使用原子写入确保数据一致性：

1. 写入临时文件：`TASK_BOARD.json.tmp.{timestamp}.{random}`
2. 原子性地重命名临时文件为目标文件
3. 如果失败，临时文件会在下次写入时被覆盖

### 并发执行

系统支持并发执行没有依赖关系的子任务：

1. 识别可并发执行的子任务（没有相互依赖）
2. 使用 `Promise.all` 并发执行
3. 如果任一子任务失败，暂停其他子任务
4. 等待用户决策后再继续
