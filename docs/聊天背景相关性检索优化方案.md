## 📋 问题诊断

### 原有问题

1. **被动检索**: 系统仅在 LLM 通过 toolcall 调用 `memory_search` 或 `novel_reference` 时才执行检索
2. **缺乏主动性**: 用户发送消息后，没有第一时间主动检索相关背景信息
3. **关键词来源单一**: 仅从用户消息抽取关键词，忽略了 Agent 定义、系统提示词、背景提示词中的丰富信息
4. **ToolCall 2.0 未注入**: 工具定义没有正确注入到系统提示词中，LLM 不知道有哪些工具可用
5. **检索时机滞后**: 检索发生在 LLM 推理过程中，而不是在 prompt 构建前

### 影响

- LLM 回答缺乏必要的上下文支撑
- 需要多次往返 toolcall 才能获取足够信息
- 用户体验差，回答质量不稳定
- 宝贵的记忆系统和小说文本资源未被充分利用

---

## 🎯 优化目标

### 核心目标

1. **主动检索**: 在用户消息进入 prompt 前，主动从多维度检索相关信息
2. **智能注入**: 将检索结果格式化后注入到 extraSystemPrompt 中
3. **多源关键词**: 从用户消息、Agent 定义、系统提示词、背景提示词中抽取关键词
4. **ToolCall 2.0 注入**: 自动将可用工具定义注入到 prompt 中
5. **性能可控**: 检索延迟控制在可接受范围内 (<2 秒)

### 预期效果

- LLM 在回答前就已经"看到"了相关记忆和参考资料
- 减少不必要的 toolcall 往返
- 提升回答质量和一致性
- 改善用户体验和响应速度

## 🔧 核心实现

### 1. 主动检索引擎 (`proactive-retrieval.ts`)

#### 检索维度

```typescript
interface RetrievalDimension {
  source: "memory" | "novel" | "agent-def" | "tool-def";
  enabled: boolean;
  weight: number;
}
```

- **memory**: 记忆系统 (MEMORY.md + memory/*.md)
- **novel**: 小说文本 (NovelsAssets/*.txt)
- **agent-def**: Agent 定义文本 (从配置加载)
- **tool-def**: ToolCall 2.0 工具定义 (从工具注册表加载)

#### 关键词抽取

```typescript
function extractKeywordsFromContexts(options: {
  userMessage: string;
  agentDefinition?: string;
  systemPrompt?: string;
  backgroundPrompt?: string;
}): string[] {
  // 合并所有文本后统一抽取关键词
  const combinedText = [agentDefinition, systemPrompt, backgroundPrompt, userMessage]
    .filter(Boolean)
    .join("\n\n");
  return extractSearchTerms(combinedText);
}
```

#### 检索流程

```
用户消息到达
    ↓
抽取关键词 (多源)
    ↓
并行检索 (记忆 + 小说 + Agent 定义 + 工具定义)
    ↓
去重 + 加权排序
    ↓
格式化输出
    ↓
注入到 extraSystemPrompt
```

### 2. 集成到 Followup Runner

在 [`followup-runner.ts`](file://d:\Git_GitHub\clawdbot\src\auto-reply\reply\followup-runner.ts) 的子任务执行前:

```typescript
// 🆕 主动检索增强
let proactiveRetrievalCtx = "";
try {
  const retrievalResult = await proactiveRetrieval(config, {
    userMessage: queued.prompt,
    sessionId: taskTree?.id,
    sessionKey: queued.run.sessionKey,
    maxSnippets: 6,
    minScore: 0.35,
    enableMemory: true,
    enableNovel: true,
    enableAgentDef: false,
    enableToolDefs: true,
  });
  
  if (retrievalResult.formattedContext) {
    proactiveRetrievalCtx = retrievalResult.formattedContext;
  }
} catch (err) {
  console.debug(`主动检索失败 (不阻塞): ${err}`);
}

// 合并到 extraSystemPrompt
const parts = [
  base,
  proactiveRetrievalCtx, // 🆕 新增
  siblingCtx,
  blueprintCtx,
  // ... 其他组件
].filter(Boolean);
```

## 📊 性能指标

### 检索延迟 (典型值)

| 场景 | 配置 | 平均延迟 |
|------|------|----------|
| 仅记忆检索 | maxSnippets=5 | ~200-500ms |
| 记忆 + 小说 | maxSnippets=8 | ~500-1500ms |
| 全通道检索 | maxSnippets=10 | ~1000-3000ms |
| 快速检索 | maxSnippets=4 | ~100-300ms |

### Token 消耗

每个检索片段平均消耗:
- 短片段 (100 字): ~50 tokens
- 中片段 (300 字): ~150 tokens
- 长片段 (500 字): ~250 tokens

建议配置:
- `maxSnippets=6-8`: 约 900-2000 tokens
- 配合预算管理系统自动压缩

## 📈 使用场景

### 场景 1: 日常对话

```
用户："上次我们说的那个项目进度怎么样了？"

系统动作:
1. 从"上次"、"项目"、"进度"抽取关键词
2. 检索 memory/ 目录下相关记录
3. 注入最近的项目讨论片段
4. LLM 基于历史上下文回答

效果:
→ LLM 直接回答:"根据昨天的记录，项目已经完成第一阶段..."
→ 无需额外的 toolcall
```

### 场景 2: 小说创作

```
用户："写一段林娜和艾伦在星港分别的场景"

系统动作:
1. 抽取"林娜"、"艾伦"、"星港"、"分别"
2. 检索 NovelsAssets 中相关段落
3. 检索记忆中角色关系设定
4. 注入风格参考片段

效果:
→ LLM 创作出符合人设和风格的场景
→ 保持角色性格一致性
→ 场景描写符合既定风格
```

### 场景 3: 子任务执行

```
子任务："撰写第三章：宗门试炼"

系统动作:
1. 检索记忆中关于第三章的讨论
2. 检索 NovelsAssets 中第一、二章内容
3. 检索总纲中第三章设定
4. 注入角色卡片和风格指南
5. 注入已完成兄弟任务的输出

效果:
→ LLM 创作出符合整体规划的第三章
→ 与前文保持连贯
→ 遵循角色设定和风格指南
```

## 🚀 优化建议

### 短期优化

1. **缓存策略**: 对相同查询缓存检索结果 (30 秒 TTL)
2. **增量检索**: 只检索自上次以来新增的记忆内容
3. **预加载**: 在用户输入时就开始预检索
4. **权重调优**: 根据用户反馈调整各通道权重

### 中期优化

1. **ToolCall 2.0 完整集成**: 从工具注册表动态加载工具定义
2. **Agent 定义加载**: 从配置文件加载 Agent 定义文本
3. **背景提示词集成**: 从技能文件或配置加载背景提示词
4. **可视化调试**: 开发检索结果可视化界面

### 长期优化

1. **自适应学习**: 基于用户反馈自动调整检索策略
2. **多轮对话追踪**: 维护对话上下文的状态机
3. **跨会话共享**: 不同会话之间共享记忆检索结果
4. **语义理解升级**: 引入更强的语义理解模型

## 📝 注意事项

### 1. Token 预算管理

主动检索会增加 token 消耗，建议:
- 配合 `allocateBudget()` 使用
- 设置合理的优先级
- 在预算紧张时自动压缩

### 2. 检索延迟

- 设置 `timeoutMs` 超时保护
- 禁用不需要的通道
- 使用快速检索模式应对延迟敏感场景

### 3. 相关性控制

- 适当提高 `minScore` 阈值
- 减少 `maxSnippets` 数量
- 定期清理记忆系统

### 4. 故障隔离

- 检索失败不应阻塞主流程
- 使用 try-catch 包裹检索逻辑
- 提供降级策略 (无检索结果时的回退)
