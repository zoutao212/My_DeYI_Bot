# 历史消息 Tool Call 格式转换 - 完整修复方案

**日期**：2026-02-04  
**问题**：LLM 重复执行历史消息中的 tool call

---

## 问题根因

### 1. API 格式混淆

使用 Gemini 模型，但通过 OpenAI 兼容接口（vectorengine），导致：
- 请求格式：OpenAI 格式（`tool_calls`）
- 响应格式：Gemini 格式（`functionCall`）
- 历史消息格式：混合格式（部分 OpenAI，部分 Gemini）

### 2. 历史消息中的 Tool Call

当前实现：
- 历史消息中保留了完整的 `functionCall` 格式
- LLM 看到历史消息中的 `functionCall`，认为是"待执行的工具调用"
- 导致 LLM 每次都重复执行第一个 tool call

**示例**：
```json
{
  "role": "model",
  "parts": [
    {
      "functionCall": {
        "name": "read",
        "args": { "path": "SOUL.md" }
      }
    }
  ]
}
```

LLM 看到这个消息，会认为："哦，我需要调用 read 工具读取 SOUL.md"

---

## 解决方案

### 核心思路

**将历史消息中的 tool call 转换为"伪代码化"的文本描述，而不是保留实际的 tool call 格式。**

### 实现步骤

#### 步骤 1：在 `convertOpenAIToGeminiFormat` 中添加历史消息标记

**位置**：`src/agents/gemini-payload-thought-signature.ts`

**修改**：
```typescript
function convertOpenAIToGeminiFormat(messages: unknown[], options?: {
  markHistoryToolCalls?: boolean;  // 新增选项：是否标记历史 tool call
}): unknown[] {
  const markHistory = options?.markHistoryToolCalls ?? false;
  
  // ... 现有代码 ...
  
  // 转换 assistant 消息
  if (role === "assistant") {
    const toolCalls = msgRec.tool_calls;
    if (Array.isArray(toolCalls) && toolCalls.length > 0) {
      // 🆕 Fix: 如果是历史消息，转换为文本描述
      if (markHistory) {
        const toolCallDescriptions = toolCalls
          .map((tc) => {
            if (!tc || typeof tc !== "object") return null;
            const tcRec = tc as Record<string, unknown>;
            const func = tcRec.function;
            if (!func || typeof func !== "object") return null;
            const funcRec = func as Record<string, unknown>;
            
            const name = typeof funcRec.name === "string" ? funcRec.name : "unknown";
            const args = typeof funcRec.arguments === "string" ? funcRec.arguments : "{}";
            
            // 转换为伪代码描述
            return `[已执行工具调用] ${name}(${args})`;
          })
          .filter(Boolean)
          .join("\n");
        
        return {
          role: "model",
          parts: [{ text: toolCallDescriptions }],
        };
      }
      
      // 当前消息：保留完整的 functionCall 格式
      const parts = toolCalls
        .map((tc) => {
          // ... 现有代码 ...
        })
        .filter(Boolean);
      
      return {
        role: "model",
        parts: parts,
      };
    }
    
    // ... 现有代码 ...
  }
  
  // ... 现有代码 ...
}
```

#### 步骤 2：在构建 payload 时区分历史消息和当前消息

**位置**：`src/agents/gemini-payload-thought-signature.ts` 的 `wrapStreamFn`

**修改**：
```typescript
// Fix 4: Convert OpenAI format to Gemini format for vectorengine
if (shouldConvertToGemini) {
  if (Array.isArray(messages)) {
    log.info(`[format] Converting OpenAI format to Gemini format for vectorengine (${messages.length} messages)`);
    
    // 🆕 Fix: 区分历史消息和当前消息
    // 最后一条用户消息是"当前消息"，之前的都是"历史消息"
    const lastUserIndex = messages.findLastIndex((msg: any) => msg?.role === "user");
    
    const convertedMessages = messages.map((msg, index) => {
      // 判断是否是历史消息
      const isHistory = index < lastUserIndex;
      
      // 转换消息
      return convertOpenAIToGeminiFormat([msg], {
        markHistoryToolCalls: isHistory,  // 历史消息标记 tool call
      })[0];
    });
    
    payloadObj.messages = convertedMessages;
    log.info(`[format] ✓ Converted to Gemini format (history tool calls marked as text)`);
  }
}
```

**位置**：`convertOpenAIToGeminiFormat` 函数

if (role === "tool") {

  if (markHistory) {

}

## 关键教训

1. **历史消息和当前消息必须区分**
   - 历史消息：已执行的 tool call，应该转换为文本描述
   - 当前消息：待执行的 tool call，保留完整格式

2. **API 格式混淆是根本问题**
   - vectorengine 声称 OpenAI 兼容，但实际上期望 Gemini 格式
   - 必须在发送前转换格式

3. **LLM 无法区分"历史"和"当前"**
   - LLM 只看到消息内容，无法判断是否已执行
   - 必须通过格式转换来明确区分

4. **伪代码化是最佳方案**
   - 保留了工具调用的信息（用于上下文理解）
   - 避免了 LLM 重复执行
