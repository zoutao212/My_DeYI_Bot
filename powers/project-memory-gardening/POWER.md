# Project Memory Gardening

## 概述

**Project Memory Gardening** 是一套系统化的代码地图构建方法论，帮助 AI 助手在新对话中快速建立项目印象，无需每次都全盘扫描代码库。

### 核心价值

- **降低定位成本**：通过稳定的索引文件，让 AI 直接找到入口、模块、调用链
- **减少 token 消耗**：避免重复扫描，只读取必要的地图文件
- **提升协作效率**：新对话可以立即开始工作，而不是花时间"摸索代码库"
- **持续演进**：随着项目发展，地图也在增量更新

### 适用场景

- 大型代码库（文件数 > 100）
- 多模块/多层架构项目
- 频繁需要 AI 协助的项目
- 团队协作项目（让所有 AI 助手共享同一份地图）

## 核心概念：代码地图体系

代码地图是一套**轻量级索引文件**，存放在 `ProjectMemory/00_索引/` 目录下：

### 必备地图文件

| 文件名 | 用途 | 示例内容 |
|--------|------|----------|
| `模块地图.md` | 目录级模块导航 | 各模块职责、边界、依赖关系 |
| `入口清单.md` | 系统入口映射 | 启动脚本 → 文件路径 → 关键函数 |
| `调用链地图.md` | 关键流程调用链 | 入口 → 中间层 → 数据层 |
| `定位速查表.md` | 功能定位指南 | "我想找 XX" → 推荐查找路径 |
| `关键词索引.md` | 关键词映射 | 关键词 → 文档/文件路径 |

### 可选细节文档

- `ProjectMemory/01_代码文件/`：按模块组织的详细代码说明
- `ProjectMemory/02_关键流程/`：复杂业务流程的深度分析

## 触发场景

满足以下任一场景时，应该调用本 Power 维护代码地图：

### 1. 新对话需要快速建立印象
用户要求"不要扫全库，直接定位"时。

### 2. 新增/变更入口
- 新增启动脚本
- 新增 API 路由
- 新增任务调度入口

### 3. 新增/重构模块
- 目录结构变化
- 模块职责边界调整
- 抽象层变化

### 4. 修复复杂 Bug
说明当前"定位路径/调用链/关键不变量"需要补全。

### 5. 新增关键配置/环境依赖
- 端口配置变化
- 环境变量新增
- 部署脚本更新
- 第三方服务集成

## 工作流程

### 第一步：明确定位目标

用一句话描述用户想定位什么：

**示例：**
- "Web UI 某按钮触发的后端处理在哪里"
- "数据库查询是在哪个模块执行的"
- "配置文件是如何加载的"

### 第二步：补齐入口映射

在 `00_索引/入口清单.md` 中记录：

```markdown
## Web API 入口

| 路由 | 文件路径 | 关键函数 | 说明 |
|------|----------|----------|------|
| GET /api/users | src/routes/users.ts | getUserList() | 获取用户列表 |
| POST /api/login | src/routes/auth.ts | handleLogin() | 用户登录 |
```

### 第三步：补齐最短调用链

在 `00_索引/调用链地图.md` 中记录：

```markdown
## 用户登录流程

入口: POST /api/login (src/routes/auth.ts)
  ↓
中间层: AuthService.authenticate() (src/services/auth.ts)
  ↓
数据层: UserRepository.findByEmail() (src/repositories/user.ts)
  ↓
外部依赖: PostgreSQL (users 表)
```

### 第四步：补齐定位速查

在 `00_索引/定位速查表.md` 中记录：

```markdown
## 我想找：用户认证相关代码

**推荐路径：**
1. 先查 `入口清单.md` 找到 `/api/login` 路由
2. 再查 `调用链地图.md` 看完整流程
3. 关键词搜索：`authenticate`, `login`, `auth`

**常见误区：**
- 不要直接搜 "user"（太宽泛）
- 认证逻辑在 services 层，不在 routes 层
```

### 第五步：补齐关键词索引

在 `00_索引/关键词索引.md` 中记录：

```markdown
## 认证相关

- `authenticate` → src/services/auth.ts
- `login` → src/routes/auth.ts, src/services/auth.ts
- `JWT` → src/utils/jwt.ts
- `session` → src/middleware/session.ts
```

### 第六步：挂载到模块地图

确保 `模块地图.md` 能导航到上述地图文件：

```markdown
## 索引导航

- [入口清单](./入口清单.md) - 系统所有入口点
- [调用链地图](./调用链地图.md) - 关键流程调用链
- [定位速查表](./定位速查表.md) - 功能定位指南
- [关键词索引](./关键词索引.md) - 快速搜索关键词
```

## 质量门槛

每次维护代码地图，必须满足以下标准：

### 必须补充的信息（至少满足）

- ✅ 至少补 1 条"入口 → 文件/函数/路由"的可定位信息
- ✅ 至少补 1 条"最短调用链"
- ✅ 至少补 3 个可搜索关键词

### 特殊场景处理

**路由归属不确定时：**
- 优先使用运行时路由表（例如 `GET /api/debug/routes`）确认实际路由
- 再回到对应文件/函数补齐代码地图
- 在 `定位速查表.md` 中注明"运行时路由表"作为定位依据

**同一路径多处注册时：**
- 必须在调用链地图中标注优先级/覆盖关系
- 在定位速查表中说明"行为与代码不一致"的风险

## 专项建档指南

### 新能力建档（以 Doc Edit 为例）

当项目新增一个能力模块时，按以下步骤建档：

#### 1. 描述能力目标
用一句话描述新能力的目标/开关。

**示例：** "Doc Edit 允许 LLM 生成结构化改写指令"

#### 2. 记录入口
在 `00_索引/入口清单.md` 增加入口，指出 UI/接口如何触发能力。

```markdown
## Doc Edit 入口

| 入口类型 | 路径 | 触发方式 |
|----------|------|----------|
| Web UI | /docs/editor | 点击"编辑文档"按钮 |
| API | POST /api/docs/edit | 提交编辑指令 |
```

#### 3. 记录调用链
在 `00_索引/调用链地图.md` 记录"入口 → 调度层 → 执行层 → 数据层/配置"最短链路。

```markdown
## Doc Edit 流程

入口: POST /api/docs/edit (server/routes/docs.ts)
  ↓
调度层: DocEditService.processEdit() (server/services/doc-edit.ts)
  ↓
执行层: LLMProvider.generateEdit() (server/providers/llm.ts)
  ↓
配置: llm_providers.yaml (白名单/模型配置)
```

#### 4. 建立子索引（如需要）
若涉及白名单/配置/脚本，在 `01_代码文件/<模块>/` 建子索引。

```markdown
# Doc Edit 模块索引

## 关键文件

- `server/routes/docs.ts` - API 路由定义
- `server/services/doc-edit.ts` - 核心业务逻辑
- `server/config/doc-edit-whitelist.json` - 白名单配置

## 验证方式

- 单元测试: `npm test -- doc-edit`
- 集成测试: `curl -X POST /api/docs/edit -d '...'`
```

#### 5. 补充关键词索引
在 `00_索引/关键词索引.md` 和 `定位速查表.md` 写入不少于 3 个可搜索关键词。

```markdown
## Doc Edit 关键词

- `doc-edit` → server/services/doc-edit.ts
- `document-editor` → server/routes/docs.ts
- `edit-whitelist` → server/config/doc-edit-whitelist.json
```

## 前端超大脚本拆分规范

当单个前端脚本文件持续膨胀（数千行、多个业务域混杂）时：

### 拆分原则

1. **按业务域/功能簇拆分**为多个静态脚本文件
2. **保留薄入口文件**负责通用工具与初始化
3. **明确依赖关系**和加载顺序

### 建档要求

#### 1. 记录入口与加载顺序
在 `00_索引/入口清单.md` 记录页面入口与脚本加载顺序：

```markdown
## 前端页面入口

| 页面 | 模板文件 | 脚本加载顺序 |
|------|----------|--------------|
| 文档编辑器 | templates/editor.html | 1. common.js<br>2. file-picker.js<br>3. editor-main.js |
```

#### 2. 补充关键词索引
在 `00_索引/关键词索引.md` 记录：

- **关键 UI 功能名**：文件选择器、多文件 prompt-sources
- **关键 DOM id**：`file-picker-list`、`prompt-sources-*-list`
- **关键函数名**：`openFilePicker`、`loadPromptSources`

```markdown
## 前端关键词

- `file-picker` → static/js/file-picker.js
- `openFilePicker()` → static/js/file-picker.js:45
- `#file-picker-list` → templates/editor.html:120
```

#### 3. 补充定位速查
在 `00_索引/定位速查表.md` 增加前端定位路径：

```markdown
## 我想找：前端某个按钮/弹窗/列表渲染在哪里

**推荐路径：**
1. 先从页面模板（`server/templates/*.html`）找 `onclick/DOM id`
2. 再根据 `<script src>` 加载顺序定位到具体静态脚本文件
3. 最后在目标脚本内按函数名/DOM id 搜索

**示例：**
- 找"文件选择器"按钮 → templates/editor.html 搜 `file-picker`
- 找点击事件 → static/js/file-picker.js 搜 `openFilePicker`
```

## 最佳实践

### 1. 先地图，后细节

- **优先维护 `00_索引/` 下的 5 个核心地图文件**
- 只在必要时才写 `01_代码文件/` 或 `02_关键流程/` 的详细文档
- 保持地图文件的轻量和可读性

### 2. 增量更新，小步迭代

- 每次只补充与当前任务相关的地图信息
- 避免一次性写成"大而全"的文档
- 随着项目演进，地图自然完善

### 3. 保持可发现性

- 所有新增内容必须挂载到 `模块地图.md`
- 关键词索引必须保持更新
- 定位速查表要包含"常见误区"

### 4. 验证地图有效性

每次更新后，问自己：
- **下次遇到同类问题，新对话需要读哪些文件就能定位？**（最好 ≤ 6 个）
- **是否减少了全库扫描需求？**
- **关键词是否足够精确？**

## 与其他工作流的协同

### 与 Maintain Rules 的分工

- **缺的是地图** → 用 Project Memory Gardening
- **缺的是约束/标准动作** → 用 Maintain Rules

**判断标准：**
- 如果问题是"不知道在哪里"→ 补地图
- 如果问题是"知道在哪，但不知道怎么做"→ 补规则

### 进化策略

如果发现地图模板缺项/不适配：
1. 先补充到对应地图文件
2. 再用 Maintain Rules 将"为什么要补、补了什么、后续如何使用"固化为规则

## 常见问题

### Q: 代码地图会不会过时？
A: 会，但这是正常的。地图是"增量维护"的，每次修改代码时顺手更新相关地图即可。

### Q: 需要为每个文件都写文档吗？
A: 不需要。只为"入口"、"关键调用链"、"难以定位的模块"写地图。大部分代码通过关键词索引就能找到。

### Q: 地图文件应该多详细？
A: 保持轻量。每个地图文件控制在 100-300 行以内。如果太长，说明应该拆分或提炼。

### Q: 如何处理多人协作？
A: 地图文件应该纳入版本控制（Git）。团队成员共同维护，就像维护 README 一样。

## 总结

Project Memory Gardening 不是"写文档"，而是**构建可定位的认知地图**。

核心原则：
- **先构建可定位的认知（地图），再扩展细节**
- **优先维护索引，按需补充细节**
- **保持轻量，增量迭代**

通过持续维护代码地图，让每次新对话都能快速建立项目印象，从而：
- 降低定位成本
- 减少 token 消耗
- 提升协作效率
- 让 AI 助手更智能
