# Maintain Rules & Skills

## 概述

**Maintain Rules & Skills** 是一套让 AI 助手具备**可持续认知构建能力**的方法论。它不是简单的"修修补补规则"，而是从一次次对话中提炼稳定模式，固化为可复用的工作流。

### 核心价值

- **模式提炼**：把反复出现的定位/调试/协作模式提炼为"最小工作流"
- **认知固化**：把有效工作流固化为规则（约束行为）、代码地图（降低扫描成本）、技能（自动化执行）
- **持续优化**：让系统随着使用而变得更智能，而不是重复犯错
- **降低成本**：减少 token 消耗，提升响应速度

### 核心原则

**先构建可定位的认知（地图），再扩展细节。**

## 触发信号

满足以下任一信号时，应该调用本 Power 进行规则/技能维护：

### 1. 重复定位成本过高
新对话/新任务总要"扫一遍代码库"才能找到入口。

**示例：**
- 每次都要问"这个功能在哪个文件？"
- 每次都要重新搜索路由定义
- 每次都要重新理解模块边界

### 2. 出现"事实与代码不一致"
例如路由冲突、配置覆盖、运行时行为与静态搜索对不上。

**示例：**
- 代码里定义了路由 A，但运行时实际响应的是路由 B
- 配置文件有多处定义，不知道哪个生效
- 函数被多处覆盖，静态搜索找不到实际执行的版本

### 3. 出现可复用套路
同类问题反复出现（例如"Web UI → 后端 → 核心模块定位"）。

**示例：**
- 每次定位前端按钮都要走一遍"模板 → 脚本 → 函数"
- 每次调试 API 都要走一遍"路由 → 服务 → 数据层"
- 每次配置环境都要重新查文档

### 4. 规则/流程缺口导致错误
你已经修复了问题，但发现根因是"缺少规则/缺少地图/缺少技能"。

**示例：**
- 修改了代码但忘记更新文档
- 新增了入口但没有记录到地图
- 解决了问题但没有固化为规则

### 5. 用户提出新的协作范式
例如强调成长性、认知构建、减少 token 消耗。

**示例：**
- 用户要求"不要每次都扫全库"
- 用户要求"记住这个套路，下次直接用"
- 用户要求"把这个流程自动化"

## 执行流程（固定闭环）

### 第一步：识别模式（Pattern）

用 3 句话写清楚：

1. **问题是什么**（现象）
   - 示例："每次定位前端按钮都要花 5 分钟搜索"

2. **为什么会反复发生**（成本/不确定性来源）
   - 示例："因为没有记录'模板 → 脚本 → 函数'的映射关系"

3. **最小可复用解法是什么**（工作流雏形）
   - 示例："在定位速查表中记录'前端定位三步法'"

### 第二步：选择固化形态（Form）

必须在以下三者中至少选一个（允许组合）：

#### 1. 规则（Rules）
约束行为与质量门槛。

**适用场景：**
- 需要强制执行的流程
- 需要避免的错误模式
- 需要统一的标准

**示例：**
- "修改代码后必须验证"
- "新增入口必须记录到地图"
- "使用中文交流"

#### 2. 代码地图（Code Map）
让新对话直接定位（优先）。

**适用场景：**
- 定位成本高
- 调用链复杂
- 入口不明确

**示例：**
- 入口清单：路由 → 文件 → 函数
- 调用链地图：入口 → 中间层 → 数据层
- 定位速查表：功能 → 查找路径

#### 3. 技能（Skills）
把工作流变成可重复执行的程序化步骤。

**适用场景：**
- 固定流程反复执行
- 多步骤操作容易遗漏
- 需要标准化的操作

**示例：**
- project_memory_gardening：维护代码地图
- safe_file_editor：安全编辑规则文件
- maintain_rules：维护规则和技能

### 第三步：草拟变更（Draft）

#### 内容要求

- **只写"最小可用"**：避免一次性写成大而全
- **必须可被检索**：包含路径/关键词/入口
- **必须可验证**：能够测试是否有效

#### 示例：规则变更

```markdown
## 新增规则：前端定位三步法

**触发场景：** 需要定位前端某个按钮/弹窗/列表渲染位置

**标准流程：**
1. 先从页面模板（`templates/*.html`）找 `onclick/DOM id`
2. 再根据 `<script src>` 加载顺序定位到具体静态脚本文件
3. 最后在目标脚本内按函数名/DOM id 搜索

**验证方式：**
- 能在 3 分钟内定位到目标代码
- 不需要全局搜索
```

#### 示例：代码地图变更

```markdown
## 新增入口：文档编辑器

| 入口类型 | 路径 | 触发方式 | 关键函数 |
|----------|------|----------|----------|
| Web UI | /docs/editor | 点击"编辑文档"按钮 | openEditor() |
| API | POST /api/docs/edit | 提交编辑指令 | handleEdit() |

**调用链：**
入口: POST /api/docs/edit (routes/docs.ts)
  ↓
服务层: DocEditService.processEdit() (services/doc-edit.ts)
  ↓
数据层: DocumentRepository.update() (repositories/document.ts)
```

#### 示例：技能变更

```markdown
## 新增技能：前端脚本拆分

**目的：** 当单个前端脚本文件持续膨胀时，按业务域拆分

**步骤：**
1. 识别业务域边界（例如：文件选择器、编辑器、预览）
2. 创建独立脚本文件（例如：file-picker.js, editor.js, preview.js）
3. 更新入口清单记录加载顺序
4. 更新关键词索引记录 DOM id 和函数名
5. 更新定位速查表记录定位路径

**验证：**
- 每个脚本文件 < 500 行
- 业务域边界清晰
- 定位速度提升
```

### 第四步：闭环验收（Gate）

每次固化必须回答以下问题：

#### 1. 定位效率
**下次遇到同类问题，新对话需要读哪些文件就能定位？**

- 目标：≤ 6 个文件
- 如果超过 6 个，说明地图不够强

#### 2. 扫描成本
**是否减少了全库扫描需求？**

- 如果没有，说明地图不够强
- 应该能通过索引直接定位

#### 3. 回滚策略
**规则/技能写错时如何修正？**

- 必须有备份或版本控制
- 必须能快速回滚

### 第五步：写入与版本化（Write & Version）

#### 写入对应文件

根据第二步选择的固化形态，写入对应文件：

- **规则** → `.kiro/steering/*.md`
- **代码地图** → `ProjectMemory/00_索引/*.md`
- **技能** → `.kiro/skills/*/SKILL.md`

#### 增加版本标记

在文件末尾或标题附近追加版本信息：

```markdown
---
最后更新: 2026-01-29
版本: v20260129_1
变更: 新增前端定位三步法
---
```

#### 保持可发现性

更新索引文件，确保新内容可被发现：

- 更新 `ProjectMemory/00_索引/模块地图.md`
- 更新 `ProjectMemory/00_索引/关键词索引.md`
- 更新 `.kiro/steering/README.md`（如果有）

## 与 Project Memory Gardening 的协同

### 分工原则

- **缺的是地图** → 用 `project_memory_gardening`
- **缺的是约束/标准动作** → 用 `maintain_rules`

### 判断标准

| 问题类型 | 使用哪个 Power | 示例 |
|----------|----------------|------|
| 不知道在哪里 | Project Memory Gardening | "这个功能在哪个文件？" |
| 知道在哪，但不知道怎么做 | Maintain Rules | "修改后需要做什么验证？" |
| 定位成本高 | Project Memory Gardening | "每次都要扫全库" |
| 反复犯同样的错误 | Maintain Rules | "又忘记更新文档了" |
| 需要固定流程 | Maintain Rules | "这个操作步骤太多容易遗漏" |

### 协同工作流

```
发现问题
  ↓
判断：缺地图 or 缺规则？
  ↓
缺地图 → Project Memory Gardening
  ├─ 补充入口清单
  ├─ 补充调用链地图
  └─ 补充定位速查表
  ↓
缺规则 → Maintain Rules
  ├─ 提炼模式
  ├─ 固化为规则/技能
  └─ 更新索引
  ↓
验证效果
  ├─ 定位速度是否提升？
  ├─ 是否减少重复错误？
  └─ 是否降低 token 消耗？
```

## 高级编辑通道：SafeFileEditor

### 什么时候用

当你需要修改规则文件，但 IDE 内置编辑工具返回"禁止写入/只读"时。

**典型场景：**
- 修改 `.kiro/steering/*.md` 规则文件
- 修改 `.kiro/skills/*/SKILL.md` 技能文件
- 对大文件做精确且可验证的替换

### 使用原则（必须遵守）

1. **先备份再修改**：规则文件属于"系统认知"，任何修改都必须可回滚
2. **先预览再落盘**：优先使用"精确替换 + diff 预览"的方式，确认修改范围
3. **最小改动**：一次只固化一个稳定模式，避免把规则写成大而全

### 推荐用法（CLI）

工具位置：`.kiro/skills/safe_file_editor/`

#### 按行号替换（最安全）

```bash
python .kiro/skills/safe_file_editor/quick_replace.py by-lines \
  <文件> <start> <end> "<新内容>" -y
```

**示例：**
```bash
python .kiro/skills/safe_file_editor/quick_replace.py by-lines \
  .kiro/steering/iflow.md 45 50 "## 新增规则\n\n修改代码后必须验证" -y
```

#### 按关键字替换（当行号不稳定）

```bash
python .kiro/skills/safe_file_editor/quick_replace.py by-keywords \
  <文件> "<主关键字>" \
  --context "<上下文1,上下文2>" \
  --end "<结束关键字>" \
  --code "<新内容>" -y
```

**示例：**
```bash
python .kiro/skills/safe_file_editor/quick_replace.py by-keywords \
  .kiro/steering/iflow.md "## 调试流程" \
  --context "验证,测试" \
  --end "## 下一节" \
  --code "## 调试流程\n\n1. 修改代码\n2. 运行测试\n3. 验证结果" -y
```

### 与 IDE 内置编辑的分工

- **能用 IDE 工具直接改**：优先用 IDE 工具（可追踪、改动更直观）
- **IDE 工具被限制**：用 SafeFileEditor 作为"升级通道"
- **完成修改后**：把变更要点同步到 `ProjectMemory/00_索引/*` 保持可发现

## 最佳实践

### 1. 模式识别要准确

不要把偶然问题当成模式：
- 至少出现 3 次才考虑固化
- 确认是"可复用"的模式，不是特例
- 确认固化后能真正降低成本

### 2. 固化要最小化

避免一次性写成大而全：
- 只固化"最小可用"的部分
- 保持规则/技能的简洁性
- 随着使用逐步完善

### 3. 验证要及时

每次固化后立即验证：
- 下次遇到同类问题，是否真的更快？
- 是否真的减少了 token 消耗？
- 是否真的避免了重复错误？

### 4. 保持可发现性

所有固化的内容必须可被发现：
- 更新索引文件
- 添加关键词
- 记录在地图中

### 5. 定期回顾

定期回顾规则/技能的有效性：
- 哪些规则/技能经常使用？（保留并优化）
- 哪些规则/技能从未使用？（考虑删除）
- 哪些规则/技能需要更新？（及时更新）

## 进化策略（自我改进）

### 发现模板缺项

如果发现模板缺项/不适配：
1. 先补充到对应模板文件
2. 再用本 Power 将"为什么要补、补了什么、后续如何使用"固化为规则

### 发现新的模式

如果发现新的可复用模式：
1. 先用 3 句话描述模式
2. 选择固化形态（规则/地图/技能）
3. 草拟变更
4. 闭环验收
5. 写入与版本化

### 发现规则冲突

如果发现规则之间有冲突：
1. 识别冲突的规则
2. 分析冲突原因
3. 确定优先级
4. 更新规则并记录决策依据

## 常见问题

### Q: 什么时候应该固化为规则？
A: 当同样的错误反复出现，或者需要强制执行某个流程时。

### Q: 什么时候应该固化为代码地图？
A: 当定位成本高，或者调用链复杂时。优先选择代码地图。

### Q: 什么时候应该固化为技能？
A: 当固定流程反复执行，或者多步骤操作容易遗漏时。

### Q: 如何判断固化是否成功？
A: 看下次遇到同类问题时，是否真的更快、更准确、更省 token。

### Q: 规则/技能会不会过时？
A: 会。定期回顾并更新是必要的。删除无用的规则/技能也是进化的一部分。

### Q: 如何避免规则/技能过多？
A: 保持"最小可用"原则，只固化真正有价值的模式。定期清理无用内容。
