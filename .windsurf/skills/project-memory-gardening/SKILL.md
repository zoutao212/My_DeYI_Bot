# 🧠 项目记忆库维护（ProjectMemory Gardening）

## 目的

- 为后续 AI 编码提供“可直接读取的代码地图”，避免每次新对话都全盘扫描代码库。
- 用稳定的索引文件把“入口在哪、模块在哪、关键调用链怎么走、关键词搜什么”固化下来。
- 通过“小步增量 + 索引优先”持续完善项目的完整印象。

## 触发场景（满足任一即可触发）

1. **新对话开始需要快速建立印象**：用户要求“不要扫全库，直接定位”。

2. **新增/变更入口**：启动脚本、主程序、API 路由、任务调度入口发生变化。

3. **新增/重构模块**：目录结构变化、职责边界变化、抽象层变化。

4. **修复复杂 Bug**：说明当前“定位路径/调用链/关键不变量”需要补全。

5. **新增关键配置/环境依赖**：端口、环境变量、部署脚本、第三方服务。

## 输入

- 本次任务的目标与结果（做了什么、为什么）。

- 影响到的“定位信息”（入口、路由、模块边界、关键函数/类、关键配置）。

- 可复现的定位路径（从“我想找 XX 功能”到“落到哪些文件/函数”）。

## 输出（写入到仓库）

必须优先维护 `ProjectMemory/00_索引/` 下的“代码地图”固定产物（尽量只改这几个文件，避免分散）：

- `模块地图.md`：目录级模块导航（你已经能从这里进入所有文档）。

- `入口清单.md`：系统入口（启动脚本/主程序/API 入口/任务入口）到文件路径的映射。

- `调用链地图.md`：关键流程的最短调用链（入口 -> 核心模块 -> 数据层/外部依赖）。

- `定位速查表.md`：按“我想找什么功能/现象/关键词”给出推荐查找路径（含关键词）。

- `关键词索引.md`：关键词 -> 文档/文件路径（保持可发现性）。

## 执行步骤（固定流程：先地图，后细节）

1. **先明确“用户想定位什么”**：用一句话描述目标（例如“Web UI 某按钮触发的后端处理在哪里”）。

2. **补齐入口映射**：把入口写进 `00_索引/入口清单.md`（入口类型 + 文件路径 + 关键函数/路由）。

3. **补齐最短调用链**：把“入口 -> 关键中间层 -> 终点（DB/外部服务）”写进 `00_索引/调用链地图.md`.

4. **补齐定位速查**：把“搜索关键词/路径优先级/常见误区”写进 `00_索引/定位速查表.md`.

4.1 **遇到路由归属不确定时，先确认运行时事实**：
    - 优先使用运行时路由表（例如 `GET /api/debug/routes`）确认 `path`/`endpoint`
    - 再回到对应文件/函数补齐代码地图
5. **挂载到模块地图**：确保 `模块地图.md` 能导航到上述地图文件。
6. **必要时再补细节文档**：只在需要时才写 `01_代码文件/` 或 `02_关键流程/` 的详细文档。

## 质量门槛（必须满足）

- 至少补 1 条“入口 -> 文件/函数/路由”的可定位信息。
- 至少补 1 条“最短调用链”。
- 至少补 3 个可搜索关键词（用于快速 grep/搜索）。
- 如果存在“同一路径多处注册/行为与代码不一致”的风险：必须把“运行时路由表”作为定位依据写进 `定位速查表.md` 或对应调用链说明中。

## 能力专项建档指南（以 Doc Edit 为例）
- **定位**：Skill 文档只规定“怎么建档”；真正的代码细节必须写进 ProjectMemory（如 `ProjectMemory/01_代码文件/...`、`00_索引/...`），便于不同项目共用同一套流程。
- **建档步骤**：
  1. 用一句话描述新能力的目标/开关（示例：Doc Edit 允许 LLM 生成结构化改写指令）。
  2. 在 `ProjectMemory/00_索引/入口清单.md` 增加入口，指出 UI/接口如何触发能力。
  3. 在 `ProjectMemory/00_索引/调用链地图.md` 记录“入口 -> 调度层 -> 执行层 -> 数据层/配置”最短链路。
  4. 若涉及白名单/配置/脚本，另在 `ProjectMemory/01_代码文件/<模块>/` 建子索引，集中列出关键文件与验证方式。
  5. 在 `ProjectMemory/00_索引/关键词索引.md` 和 `定位速查表.md` 写入不少于 3 个可搜索关键词与排查路径。
- **落地示例**：Doc Edit 相关的所有路径、调用链、配置清单，已经沉淀在 `ProjectMemory/01_代码文件/VirtualWorld_99_System/docs_editor/00_索引.md` 及其挂载的索引页；后续若更新，只需改 ProjectMemory，不要把具体文件名写回本 Skill。

## 前端超大脚本拆分（补充规范）
当出现单个前端脚本文件持续膨胀（数千行、多个业务域混杂、改动频繁易冲突）时：

1. **拆分原则**：按“业务域/功能簇”拆分为多个静态脚本文件，保留薄入口文件负责通用工具与初始化。
2. **入口记录**：必须在 `ProjectMemory/00_索引/入口清单.md` 记录页面入口与脚本加载顺序（尤其是依赖关系）。
3. **关键词索引**：必须在 `ProjectMemory/00_索引/关键词索引.md` 记录：
   - 关键 UI 功能名（例如：文件选择器、多文件 prompt-sources）
   - 关键 DOM id（例如：`file-picker-list`、`prompt-sources-*-list`）
   - 关键函数名（例如：`openFilePicker`、`loadPromptSources`）
4. **定位速查**：在 `ProjectMemory/00_索引/定位速查表.md` 增加“我想找前端某个按钮/弹窗/列表渲染在哪里”的定位路径：
   - 先从页面模板（例如 `server/templates/*.html`）找 `onclick/DOM id`
   - 再根据 `<script src>` 加载顺序定位到具体静态脚本文件
   - 最后在目标脚本内按函数名/DOM id 搜索

## 进化策略（自我改进）
- 如果发现模板缺项/不适配：
  - 先补充到对应模板文件；
  - 再用 `maintain_rules` Skill 将“为什么要补、补了什么、后续如何使用”固化为规则.
