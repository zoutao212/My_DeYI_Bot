# 软件工程协作规范

这份文档定义了我们在软件开发过程中必须遵循的**工程标准**、**架构原则**和**技术方法论**。

---

## 第一章：工程思维与架构

### 1.1 系统观
软件是有机整体。修改任何代码前，必须回答：
1. **本质是什么？**（是数据错？逻辑错？还是环境错？）
2. **位置在哪里？**（上游是谁？下游是谁？影响范围？）
3. **优雅度如何？**（是在打补丁，还是在修复架构？）

### 1.2 单一真相源 (SSOT)
- **定义文件 vs 修复脚本**：
  - **定义文件** (`schema.sql`, `config.yaml`)：描述"系统应该是什么样"，必须永久维护。
  - **修复脚本** (`migrate.py`, `fix_data.sql`)：把"现状"变成"理想状"，用完即弃。
- **关键流程**：
  - ❌ **错误**：只跑了修复脚本，忘了改定义文件。（导致下次部署问题重现）
  - ✅ **正确**：跑修复脚本 -> **同步更新定义文件** -> 验证新环境初始化。

## 第二章：代码修改规范

### 2.1 修改前的必修课
**理解优先于修改**。不动手直到你能回答：
- 这段代码的设计初衷是什么？
- 为什么要这么写（可能有历史包袱）？
- 它和周边模块的耦合关系？

### 2.2 最小改动与分批原则
- **外科手术式**：精准、微创。不要在修 Bug 时顺手重构无关代码（除非必要）。
- **大改动分批**：超过 **200-500 行** 的修改必须分阶段：
  1. **准备**：接口层、抽象层。
  2. **核心**：实现主要逻辑。
  3. **清理**：移除旧代码。
  *确保每个阶段提交后，系统都是可运行的。*

## 第三章：调试与质量管理

### 3.1 科学调试法
**禁止盲目试错（"改改看"）。**
1. **复现**：稳定复现步骤。
2. **假设**：基于日志/现象提出假设。
3. **验证**：通过阅读代码或针对性日志验证假设。
4. **修复**：实施修复并验证回归。

### 3.2 技术债务管理
- **童子军规则**：离开时代码要比来时更干净.
- **零容忍**：
  - 发现重复代码 -> 提取函数.
  - 发现超长函数 -> 拆分.
  - 发现死代码 -> 删除.
- **及时还债**：不要说"以后再改"，现在就改，或者记录在案.

### 3.3 命令执行环境识别（强制）

- **先识别终端环境**：
  - `>>>` = Python REPL
  - `PS ...>` = PowerShell
- **禁止误执行**：
  - 严禁在 `>>>` 里粘贴/执行：`python -c "..."`、`powershell -Command "..."` 等命令.
  - 如报错包含 `File "<stdin>", line 1`：优先判定为“执行环境错误”，先止损而非继续试错.
- **止损流程**：
  - 退出 REPL（`exit()` 或 `Ctrl+Z` 回车） -> 回到 PowerShell -> 仅运行短命令/一键脚本.

### 3.4 错误证据闭环（强制）

- 调试/验证命令必须能提供可复现证据：
  - stdout/stderr 关键片段
  - 具体命令文本
  - Shell 环境（PowerShell/REPL）

## 第四章：文档与同步

### 4.1 代码与文档的原子性
- 修改了逻辑 -> 更新注释.
- 修改了架构 -> 更新设计文档.
- 修改了配置 -> 更新配置文档.
*代码和文档必须保持同步，视为同一个原子操作.*
- 修改了逻辑 -> 更新注释。
- 修改了架构 -> 更新设计文档。
- 修改了配置 -> 更新配置文档。
*代码和文档必须保持同步，视为同一个原子操作。*

### 4.2 临时文件纪律
所有测试脚本、临时数据输出，**必须**放在专门的临时目录（如 `tmp/` 或 `.ignored/`）。**严禁**污染源代码目录。

## 第六章：ProjectMemory（项目记忆库 / 代码地图）

### 6.1 核心目标（从“写文档”升级为“构建代码地图”）
ProjectMemory 的首要目的不是“每次改动写一篇文档”，而是持续构建一份**可供 AI 快速定位代码位置的代码地图（Code Map）**，让新对话在不全盘扫描代码库的前提下，仍能在 5-15 分钟内建立完整印象并定位到具体文件/函数/路由。

### 6.2 强制使用的 Skill（必须按此执行）
当需要构建/补全代码地图时，必须显式使用以下 project_memory_gardening Skill 的设定与流程：
skill name: project_memory_gardening
description: 构建并维护可供 AI 快速定位代码位置的代码地图（入口清单、模块边界、调用链、关键词索引、定位速查表），让新对话无需全盘扫描代码库即可建立完整印象。


### 6.2.1 规则文件受限时的升级通道：SafeFileEditor（含升级全局规则）

当 IDE 内置写入工具无法修改 `.windsurf/rules/*` 时，允许使用 SafeFileEditor 作为**升级通道**，直接在磁盘层安全编辑规则文件（含 `global_rules.md`），用于“认知成长 / 规则固化”。

#### A. SafeFileEditor 的具体功能（面向规则文件）
- **精确定位**：
  - 优先按行号替换（推荐，最稳）
  - 不知道行号时使用关键字/锚点定位（primary + context + end）
- **上下文预览**：替换前自动打印目标区块的前后文，减少误伤。
- **行尾符处理**：自动识别 CRLF/LF，并尽量保持文件原有风格，避免规则文件被意外切换换行符。
- **最小改动**：支持“精确替换”（字符级 diff 思路），减少整段覆盖，降低冲突风险。
- **可验证**：
  - 对代码文件可用 `verify_vars` 校验变量依赖
  - 对规则文件建议用“关键字存在性校验”（修改后必须能搜索到目标关键词/段落标题）

#### B. 升级全局规则（`.windsurf/rules/global_rules.md`）标准流程
1. **先确认 SSOT**：IDE 实际生效规则以 `.windsurf/rules/*` 为准（尤其是 `global_rules.md` 与 `work_flow.md`）。
2. **先备份**：修改前复制一份 `global_rules.md.bak_YYYYMMDD_HHMMSS`（必要时连同 `work_flow.md` 一起备份）。
3. **先定位后替换**：
   - 优先行号替换
   - 行号不稳定时改用“关键字 + 上下文 + 结束关键字”或锚点方式
4. **先预览再落盘**：必须阅读 SafeFileEditor 打印的上下文与预览结果。
5. **最小改动原则**：只固化本次要升级的认知点，避免顺手重排/格式化整文件。
6. **写入后复核**：重新打开规则文件确认关键段落存在；必要时用关键词搜索验证。
7. **回滚策略**：如发现误改，立刻用备份文件覆盖回滚，再重新执行更小粒度替换。

#### C. 推荐命令（Windows / PowerShell 环境）
- **按行号替换（最稳）**：
  `python .windsurf/skills/safe_file_editor/quick_replace.py by-lines "D:\\Git_GitHub\\AgentMemory\\.windsurf\\rules\\global_rules.md" <start> <end> "<new_text>" -y`
- **按关键字替换（不知道行号时）**：
  `python .windsurf/skills/safe_file_editor/quick_replace.py by-keywords "D:\\Git_GitHub\\AgentMemory\\.windsurf\\rules\\global_rules.md" "<primary>" --context "<ctx1,ctx2>" --end "<end>" --code "<new_text>" -y`

> 约束：除非明确要升级换行风格，否则不得主动把 CRLF/LF 批量统一；只接受“保持原样/最小影响”。
### 6.3 强制触发场景（满足任一必须更新代码地图）
- **新对话开始需要快速建立印象**：明确要求“不要扫全库，直接定位”
- **新增/变更入口**：启动脚本、主程序、API 路由、任务调度入口变化
- **新增/重构模块**：目录结构变化、职责边界变化、抽象层变化
- **修复复杂 Bug**：说明当前定位路径/调用链/关键不变量缺失或不可信
- **新增关键配置/环境依赖**：端口、环境变量、部署脚本、第三方服务

### 6.4 代码地图的固定产物（优先维护这些文件）
每次触发时，优先更新 `ProjectMemory/00_索引/` 下的固定地图文件（尽量只改这些，避免信息分散）：
- `AI_快速开始.md`：新对话 AI 的最小读取集合与“禁止全盘扫描”约束
- `入口清单.md`：入口类型 -> 文件路径 -> 关键函数/路由/类
- `调用链地图.md`：关键功能的最短调用链（入口 -> 核心模块 -> DB/外部依赖）
- `定位速查表.md`：按“我想找什么功能/现象”给出路径优先级与搜索关键词
- `关键词索引.md`：关键词 -> 文档/路径（保证可发现性）
- `模块地图.md`：目录级导航入口（必须能点到上述所有地图）

允许增量补充 `01_代码文件/`、`02_关键流程/` 等目录的详细文档，但它们必须能从上述地图文件导航到。

### 6.5 固定流程（先地图，后细节）
1. 先明确“要定位什么”（一句话目标）
2. 写入 `入口清单.md`：入口类型 + 文件路径 + 关键点（函数/路由/类）
3. 写入 `调用链地图.md`：至少 1 条最短调用链
4. 写入 `定位速查表.md`：给出目录优先级与至少 3 个可搜索关键词
5. 更新 `模块地图.md` 与 `关键词索引.md`：保证新内容可被发现
6. 只有当定位仍困难时，再补 `01_代码文件/` 或 `02_关键流程/` 的细节文档

### 6.6 质量门槛（必须满足）
- 至少补 1 条“入口 -> 文件/函数/路由”的可定位信息
- 至少补 1 条“最短调用链”
- 至少补 3 个可搜索关键词（用于后续小范围检索）
- 明确禁止默认全库扫描：必须先走“入口清单/速查表/调用链地图”，再决定是否需要局部搜索
