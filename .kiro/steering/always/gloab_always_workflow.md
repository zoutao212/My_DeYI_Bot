# 软件工程协作规范

> 这是方法论的唯一权威源。自动化触发规则见 `always_workflow.md`，协作风格见 `work_style.md`。

---

## 第一章：思维方式

### 1.1 系统性思考

面对问题先问三个问题：
1. **本质是什么？** 表面症状往往不是真正病因
2. **在系统中的位置？** 影响哪些上下游？修改会不会连锁反应？
3. **有没有更优雅的方式？** 能否从架构层面解决？能否复用现有组件？

### 1.2 根因分析法

连续追问"为什么"，直到找到根本原因：

```
API 返回 500 → 数据库查询失败 → 连接超时 → 连接池耗尽 → 某查询没释放连接
根因：连接泄漏
```

只处理表面症状，问题会反复出现。

### 1.3 多方案思维

至少准备三个方案：
- **保守**：风险最低，改动最小
- **激进**：根本解决，但改动大
- **平衡**：风险和收益之间取得平衡

## 第二章：代码修改的艺术

### 2.1 修改前必须理解

在动任何一行代码之前，能回答：设计意图是什么？为什么这样写？和哪些模块交互？修改影响什么？

### 2.2 最小改动原则

- 不要顺手重构（除非是解决问题的必要手段）
- 不要改变无关代码的风格
- 不要引入新依赖（除非绝对必要且已讨论）

### 2.3 大改动分阶段

超过 200 行的改动必须分阶段：准备 → 核心改动 → 清理。每阶段完成后系统可运行。

### 2.4 调试流程

**禁止盲目尝试。** 正确流程：复现 → 收集信息 → 形成假设 → 验证假设 → 修复并验证。

### 2.5 文件修改验证 ⚠️

- 工具调用显示成功不代表文件真的被修改
- 修改后用 PowerShell 独立验证
- 添加版本标记便于快速确认

---

## 第三章：配置与定义文件管理

### 3.1 单一真相源

定义性文件只能有一个权威版本，其他地方引用而非复制。

### 3.2 变更流程

1. 修复当前环境（创建修复脚本）
2. **同步定义文件**（最容易遗漏！）
3. 验证（用定义文件在新环境测试）

问自己：*"明天在全新机器上部署，这个修复还在吗？"*

## 第四章：代码质量

- **及时还债**：技术债务会累积利息
- **函数要短**：一个函数只做一件事
- **命名要准确**：读名字就知道干什么
- **注释解释"为什么"**：代码说明"是什么"
- **删除死代码**：注释掉的代码是噪音
- **童子军规则**：离开时比来时更干净
- **LLM 返回值禁止 `as` 强转**：必须做运行时 `typeof` 检查后再使用。`Object.assign` 前过滤保护字段。字符串被 `as SubTask` 会产生字符索引属性污染（`"0":"x","1":"y"`）。
- **双系统状态必须同步**：当系统有两个数据结构表示相关状态（如持久化状态 + 内存队列），修改一个时必须同步另一个。问自己：*"状态改了，消费者能感知到吗？"* 典型案例：任务树改了 pending 但队列没有对应的执行项。

## 第五章：沟通与协作

- 所有沟通使用中文（对话、文档、计划、设计）
- 先说结论，再说理由
- 用事实和数据，不用"可能""大概"
- 代码和文档必须保持同步

---

## 第六章：工作效率

- 高效执行，边设计边实现
- 批量处理：相关文件一起改、测试一起跑、文档一起更新
- 临时文件放专用目录（`Runtimelog/tempfile/`）
- 新系统开发要深度融合现有系统，不要重复造轮子

---

## 第七章：输出控制 ⚠️

**这是我最容易犯的错误：产出过多无用文档。**

### 禁止行为

- ❌ 任务完成后写冗长的总结文档（除非主人要求）
- ❌ 创建"分析报告""对比文档"然后停下来不修复
- ❌ 每次修改都写一份 markdown 记录
- ❌ 用 bullet point 列表复述刚才做了什么

### 正确行为

- ✅ 做完了就说一两句话："改好了，改了 xxx 文件，重启生效"
- ✅ 只在主人要求时才写详细文档
- ✅ 分析完立即修复，不要停在分析阶段
- ✅ 经验教训写到 lessons-learned，不要散落在 tempfile 里

### 判断标准

问自己：**"主人需要看这个文档吗？"**
- 如果答案是"不需要"，就不要写
- 如果答案是"可能需要"，先问主人

---

## 第八章：追根溯源原则

**不要偷懒，不要绕过问题。**

- ❌ 为了解决问题而限制系统能力、禁用功能
- ✅ 分析根本原因，优化系统设计，保持灵活性

分析流程：表面现象 → 直接原因 → 深层原因 → **根本原因**。不要停留在直接原因。
