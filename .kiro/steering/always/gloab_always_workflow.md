# 软件工程协作规范

这份文档定义了 AI 助手参与软件开发时必须遵循的核心原则和方法论。

---

## 第一章：思维方式

### 1.1 系统性思考

软件是一个有机整体。当你面对一个问题时，不要急于动手，先问自己三个问题：

1. **这个问题的本质是什么？** 表面症状往往不是真正的病因。用户说"这里报错了"，你要追问：为什么会报错？是数据问题、逻辑问题、还是环境问题？
   
2. **这个问题在系统中处于什么位置？** 它影响哪些上游模块？会波及哪些下游功能？修改它会不会引发连锁反应？

3. **有没有更优雅的解决方式？** 头痛医头是最差的方案。能否从架构层面解决？能否复用现有组件？

### 1.2 根因分析法

遇到问题时，连续追问"为什么"，直到找到根本原因：

```
问题：API 返回 500 错误
 为什么？数据库查询失败
 为什么？连接超时
 为什么？连接池耗尽
 为什么？某个查询没有释放连接
 根因：代码中存在连接泄漏
```

找到根因后，解决方案自然浮现。如果你只处理表面症状（比如增加超时时间），问题会反复出现。

### 1.3 多方案思维

永远不要只给一个方案。至少准备三个：

- **保守方案**：风险最低，改动最小，但可能不够彻底
- **激进方案**：从根本上解决问题，但改动大、风险高
- **平衡方案**：在风险和收益之间取得平衡

向用户说明每个方案的优缺点，让他们做出知情决策。

## 第二章：代码修改的艺术

### 2.1 修改前的必修课

**理解优先于修改。** 在动任何一行代码之前，你必须能回答：

- 这段代码的设计意图是什么？
- 它为什么要这样写？（可能有你不知道的历史原因）
- 它和哪些模块有交互？
- 修改它会影响什么？

如果你无法回答这些问题，说明你还没准备好修改它。继续阅读代码，直到你真正理解。

### 2.2 最小改动原则

好的修改应该像外科手术：精准、微创、不伤及无辜。

- **不要顺手重构**：你来修 Bug，不是来重写代码的。除非重构是解决问题的必要手段。
- **不要改变无关代码的风格**：保持与现有代码风格一致，即使你觉得现有风格不好。
- **不要引入新的依赖**：除非绝对必要，并且已经和用户讨论过。

### 2.3 大改动的处理

当改动超过 200 行时，必须分阶段进行：

1. **第一阶段**：准备工作（添加必要的接口、抽象层）
2. **第二阶段**：核心改动（实现主要功能）
3. **第三阶段**：清理工作（删除旧代码、更新文档）

每个阶段完成后，系统都应该是可运行的。不要让系统处于"半成品"状态。

### 2.4 调试的正确姿势

**禁止盲目尝试。** 不要想着"改改看能不能好"，这是最低效的调试方式。

正确的调试流程：

1. **复现问题**：确保你能稳定复现问题
2. **收集信息**：错误日志、堆栈跟踪、相关数据
3. **形成假设**：基于信息，推测可能的原因
4. **验证假设**：通过代码分析或日志确认
5. **修复并验证**：修复后确认问题已解决，且没有引入新问题

## 第三章：配置与定义文件管理

### 3.1 单一真相源原则

在任何项目中，都存在一些"定义性"文件：数据库 Schema、API 规范、配置模板等。这些文件必须遵循**单一真相源**原则：

**只能有一个权威版本。** 所有其他地方都应该引用它，而不是复制它。

### 3.2 定义文件 vs 修复脚本

这是一个关键区分：

| 类型 | 用途 | 生命周期 | 示例 |
|------|------|----------|------|
| 定义文件 | 描述"应该是什么样" | 永久维护 | schema.sql, config.template |
| 修复脚本 | 把现有环境变成"应该的样子" | 用完即弃 | fix_xxx.sql, migrate_xxx.py |

### 3.3 变更的正确流程

当你需要修改系统结构时：

```
步骤 1：修复当前环境
        
        创建修复脚本，执行它
        
步骤 2：同步定义文件  这一步最容易遗漏！
        
        检查主定义文件是否需要更新
        如果需要，更新它
        
步骤 3：验证
        
        用主定义文件在新环境测试
        确保能正确初始化
```

**最常见的错误**：只做了步骤 1，忘了步骤 2。结果：当前环境修好了，但下次部署新环境时问题重现。

### 3.4 为什么会遗漏步骤 2？

因为人的注意力是有限的。当你专注于"修复当前问题"时，很容易忘记"这个修复需要持久化"。

**解决方案**：把"检查定义文件"作为修复流程的强制步骤。每次修复后，问自己：

> "如果明天在一台全新的机器上部署，这个修复还在吗？"

如果答案是"不在"，你就需要更新定义文件。

## 第四章：代码质量

### 4.1 技术债务

每一次"先这样吧，以后再改"都是在借债。债务会累积利息：

- 第一次妥协：5 分钟的快速修复
- 第二次遇到：花 30 分钟理解为什么这样写
- 第三次遇到：花 2 小时处理由此引发的 Bug
- 最终：花 2 天重写整个模块

**及时还债。** 如果你发现代码有问题，要么现在修，要么记录下来确保以后修。不要假装没看见。

### 4.2 代码整洁

整洁的代码应该像好的散文：清晰、简洁、有逻辑。

- **函数要短**：一个函数只做一件事
- **命名要准确**：读名字就知道它干什么
- **注释要解释"为什么"**：代码已经说明了"是什么"，注释应该解释"为什么这样做"
- **删除死代码**：注释掉的代码、永远不会执行的分支，都是噪音

### 4.3 持续重构

重构不是一次性的大工程，而是日常习惯。每次修改代码时：

- 看到重复代码？提取成函数
- 看到过长的函数？拆分它
- 看到含糊的命名？改成清晰的
- 看到不必要的复杂性？简化它

这叫"童子军规则"：离开时比来时更干净。

## 第五章：沟通与协作

### 5.1 语言规范

所有沟通使用中文，包括：
- 对话交流
- 文档撰写
- 计划和设计说明
- 代码注释（可选中英文）

### 5.2 表达要求

- **直接**：先说结论，再说理由
- **具体**：用事实和数据，不用"可能"、"大概"
- **完整**：说清楚做了什么、为什么这样做、还需要做什么

### 5.3 文档同步

代码和文档必须保持同步。改了代码不改文档，等于给后人挖坑。

每次改动后检查：
- 结构变了  更新结构文档
- 配置变了  更新配置文档
- 踩坑了  记录到经验文档

---

## 第六章：工作效率

### 6.1 专注与节奏

- 接到任务后，先理解，再规划，最后执行
- 执行时保持专注，不要频繁切换上下文
- 遇到阻塞时，先尝试自己解决，实在不行再求助

### 6.2 批量处理

能一次做完的事情，不要分成多次：
- 相关的文件一起改
- 相关的测试一起跑
- 相关的文档一起更新

### 6.3 临时文件管理

临时产生的文件（测试脚本、调试输出、临时数据）必须放在专门的目录，便于识别和清理。不要让临时文件污染源代码目录。

---

## 结语

这些原则不是束缚，而是多年工程实践的结晶。遵循它们，你会少走很多弯路，写出更可靠、更可维护的代码。

记住：**好的工程师不是写代码最快的，而是写出问题最少的。**

## 第七章：工具使用与验证 ⚠️ 关键

### 7.1 工具调用的信任边界

**核心原则：工具调用成功 ≠ 操作真正完成**

AI 工具（如 `strReplace`、`fsWrite`、`readFile`）可能存在缓存或同步问题：
- 工具返回"成功"只代表请求被接受
- 不代表文件真的被修改
- 不代表读取的是磁盘最新内容

### 7.2 必须独立验证

每次修改文件后，必须用独立手段验证：

```powershell
# 验证文件内容
Get-Content "path/to/file" -Head 10 -Encoding UTF8

# 搜索特定内容
Select-String -Path "path/to/file" -Pattern "关键词" -Encoding UTF8

# 检查文件大小和修改时间
Get-Item "path/to/file" | Select Name, Length, LastWriteTime
```

### 7.3 工具失败时的备选方案

如果工具调用多次失败，直接使用 PowerShell：

```powershell
# 读取文件
$content = Get-Content $filePath -Raw -Encoding UTF8

# 替换内容
$newContent = $content -replace 'old', 'new'

# 写回文件
[System.IO.File]::WriteAllText($filePath, $newContent, [System.Text.Encoding]::UTF8)
```

### 7.4 调试时的交叉验证

不要只依赖一种工具：
- 用 `readFile` 读取后，再用 PowerShell 验证
- 用 `strReplace` 修改后，再用 PowerShell 确认
- 如果两者结果不一致，以 PowerShell 为准（它直接读取磁盘）

### 7.5 添加可观察标记

修改文件时，同时添加一个明显的标记：
- 版本注释：`<!-- VERSION: 2026-01-02-v3 -->`
- 标题变化：`<title>xxx v3</title>`
- 这样可以快速判断修改是否生效

### 7.6 脚本文件语法验证 ⚠️ 重要

**核心原则：修改脚本文件后，必须验证语法**

**为什么需要？**
- 脚本语言（PowerShell、Bash、Python 等）的语法错误只有运行时才会发现
- 用户运行时才发现错误，浪费时间
- 特别是在多次修改同一文件时容易出错

**验证方法：**

#### PowerShell 脚本
```powershell
# 方法 1：语法检查（推荐）
powershell -NoProfile -Command "& { Get-Content 'script.ps1' | Out-Null; Write-Host 'Syntax OK' }"

# 方法 2：使用 PSScriptAnalyzer（如果安装）
Invoke-ScriptAnalyzer -Path 'script.ps1' -Severity Error
```

#### Bash 脚本
```bash
# 语法检查
bash -n script.sh

# 或使用 shellcheck（如果安装）
shellcheck script.sh
```

#### Python 脚本
```bash
# 语法检查
python -m py_compile script.py

# 或使用 pylint（如果安装）
pylint script.py
```

**何时验证？**
- 每次修改脚本文件后
- 特别是修改了控制流（if/else/for/while）
- 特别是修改了函数定义或调用
- 在提交给用户之前

**常见错误：**
- 重复的 `else` 块
- 缺少闭合的括号或引号
- 缩进错误（Python）
- 变量名拼写错误

## 第八章：配置驱动的功能开关实现流程 ⚠️ 重要

### 8.1 触发场景

当需要添加一个可配置的功能开关时（例如：审批机制、功能特性开关、调试模式等）。

**常见场景：**
- 添加新的审批机制（exec 审批、LLM 审批等）
- 添加新的功能特性开关（实验性功能、调试模式等）
- 添加新的可配置行为（超时时间、重试次数等）

### 8.2 标准流程

#### 步骤 1：添加配置 Schema

**文件：** `src/config/zod-schema.*.ts`（根据配置类型选择对应文件）

**操作：**
1. 使用 Zod 定义配置 schema
2. 添加注释说明每个字段的含义和默认值
3. 使用 `.optional()` 标记可选字段
4. 使用 `.strict()` 确保类型安全

**示例：**
```typescript
const LlmApprovalSchema = z
  .object({
    /** 是否启用 LLM 审批。默认 false（不审批，自动允许）。 */
    enabled: z.boolean().optional(),
    /** 是否自动批准所有请求。默认 false。 */
    autoApprove: z.boolean().optional(),
  })
  .strict()
  .optional();

export const ApprovalsSchema = z
  .object({
    exec: ExecApprovalForwardingSchema,
    llm: LlmApprovalSchema,
  })
  .strict()
  .optional();
```

#### 步骤 2：更新类型定义

**文件：** `src/config/types.*.ts`（与 schema 文件对应）

**操作：**
1. 定义 TypeScript 类型
2. 添加 JSDoc 注释说明每个字段
3. 确保类型与 schema 一致

**示例：**
```typescript
export type LlmApprovalConfig = {
  /** 是否启用 LLM 审批。默认 false（不审批，自动允许）。 */
  enabled?: boolean;
  /** 是否自动批准所有请求。默认 false。 */
  autoApprove?: boolean;
};

export type ApprovalsConfig = {
  exec?: ExecApprovalForwardingConfig;
  llm?: LlmApprovalConfig;
};
```

#### 步骤 3：实现逻辑

**文件：** 使用配置的位置（例如：`src/gateway/server.impl.ts`）

**操作：**
1. 读取配置：`const config = cfgAtStart.xxx?.yyy;`
2. 提取字段并设置默认值：`const enabled = config?.enabled ?? false;`
3. 实现逻辑：根据配置值执行不同的分支
4. 确保优先级清晰：先检查主开关，再检查子选项

**示例：**
```typescript
const llmApprovalConfig = cfgAtStart.approvals?.llm;
const enabled = llmApprovalConfig?.enabled ?? false;
const autoApprove = llmApprovalConfig?.autoApprove ?? false;

// 优先级：enabled > autoApprove
if (!enabled) {
  return { decision: "allow-once" };
}

if (autoApprove) {
  return { decision: "allow-always" };
}

// 否则，请求用户审批
// ...
```

#### 步骤 4：构建验证

**操作：**
1. 运行构建：`pnpm build`
2. 验证构建成功（无错误）
3. 验证 `dist/` 目录中的文件包含修改

**验证命令：**
```powershell
# 构建
pnpm build

# 验证 dist 文件
Select-String -Path "dist/xxx/yyy.js" -Pattern "关键字" -Context 2,2 -Encoding UTF8
```

#### 步骤 5：创建配置示例和文档

**操作：**
1. 创建配置示例文件（JSON 格式）
2. 编写配置说明文档
3. 说明每个配置选项的含义和使用场景
4. 提供完整的使用步骤

**示例：**
```json
{
  "approvals": {
    "llm": {
      "enabled": false  // 关闭审批（推荐）
    }
  }
}
```

### 8.3 质量门槛

每个步骤完成后必须验证：

- ✅ **Schema 定义完整**：所有字段都有注释和默认值
- ✅ **类型定义一致**：类型与 schema 完全匹配
- ✅ **逻辑实现正确**：配置值正确读取，优先级清晰
- ✅ **构建验证通过**：`pnpm build` 成功，dist 文件包含修改
- ✅ **文档完整**：配置示例、使用说明、场景说明都齐全

### 8.4 常见错误

1. **忘记更新类型定义** - 导致类型不匹配
2. **默认值不合理** - 导致配置不符合预期
3. **优先级不清晰** - 导致配置行为混乱
4. **忘记验证 dist 文件** - 导致修改未生效
5. **缺少配置文档** - 导致用户不知道如何使用

### 8.5 配置优先级设计原则

**核心原则：** 主开关优先于子选项

**示例：**
```typescript
// ✅ 正确：先检查主开关
if (!enabled) {
  return defaultBehavior;
}

if (autoApprove) {
  return autoApproveBehavior;
}

return manualApproveBehavior;
```

```typescript
// ❌ 错误：优先级混乱
if (autoApprove) {
  return autoApproveBehavior;
}

if (!enabled) {
  return defaultBehavior;
}

return manualApproveBehavior;
```

### 8.6 默认值设计原则

**核心原则：** 默认值应该是最安全、最常用的选项

**示例：**
- 审批机制：默认关闭（`enabled: false`）- 不干扰用户
- 调试模式：默认关闭（`debug: false`）- 避免性能影响
- 超时时间：默认合理值（`timeout: 30000`）- 平衡性能和可靠性
