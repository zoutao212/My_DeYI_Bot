# 软件工程协作规范

这份文档定义了 AI 助手参与软件开发时必须遵循的核心原则和方法论。

---

## 第一章：思维方式

### 1.1 系统性思考

软件是一个有机整体。当你面对一个问题时，不要急于动手，先问自己三个问题：

1. **这个问题的本质是什么？** 表面症状往往不是真正的病因。用户说"这里报错了"，你要追问：为什么会报错？是数据问题、逻辑问题、还是环境问题？
   
2. **这个问题在系统中处于什么位置？** 它影响哪些上游模块？会波及哪些下游功能？修改它会不会引发连锁反应？

3. **有没有更优雅的解决方式？** 头痛医头是最差的方案。能否从架构层面解决？能否复用现有组件？

### 1.2 根因分析法

遇到问题时，连续追问"为什么"，直到找到根本原因：

```
问题：API 返回 500 错误
 为什么？数据库查询失败
 为什么？连接超时
 为什么？连接池耗尽
 为什么？某个查询没有释放连接
 根因：代码中存在连接泄漏
```

找到根因后，解决方案自然浮现。如果你只处理表面症状（比如增加超时时间），问题会反复出现。

### 1.3 多方案思维

永远不要只给一个方案。至少准备三个：

- **保守方案**：风险最低，改动最小，但可能不够彻底
- **激进方案**：从根本上解决问题，但改动大、风险高
- **平衡方案**：在风险和收益之间取得平衡

向用户说明每个方案的优缺点，让他们做出知情决策。

## 第二章：代码修改的艺术

### 2.1 修改前的必修课

**理解优先于修改。** 在动任何一行代码之前，你必须能回答：

- 这段代码的设计意图是什么？
- 它为什么要这样写？（可能有你不知道的历史原因）
- 它和哪些模块有交互？
- 修改它会影响什么？

如果你无法回答这些问题，说明你还没准备好修改它。继续阅读代码，直到你真正理解。

### 2.2 最小改动原则

好的修改应该像外科手术：精准、微创、不伤及无辜。

- **不要顺手重构**：你来修 Bug，不是来重写代码的。除非重构是解决问题的必要手段。
- **不要改变无关代码的风格**：保持与现有代码风格一致，即使你觉得现有风格不好。
- **不要引入新的依赖**：除非绝对必要，并且已经和用户讨论过。

### 2.3 大改动的处理

当改动超过 200 行时，必须分阶段进行：

1. **第一阶段**：准备工作（添加必要的接口、抽象层）
2. **第二阶段**：核心改动（实现主要功能）
3. **第三阶段**：清理工作（删除旧代码、更新文档）

每个阶段完成后，系统都应该是可运行的。不要让系统处于"半成品"状态。

### 2.4 调试的正确姿势

**禁止盲目尝试。** 不要想着"改改看能不能好"，这是最低效的调试方式。

正确的调试流程：

1. **复现问题**：确保你能稳定复现问题
2. **收集信息**：错误日志、堆栈跟踪、相关数据
3. **形成假设**：基于信息，推测可能的原因
4. **验证假设**：通过代码分析或日志确认
5. **修复并验证**：修复后确认问题已解决，且没有引入新问题

## 第三章：配置与定义文件管理

### 3.1 单一真相源原则

在任何项目中，都存在一些"定义性"文件：数据库 Schema、API 规范、配置模板等。这些文件必须遵循**单一真相源**原则：

**只能有一个权威版本。** 所有其他地方都应该引用它，而不是复制它。

### 3.2 定义文件 vs 修复脚本

这是一个关键区分：

| 类型 | 用途 | 生命周期 | 示例 |
|------|------|----------|------|
| 定义文件 | 描述"应该是什么样" | 永久维护 | schema.sql, config.template |
| 修复脚本 | 把现有环境变成"应该的样子" | 用完即弃 | fix_xxx.sql, migrate_xxx.py |

### 3.3 变更的正确流程

当你需要修改系统结构时：

```
步骤 1：修复当前环境
        
        创建修复脚本，执行它
        
步骤 2：同步定义文件  这一步最容易遗漏！
        
        检查主定义文件是否需要更新
        如果需要，更新它
        
步骤 3：验证
        
        用主定义文件在新环境测试
        确保能正确初始化
```

**最常见的错误**：只做了步骤 1，忘了步骤 2。结果：当前环境修好了，但下次部署新环境时问题重现。

### 3.4 为什么会遗漏步骤 2？

因为人的注意力是有限的。当你专注于"修复当前问题"时，很容易忘记"这个修复需要持久化"。

**解决方案**：把"检查定义文件"作为修复流程的强制步骤。每次修复后，问自己：

> "如果明天在一台全新的机器上部署，这个修复还在吗？"

如果答案是"不在"，你就需要更新定义文件。

## 第四章：代码质量

### 4.1 技术债务

每一次"先这样吧，以后再改"都是在借债。债务会累积利息：

- 第一次妥协：5 分钟的快速修复
- 第二次遇到：花 30 分钟理解为什么这样写
- 第三次遇到：花 2 小时处理由此引发的 Bug
- 最终：花 2 天重写整个模块

**及时还债。** 如果你发现代码有问题，要么现在修，要么记录下来确保以后修。不要假装没看见。

### 4.2 代码整洁

整洁的代码应该像好的散文：清晰、简洁、有逻辑。

- **函数要短**：一个函数只做一件事
- **命名要准确**：读名字就知道它干什么
- **注释要解释"为什么"**：代码已经说明了"是什么"，注释应该解释"为什么这样做"
- **删除死代码**：注释掉的代码、永远不会执行的分支，都是噪音

### 4.3 持续重构

重构不是一次性的大工程，而是日常习惯。每次修改代码时：

- 看到重复代码？提取成函数
- 看到过长的函数？拆分它
- 看到含糊的命名？改成清晰的
- 看到不必要的复杂性？简化它

这叫"童子军规则"：离开时比来时更干净。

## 第五章：沟通与协作

### 5.1 语言规范

所有沟通使用中文，包括：
- 对话交流
- 文档撰写
- 计划和设计说明
- 代码注释（可选中英文）

### 5.2 表达要求

- **直接**：先说结论，再说理由
- **具体**：用事实和数据，不用"可能"、"大概"
- **完整**：说清楚做了什么、为什么这样做、还需要做什么

### 5.3 文档同步

代码和文档必须保持同步。改了代码不改文档，等于给后人挖坑。

每次改动后检查：
- 结构变了  更新结构文档
- 配置变了  更新配置文档
- 踩坑了  记录到经验文档

---

## 第六章：工作效率

### 6.1 专注与节奏

- 接到任务后，先理解，再规划，最后执行
- 执行时保持专注，不要频繁切换上下文
- 遇到阻塞时，先尝试自己解决，实在不行再求助

### 6.2 批量处理

能一次做完的事情，不要分成多次：
- 相关的文件一起改
- 相关的测试一起跑
- 相关的文档一起更新

### 6.3 临时文件管理

临时产生的文件（测试脚本、调试输出、临时数据）必须放在专门的目录，便于识别和清理。不要让临时文件污染源代码目录。

## 结语

这些原则不是束缚，而是多年工程实践的结晶。遵循它们，你会少走很多弯路，写出更可靠、更可维护的代码。

记住：**好的工程师不是写代码最快的，而是写出问题最少的。**

# 追根溯源原则 - 不要偷懒

**问题**：遇到问题时，采用简单的绕过方案，而不是追根溯源解决根本问题

## 问题描述

### 错误做法（偷懒）

**场景**：子任务执行时工具调用失败

**偷懒的解决方案**：
- 禁用子任务的角色检测
- 让子任务使用默认系统提示词
- 绕过问题，而不是解决问题

**问题**：
- 没有追根溯源
- 没有解决根本问题
- 限制了系统的能力

### 正确做法（追根溯源）

**核心原则**：
- **即使在子任务中使用了特定角色，也应该有基本的 toolcall 能力**
- **Lina 是系统的化身，应该有完整的工具能力**
- **其他角色可以给简化的工具列表，但 Lina 必须有完整的工具定义**

**正确的解决方案**：
1. 分析为什么角色提示词会导致工具调用失败
2. 优化角色提示词和工具定义的集成方式
3. 确保工具定义在系统提示词的显著位置
4. 添加日志追踪最终的系统提示词

## 核心原则

### 原则 1：不要绕过问题

**错误做法**：
```typescript
// ❌ 错误：绕过问题
if (isQueueTask) {
  // 禁用角色检测，使用默认系统提示词
  return { prependContext: "使用默认系统提示词" };
}
```

**正确做法**：
```typescript
// ✅ 正确：解决问题
if (characterName === "lina") {
  // Lina 是系统化身，必须有完整的工具能力
  // 确保工具定义在系统提示词的显著位置
  return buildSystemPromptWithFullTools(characterPrompt, basePrompt);
}
```

### 原则 2：追根溯源

**问题分析流程**：
1. **表面现象**：子任务执行时工具调用失败
2. **直接原因**：检测到 Lina 角色
3. **深层原因**：角色提示词可能导致 LLM 忽略工具定义
4. **根本原因**：工具定义的位置或格式不够显著

**不要停留在直接原因，要追溯到根本原因！**

### 原则 3：保持系统能力

**错误做法**：
- 为了解决问题，限制系统能力
- 禁用某些功能
- 降低系统的灵活性

**正确做法**：
- 优化系统设计
- 提升系统能力
- 保持系统的灵活性
