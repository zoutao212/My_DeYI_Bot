# 自动化触发规则

> 软件工程方法论见 `gloab_always_workflow.md`，本文件只定义自动化触发机制。

---

## 代码地图维护（Project Memory Gardening）

核心原则：先构建可定位的认知（地图），再扩展细节。

### 触发场景

| 场景 | 行动 |
|------|------|
| 新增/变更 API 路由、启动脚本、任务调度入口 | 更新 `ProjectMemory/00_索引/入口清单.md` |
| 目录结构变化、模块职责调整、抽象层变化 | 更新 `模块地图.md` 和 `调用链地图.md` |
| 修复复杂 Bug、发现运行时与代码不一致 | 更新 `调用链地图.md` 和 `定位速查表.md` |
| 用户说"快速定位""更新代码地图""记录入口" | 激活 Power 执行 |

### 质量门槛

每次至少：补 1 条可定位信息 + 1 条最短调用链 + 3 个可搜索关键词，挂载到 `模块地图.md`。

优先级：代码地图 > 详细文档

---

## 规则与技能固化（Maintain Rules）

核心原则：从交互中提炼稳定模式，固化为规则、地图、技能。

### 触发场景

| 场景 | 行动 |
|------|------|
| 同样的流程/步骤/错误重复 >=3 次 | 提炼模式，固化为规则或技能 |
| 路由冲突、配置覆盖、运行时与静态搜索不一致 | 记录到 `定位速查表.md` |
| 修改代码忘更新文档、新入口没记录、问题没固化 | 补充规则到 `.kiro/steering/` |
| 用户说"记住这个套路""固化为规则" | 激活 Power 执行 |

### 固化流程

1. 用 3 句话描述：问题、原因、解法
2. 选形态：规则 / 代码地图 / 技能
3. 写"最小可用"，保持可检索
4. 验证定位效率，写入版本化

### 分工

- 不知道在哪里 -> project-memory-gardening
- 知道在哪但不知道怎么做 -> maintain-rules

---

## 任务完成后自动回顾

每次完成任务后检查：

1. 需要更新代码地图吗？（新入口/调用链/关键词）
2. 出现可复用模式了吗？（重复流程/固定步骤）
3. 需要补充规则吗？（新最佳实践/常见错误）

如果有，激活对应 Power 执行维护。

---

## 效果目标

- 新对话定位时间 < 3 分钟
- 需要读取的文件数 <= 6 个
- 同类问题不再重复出现

---

## 防噪音规则

自动化触发不是每次都要执行。判断标准：

- 改了一行 CSS -> 不需要更新代码地图
- 修了一个小 Bug -> 不需要固化新规则
- 只有重大变更（新模块、新入口、新架构模式）才触发维护
- 不确定要不要触发 -> 不触发。宁可少维护，不要产生噪音文档
