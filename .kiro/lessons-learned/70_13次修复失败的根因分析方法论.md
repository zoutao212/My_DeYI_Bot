# 13 次修复失败的根因分析方法论

**日期**：2026-02-03  
**问题**：vectorengine API 无限循环问题，修复了 13 次都失败

---

## 问题描述

**现象**：LLM 一直重复调用 write 工具，无法停止

**修复次数**：13 次（全部失败）

**失败原因**：
- 第 1-8 次：假设错误的数据格式，在错误的位置修复
- 第 9 次：代码正确，但 `.buildstamp` 没有更新
- 第 10 次：删除 `.buildstamp`，但用户没有重启
- 第 11 次：队列匹配逻辑正确，但 `effectiveProvider` 是空字符串
- 第 12 次：修复 `effectiveProvider`，但问题不在格式转换
- 第 13 次：找到真正的根本原因（工具结果为空），但修复位置可能错误

---

## 根本问题

### 问题 1：没有验证假设

**错误做法**：
- 看到一个可疑点就立即修复
- 假设问题在格式转换、队列匹配、provider 空字符串等
- 没有验证假设是否正确

**正确做法**：
1. **提取实际数据**：从 trace 日志中提取实际的 payload、response、tool result
2. **对比数据**：对比成功和失败的数据，找出真正的差异
3. **验证假设**：确认问题的真正位置，再修复

### 问题 2：没有追踪完整数据流

**错误做法**：
- 只看了部分环节（例如格式转换）
- 没有追踪从工具执行到 LLM 接收的完整流程

**正确做法**：
1. **追踪完整数据流**：
   ```
   工具执行 → 工具结果 → session 保存 → 格式转换 → LLM 接收
   ```
2. **每一步都验证**：
   - 工具执行：返回了什么？
   - 工具结果：保存到 session 了吗？
   - session 保存：格式正确吗？
   - 格式转换：转换后的格式正确吗？
   - LLM 接收：LLM 收到了什么？

### 问题 3：修复位置错误

**错误做法**：
- 在格式转换层修复（中间环节）
- 在 session 保存层修复（中间环节）

**正确做法**：
- **在源头修复**：在工具执行层修复，确保工具返回的数据就是正确的
- **不要打补丁**：不要在中间环节打补丁，这样会导致问题在其他地方重现

### 问题 4：没有添加可观察性

**错误做法**：
- 修复后直接构建
- 相信"构建成功"就代表修复生效
- 没有添加日志验证

**正确做法**：
1. **添加日志**：在关键位置添加日志，验证修复是否生效
2. **验证日志**：重启后检查日志，确认修复代码被执行
3. **不要盲目相信**：不要相信"构建成功"，要看实际的运行日志

## 标准调试流程

### 第一步：提取实际数据

**不要只看代码，要看实际数据！**

```powershell
# 提取 trace 日志
$trace = Get-Content "C:\Users\zouta\.clawdbot\runtimelog\trace__*.jsonl" -Encoding UTF8 | ForEach-Object { $_ | ConvertFrom-Json }

# 提取工具结果
$toolEnd = $trace | Where-Object { $_.event -eq "tool.end" -and $_.payload.toolName -eq "write" }
$toolEnd[0].payload | ConvertTo-Json -Depth 5

# 提取 LLM payload
$llmPayload = $trace | Where-Object { $_.event -eq "llm.payload" }
$llmPayload[0].payload.payload | ConvertTo-Json -Depth 10 | Out-File "payload.json"

# 提取 LLM response
$llmDone = $trace | Where-Object { $_.event -eq "llm.done" }
$llmDone[0].payload.response | ConvertTo-Json -Depth 10 | Out-File "response.json"
```

### 第二步：对比数据

**对比成功和失败的数据，找出真正的差异**

```powershell
# 对比两个 payload
$p1 = Get-Content "payload_success.json" | ConvertFrom-Json
$p2 = Get-Content "payload_failure.json" | ConvertFrom-Json

# 对比 messages 数量
Write-Host "messages 数量: $($p1.messages.Count) vs $($p2.messages.Count)"

# 对比每条消息
for ($i = 0; $i -lt [Math]::Max($p1.messages.Count, $p2.messages.Count); $i++) {
    if ($p1.messages[$i].content -eq $null) {
        Write-Host "⚠️ message[$i].content 是 null"
    }
}
```

### 第三步：追踪完整数据流

**从工具执行到 LLM 接收，每一步都要验证**

1. **工具执行**：
   - 检查 `tool.end` 事件
   - 确认工具返回了什么（result、tail、meta）

2. **session 保存**：
   - 检查 `session.append` 事件（如果有）
   - 确认 toolResult 消息的 content 字段

3. **格式转换**：
   - 检查 `llm.payload` 事件
   - 确认 payload 中的 messages 格式

4. **LLM 接收**：
   - 检查 `llm.done` 事件
   - 确认 LLM 的响应

### 第四步：在源头修复

**不要在中间环节打补丁**

**错误的修复位置**：
- ❌ 格式转换层（`convertOpenAIToGeminiFormat`）
- ❌ session 保存层（`sanitizeSessionHistory`）
- ❌ payload 构建层（`buildPayload`）

**正确的修复位置**：
- ✅ 工具执行层（`wrapToolWithResultFallback`）
- ✅ 工具创建层（`createWriteTool`）

### 第五步：添加可观察性

**添加日志验证修复是否生效**

## 关键教训

### 1. 不要只看代码，要看实际数据

**12 次修复全部失败，因为没有看实际的 trace 数据**

- 实际数据显示 `result` 字段为空
- 但我们一直在修复格式转换、队列匹配等

### 2. 不要假设，要验证

**每次修复都假设问题在某个地方，但没有验证**

- 假设问题在格式转换 → 没有验证
- 假设问题在队列匹配 → 没有验证
- 假设问题在 provider 空字符串 → 没有验证

### 3. 追踪完整数据流

**只看部分环节，没有追踪完整流程**

- 只看了格式转换，没有看工具执行
- 只看了 payload，没有看 response
- 只看了代码，没有看实际数据

### 4. 在源头修复

**在中间环节打补丁，导致问题在其他地方重现**

- 在格式转换层修复 → 问题在工具执行层
- 在 session 保存层修复 → 问题在工具执行层

### 5. 添加可观察性

**没有添加日志，不知道修复是否生效**

- 相信"构建成功"就代表修复生效
- 没有验证修复代码是否被执行
- 没有验证修复后的数据是否正确

## 适用场景

**当遇到以下情况时，使用本方法论：**

1. **修复多次都失败**：修复了 3 次以上都失败
2. **问题难以定位**：不知道问题在哪里
3. **数据流复杂**：涉及多个环节的数据转换
4. **修复无效**：修复后问题仍然存在
