# 字段在数据流中丢失的调试方法论

> **背景**：vectorengine API 无限循环问题，8 次修复全部失败，根本原因是 `toolName` 字段在数据流中丢失

---

## 问题现象

当你遇到以下情况时，可能是字段在数据流中丢失了：

1. ✅ 源数据中有关键字段（例如：session 文件中有 `toolName`）
2. ✅ 修改代码尝试保留字段
3. ❌ 但最终 API 收到的数据中字段丢失或变成默认值（例如：`functionResponse.name` 是 `"unknown"`）
4. ❌ 多次修复都无效

---

## 根本原因

**字段在数据流的某个环节丢失了，但你没有追踪完整的数据流。**

### 典型场景

```
源数据（session 文件）
  ↓ 有字段：toolName: "write"
环节 1（sanitizeSessionHistory）
  ↓ 转换格式：toolResult → tool
  ↓ ❌ 字段丢失：toolName 没有保存到 content 中
环节 2（convertOpenAIToGeminiFormat）
  ↓ 转换格式：tool → functionResponse
  ↓ ❌ 无法提取：从 content 中提取不到 toolName
最终结果
  ↓ functionResponse.name: "unknown"
```

---

## 调试方法论

### 第一步：追踪完整的数据流

**不要只看单个环节，要追踪整个数据流。**

#### 1. 识别数据流的所有环节

```
源数据 → 环节 1 → 环节 2 → 环节 3 → ... → 最终结果
```

**示例**：
```
session 文件
  ↓
sanitizeSessionHistory
  ↓
convertOpenAIToGeminiFormat
  ↓
API payload
  ↓
LLM
```

#### 2. 检查每个环节的输入和输出

**方法**：
- 在每个环节添加日志
- 记录关键字段的值
- 对比输入和输出

**示例**：
```typescript
// 环节 1：sanitizeSessionHistory
log.info(`[sanitize] Input: role=${msg.role}, toolName=${msg.toolName}`);
// ... 处理 ...
log.info(`[sanitize] Output: role=${msg.role}, content=${msg.content}`);

// 环节 2：convertOpenAIToGeminiFormat
log.info(`[format] Input: role=${msg.role}, content=${msg.content}`);
// ... 处理 ...
log.info(`[format] Output: role=${msg.role}, functionResponse.name=${msg.parts[0].functionResponse.name}`);
```

#### 3. 找到字段丢失的环节

**判断标准**：
- 输入有字段，输出没有字段 → 字段在这个环节丢失了
- 输入没有字段，输出也没有字段 → 字段在更早的环节丢失了

---

### 第二步：分析字段丢失的原因

#### 常见原因 1：格式转换时字段被丢弃

**示例**：
```typescript
// ❌ 错误：只转换了 role 和 content，toolName 被丢弃了
msg.role = "tool";
msg.content = extractContent(msg.content);
// toolName 字段没有被保留
```

**修复**：
```typescript
// ✅ 正确：将 toolName 保存到 content 中
msg.role = "tool";
msg.content = JSON.stringify({
  tool: msg.toolName,  // ← 保留关键字段
  result: extractContent(msg.content)
});
```

#### 常见原因 2：字段在副本中，源数据没有更新

**示例**：
```typescript
// ❌ 错误：修改的是副本
const copy = { ...msg };
copy.toolName = "write";
// 源数据 msg 没有被修改
```

**修复**：
```typescript
// ✅ 正确：直接修改源数据
msg.toolName = "write";
```

#### 常见原因 3：字段在中间环节，但下一个环节无法访问

**示例**：
```typescript
// 环节 1：将 toolName 保存到临时变量
const toolName = msg.toolName;
msg.role = "tool";

// 环节 2：无法访问 toolName（因为没有保存到 msg 中）
const name = extractToolName(msg);  // ← 无法提取
```

**修复**：
```typescript
// ✅ 正确：将 toolName 保存到 msg 中
msg.role = "tool";
msg.content = JSON.stringify({
  tool: msg.toolName,  // ← 保存到 content 中
  result: msg.content
});
```

---

### 第三步：修复字段丢失问题

#### 修复原则

1. **字段必须在整个数据流中保留**
   - 不能在中间环节丢失
   - 必须以某种形式传递到下一个环节

2. **选择合适的传递方式**
   - 直接保留字段（如果格式支持）
   - 包装到其他字段中（如果格式不支持）
   - 使用队列匹配（如果顺序固定）

3. **验证每个环节**
   - 添加日志验证字段是否保留
   - 检查输入和输出
   - 确保字段在整个数据流中都存在

#### 修复示例

**场景**：`toolName` 字段在 `sanitizeSessionHistory` → `convertOpenAIToGeminiFormat` 的转换过程中丢失

**修复方案 1：包装到 content 中**
```typescript
// sanitizeSessionHistory
if (msg.role === "toolResult") {
  msg.role = "tool";
  msg.content = JSON.stringify({
    tool: msg.toolName,  // ← 包装到 content 中
    result: msg.content
  });
}

// convertOpenAIToGeminiFormat
if (role === "tool") {
  const content = JSON.parse(msg.content);
  const name = content.tool || "unknown";  // ← 从 content 中提取
  return {
    role: "user",
    parts: [{
      functionResponse: {
        name: name,  // ← 使用提取的 name
        response: content.result
      }
    }]
  };
}
```

**修复方案 2：使用队列匹配**
```typescript
// 维护一个队列
const pendingToolNames: string[] = [];

// 环节 1：记录 toolName
if (msg.role === "assistant" && msg.tool_calls) {
  for (const tc of msg.tool_calls) {
    pendingToolNames.push(tc.function.name);
  }
}

// 环节 2：从队列中匹配
if (msg.role === "tool") {
  const name = pendingToolNames.shift() || "unknown";
  return {
    role: "user",
    parts: [{
      functionResponse: {
        name: name,  // ← 使用队列中的 name
        response: msg.content
      }
    }]
  };
}
```

## 验证方法

### 1. 日志验证

**在每个环节添加日志**：
```typescript
log.info(`[环节名] Input: 关键字段=${值}`);
// ... 处理 ...
log.info(`[环节名] Output: 关键字段=${值}`);
```

**检查日志**：
```powershell
# 搜索日志
Select-String -Path "trace__*.jsonl" -Pattern "关键字段" -Encoding UTF8
```

### 2. Payload 验证

**提取最终 payload**：
```powershell
$trace = Get-Content "trace__*.jsonl" -Encoding UTF8 | ConvertFrom-Json
$payloads = $trace | Where-Object { $_.event -eq "llm.payload" }
$lastPayload = $payloads[-1]

# 检查关键字段
$lastPayload.payload.payload.messages | ForEach-Object {
    if ($_.parts.functionResponse) {
        Write-Host "functionResponse.name: $($_.parts[0].functionResponse.name)"
    }
}
```

### 3. 端到端验证

**完整流程**：
1. 检查源数据（session 文件）
2. 检查每个环节的日志
3. 检查最终 payload
4. 确认字段在整个数据流中都存在

## 预防措施

### 1. 设计时考虑字段保留

**在设计数据转换时，明确列出需要保留的字段**：
- 哪些字段是关键字段？
- 这些字段在转换后如何保留？
- 如果格式不支持，如何传递？

### 2. 添加字段保留测试

**为每个关键字段添加测试**：
```typescript
test("toolName should be preserved in conversion", () => {
  const input = { role: "toolResult", toolName: "write", content: "..." };
  const output = sanitizeSessionHistory([input]);
  
  // 验证 toolName 被保留（可能在 content 中）
  const content = JSON.parse(output[0].content);
  expect(content.tool).toBe("write");
});
```

### 3. 文档化字段传递方式

**在代码注释中说明字段如何传递**：
```typescript
/**
 * 将 toolResult 转换为 OpenAI 格式
 * 
 * 字段传递：
 * - toolName → content.tool（JSON 格式）
 * - content → content.result（JSON 格式）
 * 
 * 这样 convertOpenAIToGeminiFormat 可以从 content 中提取 toolName
 */
```

## 实战案例：vectorengine API 无限循环

### 问题现象

- vectorengine API 不断重复调用同一个工具（write）
- 从 seq=2 到 seq=17+ 无限循环

### 根本原因

- `functionResponse.name` 是 `"unknown"`
- LLM 认为工具结果不匹配，继续调用
- **根本原因**：`toolName` 字段在数据流中丢失了

### 数据流分析

```
session 文件
  ↓ role: "toolResult", toolName: "write"
sanitizeSessionHistory
  ↓ role: "tool", content: "..." （❌ toolName 丢失）
convertOpenAIToGeminiFormat
  ↓ role: "user", functionResponse.name: "unknown" （❌ 无法提取）
API
  ↓ LLM 认为工具结果不匹配
无限循环
```

### 修复方案

**在 sanitizeSessionHistory 中将 toolName 包装到 content 中**：
```typescript
if (msg.role === "toolResult") {
  msg.role = "tool";
  msg.content = JSON.stringify({
    tool: msg.toolName,  // ← 保留 toolName
    result: extractContent(msg.content)
  });
}
```

**为什么有效**：
- `convertOpenAIToGeminiFormat` 会从 `content` 中提取 `tool` 字段
- 代码在第 340 行：`if (typeof respRec.tool === "string") { name = respRec.tool; }`
- 这样 `functionResponse.name` 就是正确的工具名称

### 修复次数

- **8 次修复**，前 7 次全部失败
- 失败原因：没有追踪完整的数据流，没有发现字段丢失

## 关键教训

### 1. 数据流追踪要完整

**不要只看单个环节，要追踪整个数据流。**

- 识别所有环节
- 检查每个环节的输入和输出
- 找到字段丢失的环节

### 2. 字段保留原则

**关键字段必须在整个数据流中保留。**

- 不能在中间环节丢失
- 必须以某种形式传递到下一个环节
- 选择合适的传递方式

### 3. 验证要及时

**每次修改后必须验证字段是否保留。**

- 添加日志验证
- 检查 payload
- 端到端验证

### 4. 多次修复无效时，向上追溯

**如果多次修复都无效（≥3 次），说明问题不在当前环节。**

- 向上追溯数据流
- 找到字段丢失的真正环节
- 在源头修复
