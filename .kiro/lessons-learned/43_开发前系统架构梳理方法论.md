# 开发前系统架构梳理方法论

> **核心原则**：在编写任何新代码之前，必须先理解现有系统的架构和功能，避免重复造轮子。

---

## 问题识别

### 触发信号

当出现以下情况时，说明缺少"开发前系统架构梳理"：

1. **重复实现已有功能**
   - 新代码与现有代码功能重复
   - 不同模块实现了相同的逻辑
   - 示例：TaskManager 重复了 TaskDelegator 的功能

2. **没有正确集成**
   - 新功能独立实现，没有使用现有组件
   - 新模块与现有模块没有正确对接
   - 示例：Lina 系统独立实现，没有使用 Butler 的能力

3. **缺少统一的存储路径**
   - 相似功能的文件散落在不同位置
   - 没有统一的命名规范
   - 示例：角色卡文件没有统一的存储路径

### 根本原因

- **急于实现功能**：看到需求就开始编码，忽略了系统整体设计
- **缺少强制步骤**：没有"开发前必须梳理架构"的强制要求
- **不熟悉现有代码**：对现有系统的功能和架构不了解

## 标准流程

### 第一步：阅读架构文档

**必读文档**：
1. `docs/dev/` - 开发者文档
2. `AGENTS.md` - Agent 配置和规范
3. `README.md` - 项目概述
4. `.kiro/specs/` - 现有的设计文档

**关键问题**：
- 系统的整体架构是什么？
- 有哪些核心组件？
- 各组件的职责是什么？

### 第二步：检查现有功能

**检查清单**：
- [ ] 是否有类似的功能已经实现？
- [ ] 是否有可复用的组件？
- [ ] 是否有相关的工具类或辅助函数？

**检查方法**：
```bash
# 搜索相关关键词
rg "TaskManager|TaskDelegator" src/

# 查看相关目录
ls -la src/agents/butler/
ls -la src/agents/task-board/

# 查看相关文件
cat src/agents/butler/agent.ts
cat src/agents/butler/task-delegator.ts
```

### 第三步：对比功能需求

**对比表格**：

| 功能 | 现有实现 | 新需求 | 是否重复 | 集成方案 |
|------|----------|--------|----------|----------|
| 任务管理 | TaskDelegator | TaskManager | ✅ 重复 | 使用 TaskDelegator |
| 记忆管理 | MemoryService | MemoryManager | ✅ 重复 | 直接使用 MemoryService |
| 提醒管理 | ❌ 未实现 | ReminderManager | ❌ 不重复 | 新增功能 |

### 第四步：设计集成方案

**集成原则**：
1. **优先复用**：能用现有组件就不要重新实现
2. **正确对接**：新功能要正确集成到现有系统
3. **避免重复**：删除重复代码，只保留独有功能

**集成方案示例**：

```typescript
// ❌ 错误：重复实现
class LinaAgent {
  private taskManager: TaskManager;  // 重复了 TaskDelegator
  private memoryManager: MemoryManager;  // 重复了 MemoryService
}

// ✅ 正确：复用现有组件
class LinaAgent extends VirtualWorldAgent {
  constructor(
    private butlerAgent: ButlerAgent,  // 使用 Butler 的能力
    private reminderManager: ReminderManager,  // 独有功能
  ) {
    super("栗娜", LINA_PROFILE, llmProvider);
  }
}
```

**文档模板**：


## 实战案例

### 案例：Lina 日常事务管理系统

**问题**：
- 盲目开发，重复实现了 Butler 的功能
- TaskManager 重复了 TaskDelegator
- MemoryManager 重复了 MemoryService

**梳理过程**：

1. **阅读架构文档**
   - 阅读 `docs/dev/multi-layer-architecture.md`
   - 理解多层 Agent 架构
   - 理解 Butler Agent 的职责

2. **检查现有功能**
   - 发现 `src/agents/butler/task-delegator.ts` 已有任务委托功能
   - 发现 `src/agents/memory/service.ts` 已有记忆管理功能
   - 发现没有提醒管理功能

3. **对比功能需求**
   - TaskManager vs TaskDelegator：功能完全重复
   - MemoryManager vs MemoryService：MemoryManager 是不必要的封装
   - ReminderManager：独有功能，应该保留

4. **设计集成方案**
   - 删除 TaskManager，使用 TaskDelegator
   - 删除 MemoryManager，直接使用 MemoryService
   - 保留 ReminderManager
   - Lina 作为 VirtualWorld Agent，使用 Butler 的能力

5. **创建梳理文档**
   - 创建 `Runtimelog/tempfile/Clawdbot系统架构梳理_20260131.md`
   - 记录重复功能分析
   - 记录推荐集成方案

**结果**：
- ✅ 避免了重复造轮子
- ✅ 正确集成到现有系统
- ✅ 节省了开发时间

## 常见错误

### 错误 1：急于编码

**表现**：
- 看到需求就开始写代码
- 没有检查现有功能
- 没有设计集成方案

**后果**：
- 重复实现已有功能
- 代码无法集成
- 浪费开发时间

**解决**：
- 强制执行"开发前梳理架构"
- 创建梳理文档
- 确认集成方案后再编码

### 错误 2：只看局部

**表现**：
- 只关注要实现的功能
- 不关注系统整体架构
- 不关注与其他模块的关系

**后果**：
- 功能孤立，无法集成
- 破坏系统架构
- 增加维护成本

**解决**：
- 从系统整体视角思考
- 理解各模块的职责和关系
- 设计正确的集成方案

### 错误 3：不创建文档

**表现**：
- 只在脑子里想
- 不记录梳理过程
- 不记录集成方案

**后果**：
- 思路不清晰
- 容易遗漏问题
- 无法回顾和验证

**解决**：
- 创建梳理文档
- 记录对比表格
- 记录集成方案

## 关键教训

1. **开发前必须先梳理架构**
   - 不要急于编码
   - 先理解现有系统
   - 再设计集成方案

2. **检查是否有重复功能**
   - 搜索相关关键词
   - 查看相关目录和文件
   - 对比功能需求

3. **设计集成方案而不是独立实现**
   - 优先复用现有组件
   - 正确对接到现有系统
   - 避免重复造轮子

4. **创建梳理文档**
   - 记录梳理过程
   - 记录功能对比
   - 记录集成方案

5. **获得用户确认**
   - 与用户讨论集成方案
   - 确认没有重复功能
   - 确认符合架构规范
