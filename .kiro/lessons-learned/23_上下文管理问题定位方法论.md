# 上下文管理问题定位方法论

> **触发场景**：AI 对话不连贯、任务目标丢失、反复询问同样的问题

---

## 核心原则

**上下文管理问题的本质是"记忆丢失"。**

AI 需要记住：
- 用户的任务目标
- 已完成的工作
- 遇到的问题
- 下一步计划

如果这些信息丢失，AI 就会：
- 忘记任务目标
- 重复做同样的事
- 反复询问同样的问题

---

## 标准定位流程

### 第一步：识别症状

**从用户反馈中识别症状：**

| 症状 | 说明 | 可能原因 |
|------|------|---------|
| **任务目标丢失** | AI 忘记了用户的任务 | 历史消息被裁剪 |
| **重复询问** | AI 反复问同样的问题 | 记忆检索失效 |
| **工作重复** | AI 重复做同样的事 | 缺少任务状态跟踪 |
| **无法恢复** | 出错后无法继续 | 缺少错误恢复机制 |

### 第二步：检查历史消息裁剪

**定位代码：**
```bash
grepSearch -query "limitHistoryTurns|pruneContextMessages"
```

**检查要点：**

1. **是否保留任务目标？**
   - 第一条用户消息是否被保留？
   - 如果被裁剪，任务目标就会丢失

2. **是否有摘要机制？**
   - 旧消息是否被压缩成摘要？
   - 如果直接丢弃，上下文就会丢失

3. **裁剪阈值是否合理？**
   - 保留多少轮对话？
   - 是否太激进？

**示例问题：**
```typescript
// ❌ 错误：直接丢弃旧消息
export function limitHistoryTurns(messages, limit) {
  // 只保留最后 N 轮，直接丢弃所有更早的消息
  return messages.slice(lastUserIndex);
}

// ✅ 正确：保留任务目标
export function limitHistoryTurns(messages, limit, options) {
  const taskGoal = options?.preserveTaskGoal 
    ? messages.find(m => m.role === "user")
    : null;
  
  const limited = messages.slice(lastUserIndex);
  return taskGoal ? [taskGoal, ...limited] : limited;
}
```

### 第三步：检查记忆系统

**定位代码：**
```bash
grepSearch -query "memory_search|memory_get"
```

**检查要点：**

1. **是否依赖外部 API？**
   - 如果依赖 embedding API，API 不可用时会怎样？
   - 是否有降级方案？

2. **是否有本地降级？**
   - 是否有基于关键词的搜索？
   - 是否有基于文件名的搜索？

3. **错误处理是否完善？**
   - API 失败时是否有明确的错误信息？
   - 是否会静默失败？

**示例问题：**
```typescript
// ❌ 错误：API 不可用时直接返回空
if (!manager) {
  return jsonResult({ results: [], disabled: true, error });
}

// ✅ 正确：降级到关键词搜索
if (!manager) {
  const keywordResults = await keywordSearch({ query, memoryDir });
  return jsonResult({ 
    results: keywordResults, 
    provider: "keyword",
    fallback: true 
  });
}
```

### 第四步：检查任务状态跟踪

**定位代码：**
```bash
grepSearch -query "task.*tracker|task.*state|task.*goal"
```

**检查要点：**

1. **是否有显式的任务状态？**
   - 是否记录了任务目标？
   - 是否记录了已完成的步骤？
   - 是否记录了遇到的问题？

2. **任务状态是否持久化？**
   - 是否保存到文件？
   - 是否注入到系统提示词？

3. **任务状态是否可恢复？**
   - 出错后是否能恢复任务？
   - 是否能从中断处继续？

**示例问题：**
```typescript
// ❌ 错误：没有任务状态跟踪
// 每次请求都是独立的，无法追踪任务进度

// ✅ 正确：显式跟踪任务状态
class TaskTracker {
  private state: TaskState | null = null;
  
  extractGoal(messages: AgentMessage[]): void {
    const firstUser = messages.find(m => m.role === "user");
    if (firstUser) {
      this.state = {
        goal: extractText(firstUser.content),
        progress: [],
        blockers: [],
        nextSteps: [],
      };
    }
  }
  
  generateSummary(): string {
    return `
[当前任务]
目标：${this.state.goal}
已完成：${this.state.progress.join(", ")}
    `.trim();
  }
}
```

### 第五步：检查工具调用恢复

**定位代码：**
```bash
grepSearch -query "tool.*retry|tool.*fallback|executeToolWithRetry"
```

**检查要点：**

1. **是否有重试机制？**
   - 工具调用失败后是否会重试？
   - 重试次数是否合理？

2. **是否有降级策略？**
   - 是否有备选方案？
   - 是否会自动切换？

3. **错误信息是否明确？**
   - 是否告诉用户为什么失败？
   - 是否告诉用户下一步怎么办？

**示例问题：**
```typescript
// ❌ 错误：失败后不重试
try {
  return await toolFn();
} catch (err) {
  throw err;  // 直接抛出错误
}

// ✅ 正确：失败后重试
for (let i = 0; i < maxRetries; i++) {
  try {
    return await toolFn();
  } catch (err) {
    if (i === maxRetries - 1) throw err;
    log.warn(`Retry ${i + 1}/${maxRetries}`);
  }
}
```

## 常见问题模式

### 模式 1：过度裁剪

**表现：**
- 长对话中任务目标丢失
- AI 反复询问"你想做什么"

**根本原因：**
- 历史消息被直接丢弃
- 没有保留任务目标
- 没有摘要机制

**解决方案：**
1. 保留第一条用户消息（任务目标）
2. 添加摘要机制（压缩旧消息）
3. 调整裁剪阈值（保留更多轮）

### 模式 2：记忆系统单点依赖

**表现：**
- 记忆检索失效
- AI 无法回忆之前的对话

**根本原因：**
- 依赖外部 embedding API
- API 不可用时直接返回空
- 没有降级方案

**解决方案：**
1. 添加关键词搜索降级
2. 添加文件名搜索降级
3. 完善错误处理

### 模式 3：缺少任务状态跟踪

**表现：**
- AI 不知道自己在做什么
- 出错后无法恢复

**根本原因：**
- 没有显式的任务状态
- 没有记录已完成的步骤
- 没有记录遇到的问题

**解决方案：**
1. 实现 TaskTracker 类
2. 记录任务目标、进度、问题
3. 将任务状态注入到系统提示词

### 模式 4：工具调用缺少恢复

**表现：**
- 工具调用失败后卡死
- 编码问题导致无法继续

**根本原因：**
- 没有重试机制
- 没有降级策略
- 错误信息不明确

**解决方案：**
1. 实现重试机制
2. 添加降级策略（如切换到 Python 脚本）
3. 完善错误信息

## 验证方法

### 测试用例 1：长对话任务保持

```typescript
it("should preserve task goal in long conversations", async () => {
  const messages = [
    { role: "user", content: "写一个 10000 字的角色卡" },
    ...generateManyMessages(100),
    { role: "user", content: "继续写" },

    preserveTaskGoal: true

});
```

## 总结

上下文管理问题的定位流程：

1. **识别症状**：任务目标丢失、记忆检索失效、工作重复、无法恢复
2. **检查历史消息裁剪**：是否保留任务目标、是否有摘要机制
3. **检查记忆系统**：是否依赖外部 API、是否有降级方案
4. **检查任务状态跟踪**：是否有显式状态、是否持久化
5. **检查工具调用恢复**：是否有重试机制、是否有降级策略

通过系统化的定位方法，可以：
- 快速找到根本原因
- 提出针对性的改进方案
- 避免重复犯错
