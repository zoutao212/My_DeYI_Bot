# 系统提示词过长导致模型行为异常

**日期**：2026-02-02  
**问题**：模型应该使用 functionCall，但返回了文本模拟  
**根因**：系统提示词过长（52KB），导致模型"迷失"

---

## 问题现象

### 症状
- 模型应该使用 function calling 机制调用工具
- 但实际返回了文本模拟工具调用（如 `调用 editor 工具进行写入操作：{ "tool": "editor", ... }`）
- 用户消息中包含了误导性的格式（如 `调用 editor 工具进行写入操作：...`）

### 对比测试结果
| 测试场景 | 系统提示词大小 | 结果 |
|----------|----------------|------|
| 直接 API 测试 | 简短（~1KB） | ✅ 正确返回 functionCall |
| Clawdbot | 52KB | ❌ 返回文本模拟 |

---

## 根本原因

### 1. 系统提示词过长
Clawdbot 的系统提示词包含：
- 大量工具描述（每个工具都有详细说明）
- 详细的使用指南（平台命令规范、工具调用风格等）
- 多个章节（工具、技能、记忆、文档、沙箱、运行时等）
- 总大小：52KB

### 2. 模型"迷失"
当系统提示词过长时：
- 模型在海量信息中"迷失"
- 不知道应该使用 function calling 机制
- 退化为文本模拟工具调用

### 3. 用户消息格式误导
用户消息中包含了类似工具调用的格式：
```
调用 editor 工具进行写入操作：
C:\Users\zouta\clawd\...
```

这进一步误导模型模仿文本格式，而不是使用 functionCall。

## 调试方法

### 标准流程

#### 1. 隔离变量：最小化测试
创建最小化的 API 测试 payload：
```json
{
  "model": "gemini-3-flash-preview",
  "contents": [{"role": "user", "parts": [{"text": "写入 hello 到 test.txt"}]}],
  "systemInstruction": {"parts": [{"text": "你是一个助手。使用 write 工具写入文件。"}]},
  "tools": [{"functionDeclarations": [{"name": "write", "parameters": {...}}]}]
}
```

#### 2. 对比测试
- 直接调用 API（使用 curl 或 PowerShell）
- 对比 Clawdbot 的完整 payload
- 找出差异点

#### 3. 添加调试日志
在 `src/agents/session-tool-result-guard.ts` 中添加：
```typescript
log.info(`[guard] [DEBUG] Full assistant message: ${JSON.stringify(msg, null, 2).slice(0, 2000)}`);
```

#### 4. 验证假设
- 如果直接 API 测试成功 → 问题在 Clawdbot 的 payload
- 如果系统提示词是主要差异 → 问题在系统提示词过长

---

## 解决方案

### 方案 1：精简系统提示词（推荐）

**目标**：将系统提示词从 52KB 减少到 10KB 以内

**精简策略**：
1. **移除冗余的工具描述**
   - 工具 schema 已经包含描述
   - 不需要在系统提示词中重复详细说明

2. **移除过于详细的使用指南**
   - 保留核心指令（如"必须使用 function calling"）
   - 移除详细的示例和说明

3. **合并相似章节**
   - 将多个小章节合并为一个
   - 减少重复内容

4. **移除非关键信息**
   - 移除平台特定的命令指南（可以在需要时动态添加）
   - 移除详细的 CLI 快速参考

**实施位置**：`src/agents/system-prompt.ts`

### 方案 2：动态系统提示词

**思路**：根据上下文动态生成系统提示词
- 基础提示词：核心指令 + 工具列表
- 按需添加：只在需要时添加详细说明

**优点**：
- 保持系统提示词简短
- 根据场景提供必要信息

**缺点**：
- 实现复杂度高
- 需要设计动态添加逻辑

### 方案 3：用户消息格式规范

**思路**：引导用户使用简单自然语言
- ✅ 正确：`写入 hello 到 test.txt`
- ❌ 错误：`调用 editor 工具进行写入操作：{ "tool": "editor", ... }`

**实施**：
- 在 UI 中添加提示
- 在文档中说明正确用法

### 1. 系统提示词大小

```powershell
# 检查系统提示词大小
$prompt = Get-Content "path/to/system-prompt.txt" -Raw -Encoding UTF8

```

**目标**：< 10KB

发送简单指令：
```
写入 hello world 到 test.txt
```

**预期**：
- API 返回 functionCall
- 工具被正确调用
- 文件被成功写入

### 3. 日志验证
查看日志中的 `[DEBUG] Full assistant message`：
```powershell
Get-Content "C:\Users\zouta\.clawdbot\runtimelog\*.jsonl" -Tail 50 | Select-String "DEBUG.*Full assistant message"
```

**预期**：
- 看到 `functionCall` 字段
- 没有文本模拟

## 关键教训

### 1. 系统提示词不是越详细越好
- 过长的系统提示词会导致模型"迷失"
- 保持简洁，只包含核心指令

### 2. 最小化测试是定位问题的最快方法
- 用最简单的 payload 测试 API
- 逐步添加复杂度，找出问题点

### 3. 添加调试日志比盲目猜测更有效
- 记录 API 返回的原始数据结构
- 记录关键处理步骤

### 4. 用户消息格式很重要
- 避免在消息中包含类似工具调用的格式
- 使用简单自然语言

### 5. 对比测试是验证假设的关键
- 直接 API 测试 vs Clawdbot 测试
- 找出差异点，验证假设

---

## 适用场景

当遇到以下情况时，考虑系统提示词过长问题：

- ✅ 工具调用失败，但 API 测试成功
- ✅ 系统提示词超过 20KB
- ✅ 模型返回文本模拟而不是 functionCall
- ✅ 直接 API 测试和 Clawdbot 测试结果不一致
