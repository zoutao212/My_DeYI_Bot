# 工具调用验证与重试机制

> **核心教训**：工具调用返回"成功"不代表操作真的完成，必须验证实际效果。

---

## 问题现象

**触发场景**：
- 调用 write 工具写入文件，返回"成功"，但文件没有被写入
- 调用 exec 工具执行命令，返回"成功"，但命令没有执行
- 调用 read 工具读取文件，返回内容，但内容是旧的（缓存）

**症状**：
- 用户以为操作成功了，但实际失败了
- 后续操作基于错误的假设，导致连锁错误
- 调试时发现"工具调用成功但结果不对"

---

## 根本原因

### 1. 工具调用流程的假设错误

**错误假设**：
```
工具调用返回"成功" = 操作真的完成
```

**实际情况**：
```
工具调用返回"成功" = 工具代码执行完毕（不代表操作生效）
```

**可能的原因**：
- **缓存问题**：工具读取的是缓存，不是磁盘最新内容
- **文件系统延迟**：写入操作还在缓冲区，没有刷新到磁盘
- **权限问题**：操作被拒绝，但工具没有检测到
- **路径问题**：操作了错误的文件/目录
- **工具实现问题**：工具本身有 bug

### 2. 缺少验证机制

**当前流程**：
```
LLM 调用工具 → 工具返回结果 → 系统认为完成
```

**缺少的环节**：
```
LLM 调用工具 → 工具返回结果 → [验证实际效果] → 系统认为完成
```

### 3. 缺少重试机制

**当前行为**：
- 工具调用失败 → 直接报错 → 放弃

**应该的行为**：
- 工具调用失败 → 判断是否可重试 → 重试 → 成功/失败

---

## 解决方案

### 方案 1：添加工具调用验证机制（短期）

**原则**：每次工具调用后，用独立的方法验证结果。

#### 对于文件写入（write 工具）

**验证方法**：
```powershell
# 写入后立即读取验证
$content = Get-Content "path/to/file" -Raw -Encoding UTF8
if ($content -match "预期内容") {
    Write-Host "✓ 文件写入成功"
} else {
    Write-Host "✗ 文件写入失败"
}
```

**实现位置**：
- 在 `src/agents/pi-tools.read.ts` 中添加验证逻辑
- 在工具调用后自动执行验证

#### 对于命令执行（exec 工具）

**验证方法**：
```powershell
# 检查退出码
if ($LASTEXITCODE -eq 0) {
    Write-Host "✓ 命令执行成功"
} else {
    Write-Host "✗ 命令执行失败，退出码：$LASTEXITCODE"
}

# 检查输出
if ($output -match "预期输出") {
    Write-Host "✓ 输出符合预期"
} else {
    Write-Host "✗ 输出不符合预期"
}
```

**实现位置**：
- 在 `src/agents/bash-tools.exec.ts` 中添加验证逻辑
- 在命令执行后自动检查退出码和输出

#### 对于文件读取（read 工具）

**验证方法**：
```powershell
# 检查文件修改时间
$lastWrite = (Get-Item "path/to/file").LastWriteTime
if ($lastWrite -gt $cacheTime) {
    Write-Host "✓ 读取的是最新内容"
} else {
    Write-Host "⚠️ 可能读取的是缓存"
}
```

**实现位置**：
- 在 `src/agents/pi-tools.read.ts` 中添加缓存检测
- 在读取前检查文件修改时间

### 方案 2：添加简单的重试机制（中期）

**原则**：对于可重试的错误，自动重试。

#### 错误分类

**可重试错误**：
- 网络超时（ETIMEDOUT）
- 文件被占用（EBUSY）
- 资源暂时不可用（EAGAIN）
- 临时性错误

**不可重试错误**：
- 文件不存在（ENOENT）
- 权限不足（EACCES）
- 参数错误（EINVAL）
- 永久性错误

#### 重试策略

```typescript
// 伪代码
async function executeToolWithRetry(tool, params, options = {}) {
  const maxRetries = options.maxRetries || 3;
  const baseDelay = options.baseDelay || 1000;
  
  let lastError;
  
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      // 执行工具
      const result = await tool.execute(params);
      
      // 验证结果
      const verified = await verifyToolResult(tool.name, params, result);
      
      if (verified) {
        return result;
      }
      
      // 验证失败，视为错误
      throw new Error(`Tool ${tool.name} execution failed verification`);
      
    } catch (err) {
      lastError = err;
      
      // 判断是否可重试
      if (!isRetryableError(err)) {
        throw err;
      }
      
      // 最后一次尝试，不再重试
      if (attempt === maxRetries - 1) {
        break;
      }
      
      // 等待后重试（指数退避）
      const delay = baseDelay * Math.pow(2, attempt);
      await sleep(delay);
      
      log.warn(`[tool-retry] Retrying ${tool.name} (attempt ${attempt + 2}/${maxRetries})`);
    }
  }
  
  throw lastError;
}
```

**实现位置**：
- 创建 `src/agents/tool-retry.ts` 文件
- 在 `src/agents/pi-embedded-runner/run/attempt.ts` 中集成

### 方案 3：改进错误提示（短期）

**原则**：提供详细的错误信息和修复建议。

#### 错误信息格式

```typescript
interface ToolError {
  tool: string;
  params: unknown;
  error: string;
  suggestion: string;
  retryable: boolean;
  attempts?: number;
}
```

#### 错误建议示例

```typescript
function getErrorSuggestion(tool: string, error: Error): string {
  if (tool === 'write' && error.code === 'EACCES') {
    return '文件权限不足，请检查文件权限或使用管理员权限运行';
  }
  
  if (tool === 'exec' && error.code === 'ETIMEDOUT') {
    return '命令执行超时，可能是网络问题或命令耗时过长，建议增加超时时间或检查网络';
  }
  
  if (tool === 'read' && error.code === 'ENOENT') {
    return '文件不存在，请检查文件路径是否正确';
  }
  
  return '请检查参数是否正确，或查看详细错误信息';
}
```

**实现位置**：
- 创建 `src/agents/tool-errors.ts` 文件
- 在所有工具中使用统一的错误格式

## 验证方法

### 验证验证机制

**测试场景**：
1. 写入文件后立即读取，检查内容是否正确
2. 执行命令后检查退出码和输出
3. 读取文件时检查是否是最新内容

**预期结果**：
- 如果操作失败，能够立即检测到
- 不会出现"工具调用成功但结果不对"的情况

### 验证重试机制

**测试场景**：
1. 模拟网络超时错误，检查是否自动重试
2. 模拟文件被占用错误，检查是否自动重试
3. 模拟权限不足错误，检查是否不重试

**预期结果**：
- 可重试错误会自动重试
- 不可重试错误会立即报错
- 重试次数符合预期

### 验证错误提示

**测试场景**：
1. 触发各种错误，检查错误提示是否详细
2. 检查是否提供了修复建议
3. 检查是否标记了是否可重试

**预期结果**：
- 错误提示详细且友好
- 提供了有用的修复建议
- 用户能够根据提示解决问题

---

## 关键教训

1. **工具调用不是"调用即成功"**
   - 必须验证实际效果
   - 不能只看返回值

2. **验证机制是必须的**
   - 每次工具调用后都要验证
   - 用独立的方法验证，不依赖工具本身

3. **重试机制可以大幅提升成功率**
   - 很多错误是临时的
   - 自动重试可以避免用户手动重试

4. **错误提示要友好**
   - 详细的错误信息
   - 有用的修复建议
   - 明确是否可重试

5. **分层验证**
   - 工具层验证：工具本身的错误处理
   - 调用层验证：调用后的结果验证
   - 系统层验证：系统级的监控和日志
