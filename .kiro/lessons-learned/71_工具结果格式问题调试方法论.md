# 工具结果格式问题调试方法论

> **场景**：工具结果被发送给 LLM，但 LLM 没有理解或没有使用工具结果

---

## 问题识别

### 典型症状

1. **LLM 不断重复调用相同的工具**
2. **LLM 的 thinking 没有变化**（说明 LLM 没有"看到"工具结果）
3. **LLM 说"I've begun"而不是"I've completed"**（说明 LLM 认为任务还没开始）
4. **工具确实被执行了，结果也被发送了**（从 trace 日志可以确认）

### 与其他问题的区别

| 问题类型 | LLM thinking | 工具执行 | 工具结果发送 |
|---------|-------------|---------|-------------|
| **工具未执行** | 正常变化 | ❌ 未执行 | ❌ 未发送 |
| **工具结果未发送** | 正常变化 | ✅ 已执行 | ❌ 未发送 |
| **工具结果格式错误** | ⚠️ 不变化 | ✅ 已执行 | ✅ 已发送 |
| **系统提示词问题** | ⚠️ 不变化 | ✅ 已执行 | ✅ 已发送 |

**关键区分**：如果工具执行了、结果也发送了，但 LLM thinking 不变化，说明是**格式问题**。

---

## 调试流程

### 第一步：确认工具结果确实被发送

```powershell
# 提取 llm.payload 事件
$trace = Get-Content "trace.jsonl" -Encoding UTF8 -Raw
$lines = $trace -split "`n"
$payloadLine = $lines | Where-Object { $_ -match '"event":"llm\.payload"' -and $_ -match '"seq":2' } | Select-Object -First 1

$json = $payloadLine | ConvertFrom-Json
$messages = $json.payload.payload.messages

# 检查是否有 functionResponse
foreach ($msg in $messages) {
    if ($msg.role -eq "user" -and $msg.parts) {
        foreach ($part in $msg.parts) {
            if ($part.functionResponse) {
                Write-Host "✓ 找到 functionResponse: $($part.functionResponse.name)"
            }
        }
    }
}
```

**判断标准**：
- ✅ 如果找到 functionResponse，说明工具结果被发送了
- ❌ 如果没有找到，说明是发送问题，不是格式问题

### 第二步：检查 LLM 的 thinking 是否变化

```powershell
# 提取 llm.done 事件
$llmDone = $trace | Where-Object { $_.event -eq "llm.done" }

for ($i = 0; $i -lt $llmDone.Count; $i++) {
    $event = $llmDone[$i]
    $samples = $event.payload.responseSummary.samples
    $doneEvent = $samples | Where-Object { $_.type -eq "done" }
    
    foreach ($content in $doneEvent.message.content) {
        if ($content.type -eq "thinking") {
            Write-Host "seq=$($event.payload.seq) thinking: $($content.thinking.Substring(0, 100))..."
        }
    }
}
```

**判断标准**：
- ✅ 如果 thinking 变化了（例如从"I've begun"变成"I've completed"），说明 LLM 理解了工具结果
- ❌ 如果 thinking 完全相同，说明 LLM 没有理解工具结果

### 第三步：检查工具结果的完整格式

```powershell
# 提取第一个 functionResponse 的完整结构
$msg = $messages | Where-Object { $_.role -eq "user" -and $_.parts[0].functionResponse } | Select-Object -First 1
$fr = $msg.parts[0].functionResponse

Write-Host "=== functionResponse 完整结构 ===" -ForegroundColor Cyan
$fr | ConvertTo-Json -Depth 10 | Write-Host
```

**检查清单**：
- [ ] `name` 字段是否正确？
- [ ] `response` 字段是什么类型？（对象、字符串、数组？）
- [ ] `response` 是否有嵌套结构？（例如 `{result: "..."}`）
- [ ] 是否有额外的字段？（例如 `thought_signature`、`thoughtSignature`）
- [ ] `response` 内容是否包含元数据？（例如 "Read text file [auto]"）

### 第四步：对比 API 官方文档

**Gemini API 的 functionResponse 格式**：
```json
{
  "role": "user",
  "parts": [{
    "functionResponse": {
      "name": "tool_name",
      "response": {
        // 工具返回的任意 JSON 对象
      }
    }
  }]
}
```

**OpenAI API 的 tool 格式**：
```json
{
  "role": "tool",
  "tool_call_id": "call_xxx",
  "content": "工具返回的字符串或 JSON"
}
```

**对比检查**：
- [ ] 我们的格式是否符合 API 文档？
- [ ] 是否有多余的字段？
- [ ] 是否有缺失的字段？

### 第五步：逐步简化格式

**策略**：从最简单的修复开始，逐步深入

#### 修复 1：移除干扰字段（最简单）

**目标**：移除 `thought_signature`、`thoughtSignature` 等干扰字段

**修改位置**：`src/agents/session-tool-result-guard.ts`

```typescript
if ("functionResponse" in rec) {
  const functionResponse = rec.functionResponse as Record<string, unknown>;
  
  // 移除 signature 字段
  delete functionResponse.thought_signature;
  delete functionResponse.thoughtSignature;
  
  // 移除 response 中的 signature 字段
  if (functionResponse.response && typeof functionResponse.response === "object") {
    const response = functionResponse.response as Record<string, unknown>;
    delete response.thought_signature;
    delete response.thoughtSignature;
  }
}
```

#### 修复 2：简化 response 结构（中等难度）

**目标**：移除 `{result: "..."}` 包装，直接使用内容

**修改位置**：`src/agents/gemini-payload-thought-signature.ts`

```typescript
// 转换 tool 消息时
let response = {};
if (typeof content === "string") {
  try {
    const parsed = JSON.parse(content);
    // 如果是 {result: "..."} 格式，提取 result
    if (parsed && typeof parsed === "object" && "result" in parsed) {
      response = parsed.result;  // 直接使用 result 内容
    } else {
      response = parsed;
    }
  } catch (e) {
    response = content;
  }
}
```

#### 修复 3：清理元数据（最复杂）

**目标**：移除 "Read text file [auto]" 等元数据

**修改位置**：格式转换层

```typescript
if (typeof response === "string") {
  // 移除 "Read text file [auto]" 前缀
  response = response.replace(/^Read (?:text|image) file \[.*?\]\n\n/, "");
  
  // 移除 YAML frontmatter
  response = response.replace(/^---\n.*?\n---\n\n/s, "");
}
```

## 验证方法

### 验证 1：检查 functionResponse 格式

```powershell
# 重新提取 payload
$payloadLine = $lines | Where-Object { $_ -match '"event":"llm\.payload"' -and $_ -match '"seq":2' } | Select-Object -First 1
$json = $payloadLine | ConvertFrom-Json
$fr = $json.payload.payload.messages[3].parts[0].functionResponse

# 检查是否还有 signature 字段
if ($fr.thought_signature -or $fr.thoughtSignature) {
    Write-Host "❌ 还有 signature 字段" -ForegroundColor Red
} else {
    Write-Host "✓ 没有 signature 字段" -ForegroundColor Green
}
```

### 验证 2：检查 LLM thinking 是否变化

```powershell
# 提取 seq=1 和 seq=2 的 thinking
$seq1Thinking = ...
$seq2Thinking = ...

if ($seq1Thinking -eq $seq2Thinking) {
    Write-Host "❌ thinking 没有变化" -ForegroundColor Red
} else {
    Write-Host "✓ thinking 变化了" -ForegroundColor Green
}
```

### 验证 3：检查 LLM 是否停止重复调用

```powershell
# 统计工具调用次数
$toolCalls = $trace | Where-Object { $_.event -eq "tool.start" }
$readCalls = $toolCalls | Where-Object { $_.payload.name -eq "read" }

Write-Host "read 工具调用次数: $($readCalls.Count)"

# 如果 > 4 次，说明还在重复
if ($readCalls.Count -gt 4) {
    Write-Host "❌ 还在重复调用" -ForegroundColor Red
} else {
    Write-Host "✓ 不再重复调用" -ForegroundColor Green
}
```

## 常见错误模式

### 错误 1：只检查工具执行，不检查 LLM thinking

**问题**：工具执行成功，但 LLM 没有理解结果

**正确做法**：
1. 先检查工具是否执行
2. 再检查工具结果是否发送
3. **最后检查 LLM thinking 是否变化**

### 错误 2：只修改一个地方，不检查整个数据流

**问题**：修改了格式转换层，但 session 保存层又加回了干扰字段

**正确做法**：
1. 追踪完整数据流：工具执行 → 格式转换 → session 保存 → payload 构建
2. 在每个环节都检查格式
3. 确保修复在整个数据流中生效

### 错误 3：盲目修改格式，不验证 API 文档

**问题**：修改后的格式不符合 API 要求，导致 API 报错

**正确做法**：
1. 先查看 API 官方文档
2. 对比当前格式和官方格式
3. 只修改不符合的部分
4. 修改后验证 API 是否接受

---

## 关键教训

1. **工具结果被发送 ≠ LLM 理解了工具结果**
   - 必须检查 LLM thinking 是否变化
   - 必须检查 LLM 是否停止重复调用

2. **格式问题可能在多个环节**
   - 工具返回格式
   - 格式转换层
   - session 保存层
   - payload 构建层

3. **干扰字段可能很隐蔽**
   - signature 字段
   - 元数据前缀
   - 嵌套结构

4. **修复要从简单到复杂**
   - 先移除干扰字段
   - 再简化结构
   - 最后清理内容

5. **验证要全面**
   - 验证格式
   - 验证 LLM thinking
   - 验证行为

## 实战案例

### 案例：vectorengine 工具结果格式问题

**问题**：LLM 不断重复调用 read 工具

**调试过程**：
1. 确认工具执行了 ✅
2. 确认工具结果发送了 ✅
3. 检查 LLM thinking：完全相同 ❌
4. 检查 functionResponse 格式：包含 `thought_signature` 和 `thoughtSignature` ❌
5. 修复：移除 signature 字段
6. 验证：LLM thinking 变化了 ✅

**修复位置**：`src/agents/session-tool-result-guard.ts`

**修复内容**：在保存 functionResponse 到 session 之前，移除所有 signature 字段

**结果**：LLM 不再重复调用工具
