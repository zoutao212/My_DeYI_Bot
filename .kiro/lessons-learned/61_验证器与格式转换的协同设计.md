# 验证器与格式转换的协同设计原则

> **背景**：vectorengine API 兼容性问题调试，发现验证器只支持单一格式导致误报

---

## 问题场景

当系统需要支持多种 API 格式时（例如 OpenAI 和 Gemini），如果验证器只支持单一格式，会导致：

1. **误报**：验证器报告缺少字段，但实际上字段存在（只是格式不同）
2. **调试困难**：日志显示格式转换成功，但验证器报告错误
3. **维护成本高**：每次添加新格式都要修改验证器

### 典型表现

```
[format] Converting OpenAI format to Gemini format  ← 格式转换成功
[payload-validator] ⚠️  messages[2].tool_calls[0]: missing 'thought_signature'  ← 验证器误报
```

**原因**：验证器在检查 `tool_calls`（OpenAI 格式），但此时 payload 已经被转换为 Gemini 格式（`functionCall`）。

---

## 根本原因

### 1. 验证器设计时只考虑了单一格式

**错误做法**：
```typescript
// 只检查 OpenAI 格式
if (role === "assistant") {
  if (m.tool_calls && Array.isArray(m.tool_calls)) {
    // 检查 tool_calls
  }
}
```

**问题**：当 payload 被转换为 Gemini 格式后，验证器找不到 `tool_calls`，报告误报。

### 2. 验证器与格式转换的执行顺序不明确

**执行流程**：
1. `geminiPayloadThoughtSignaturePatcher.nextOnPayload`（格式转换）
2. `llmCallConsoleLogger.nextOnPayload`（验证）

**问题**：如果验证器在格式转换之前执行，会看到转换前的格式；如果在格式转换之后执行，会看到转换后的格式。

### 3. 验证器没有考虑格式转换的场景

**设计缺陷**：验证器假设 payload 格式是固定的，没有考虑格式转换的场景。

---

## 解决方案

### 原则 1：验证器必须支持所有可能的格式

**正确做法**：
```typescript
// 同时支持 OpenAI 和 Gemini 格式
if (role === "assistant" || role === "model") {
  // 检查 OpenAI 格式
  if (m.tool_calls && Array.isArray(m.tool_calls)) {
    // 检查 tool_calls
  }
  // 检查 Gemini 格式
  else if (m.parts && Array.isArray(m.parts)) {
    // 检查 parts[].functionCall
  }
}
```

**关键点**：
- 使用 `if-else if` 结构，根据实际格式选择验证逻辑
- 不要假设只有一种格式
- 支持所有可能的格式变体

### 原则 2：验证器的执行时机必须明确

**推荐做法**：验证器在格式转换之后执行

**原因**：
- 格式转换可能添加字段（如 `thought_signature`）
- 验证器应该验证最终发送给 API 的格式
- 避免在格式转换前验证导致误报

**实现**：
```typescript
const nextOnPayload = (payload: unknown) => {
  // 1. 先调用下一个 wrapper（格式转换）
  options?.onPayload?.(payload);
  
  // 2. 再验证（验证转换后的格式）
  const validation = validateAndLogPayload({ payload, ... });
};
```

### 原则 3：验证器的设计要考虑格式转换

**设计要点**：
1. **格式检测**：根据 payload 中的字段判断格式
2. **双格式支持**：同时支持 OpenAI 和 Gemini 格式
3. **字段位置灵活**：字段可能在不同位置（如 `thought_signature` 可能在 wrapper 或内部对象）

**示例**：
```typescript
// 检查 Gemini 格式的 thought_signature
if (hasParts) {
  const parts = m.parts as unknown[];
  for (let j = 0; j < parts.length; j++) {
    const part = parts[j];
    if (part && typeof part === "object") {
      const p = part as Record<string, unknown>;
      if (p.functionCall && typeof p.functionCall === "object") {
        const fc = p.functionCall as Record<string, unknown>;
        // 检查 wrapper 或内部对象
        if (!fc.thought_signature && !fc.thoughtSignature && 
            !p.thought_signature && !p.thoughtSignature) {
          warnings.push(`missing 'thought_signature'`);
        }
      }
    }
  }
}
```

## 实战案例：vectorengine API

### 问题

vectorengine 使用 OpenAI endpoint (`/v1/chat/completions`) 但期望 Gemini 格式的 payload。

**数据流**：
1. 系统生成 OpenAI 格式的 payload
2. `geminiPayloadThoughtSignaturePatcher` 将其转换为 Gemini 格式
3. `llmCallConsoleLogger` 验证 payload
4. 发送给 API

**问题**：验证器只支持 OpenAI 格式，报告 `tool_calls` 缺少 `thought_signature`，但实际上 payload 已经被转换为 Gemini 格式（`functionCall`）。

### 修复

**文件 1：`src/agents/payload-validator.ts`**

扩展验证逻辑，同时支持 OpenAI 和 Gemini 格式：

```typescript
// 支持 role: function（Gemini 格式）
if (role === "tool" || role === "function") {
  const hasToolCallId = m.tool_call_id && typeof m.tool_call_id === "string";
  const hasParts = Array.isArray(m.parts);
  
  if (!hasToolCallId && !hasParts) {
    errors.push(`role=${role} requires 'tool_call_id' (OpenAI) or 'parts' (Gemini)`);
  }
  
  // 检查 Gemini 格式的 thought_signature
  if (hasParts) {
    // 检查 parts[].functionResponse
  }
}

// 支持 role: model（Gemini 格式）
else if (role === "assistant" || role === "model") {
  // 检查 OpenAI 格式
  if (m.tool_calls && Array.isArray(m.tool_calls)) {
    // ...
  }
  // 检查 Gemini 格式
  else if (m.parts && Array.isArray(m.parts)) {
    // 检查 parts[].functionCall
  }
}
```

**文件 2：`src/agents/llm-call-console-log.ts`**

确认验证器在格式转换之后执行：

```typescript
const nextOnPayload = (payload: unknown) => {
  // 1. 先调用下一个 wrapper（格式转换）
  options?.onPayload?.(payload);
  
  // 2. 再验证（验证转换后的格式）
  const validation = validateAndLogPayload({ payload, ... });
};
```

### 效果

- ✅ 不再有误报
- ✅ 验证器同时支持 OpenAI 和 Gemini 格式
- ✅ 未来添加新格式时，只需扩展验证器

## 调试方法

### 方法 1：查看验证警告的字段名

```
messages[2].tool_calls[0]: missing 'thought_signature'  ← OpenAI 格式
messages[2].parts[0].functionCall: missing 'thought_signature'  ← Gemini 格式
```

如果验证器报告 `tool_calls`，但格式已经转换为 Gemini，说明验证器不支持 Gemini 格式。

### 方法 2：查看日志顺序

```
[format] Converting OpenAI format to Gemini format  ← 格式转换
[payload-validator] ⚠️  messages[2].tool_calls[0]  ← 验证器看到的格式
```

如果格式转换在前，验证在后，但验证器报告的是转换前的格式，说明验证器不支持转换后的格式。

### 方法 3：检查验证器代码

查看验证器是否同时支持多种格式：

```typescript
// ❌ 只支持单一格式
if (role === "assistant") {
  if (m.tool_calls) { /* ... */ }
}

// ✅ 支持多种格式
if (role === "assistant" || role === "model") {
  if (m.tool_calls) { /* OpenAI */ }
  else if (m.parts) { /* Gemini */ }
}
```

## 设计检查清单

在设计验证器时，必须检查：

- [ ] **格式支持**：验证器是否支持所有可能的格式？
- [ ] **执行时机**：验证器是在格式转换之前还是之后执行？
- [ ] **字段位置**：字段可能在哪些位置？（wrapper、内部对象、数组元素）
- [ ] **格式检测**：如何判断 payload 是哪种格式？
- [ ] **误报处理**：如何避免因格式不同导致的误报？
- [ ] **扩展性**：未来添加新格式时，是否容易扩展？

---

## 关键教训

### 1. 验证器必须支持所有格式

当系统支持多种 API 格式时，验证器必须同时支持所有格式，否则会产生误报。

### 2. 验证器的执行时机很重要

- 在格式转换之前验证：看到转换前的格式
- 在格式转换之后验证：看到转换后的格式

**推荐**：在格式转换之后验证，因为格式转换可能添加字段。

### 3. 不要假设只有一种格式

即使当前只支持一种格式，也要考虑未来可能添加新格式。设计时要留有扩展空间。

### 4. 格式检测要灵活

根据 payload 中的实际字段判断格式，不要硬编码格式假设。

### 5. 字段位置要灵活

字段可能在不同位置（wrapper、内部对象、数组元素），验证器要能处理所有情况。

---

## 适用场景

本原则适用于以下场景：

1. **多 API 格式支持**：系统需要支持多种 API 格式（OpenAI、Gemini、Anthropic 等）
2. **格式转换**：系统需要在不同格式之间转换
3. **Payload 验证**：需要验证 payload 格式是否正确
4. **中转 API**：使用标准 endpoint 但期望特定格式的 payload
