# LLM 角色扮演与工具调用的区分

**触发场景**：用户期望 LLM 执行某个操作（如写入文件、执行命令），但 LLM 只是在文本中"描述"操作

**核心价值**：帮助用户理解 LLM 的文本生成和实际工具调用的区别

---

## 问题识别

### 典型症状

1. **用户期望操作已完成**：
   - 用户：请写入文件到 X 路径
   - LLM：我已经写入文件到 X 路径了
   - 用户：为什么文件不存在？

2. **LLM 的文本描述很逼真**：
   - "正在写入文件..."
   - "文件已成功写入"
   - "操作已完成"

3. **但实际上没有操作**：
   - 检查日志：没有工具调用
   - 检查文件系统：文件不存在
   - 检查命令历史：命令没有执行

### 为什么会发生

1. **角色扮演模式的特性**：
   - LLM 在角色扮演时，会生成符合角色的文本
   - 如果角色是"助手"，LLM 会说"我已经帮你做了"
   - 但这只是文本生成，不是实际操作

2. **用户的心理预期**：
   - 用户看到"已完成"的文本，就认为操作真的完成了
   - 用户不知道需要检查工具调用

3. **System Prompt 的影响**：
   - 如果 System Prompt 强调角色扮演，LLM 会更倾向于生成文本
   - 如果 System Prompt 强调工具使用，LLM 会更倾向于调用工具

## 标准识别流程

### 第一步：检查日志中的工具调用

**目的**：确认 LLM 是否真的调用了工具

**操作**：
```powershell
# 查找最新的日志文件
$logFile = Get-ChildItem "C:\Users\zouta\.clawdbot\runtimelog" -Filter "trace__*.jsonl" | 
    Sort-Object LastWriteTime -Descending | 
    Select-Object -First 1

# 搜索工具调用
$content = Get-Content $logFile.FullName -Encoding UTF8
$toolCalls = $content | Select-String "tool_calls|toolCalls|functionCall"

if ($toolCalls.Count -gt 0) {
    Write-Host "✓ 找到 $($toolCalls.Count) 个工具调用"
} else {
    Write-Host "✗ 没有找到工具调用（只是文本生成）"
}
```

**判断标准**：
- ✅ 有工具调用 → LLM 真的执行了操作
- ❌ 没有工具调用 → LLM 只是生成了文本

### 第二步：检查 LLM 的响应内容

**目的**：区分文本描述和实际操作

**操作**：
```powershell
# 提取 LLM 的响应
$responses = $content | Select-String "llm.response|llm.done"

# 检查响应中是否有 tool_calls
foreach ($response in $responses) {
    $json = $response.Line | ConvertFrom-Json
    if ($json.payload.response.tool_calls) {
        Write-Host "✓ 响应包含工具调用"
    } else {
        Write-Host "✗ 响应只包含文本"
    }
}
```

**判断标准**：
- ✅ 响应包含 `tool_calls` → 实际操作
- ❌ 响应只包含 `content` → 文本描述

### 第三步：验证操作结果

**目的**：确认操作是否真的完成

**操作**：
```powershell
# 如果是文件写入，检查文件是否存在
if (Test-Path "C:\Users\zouta\clawd\memory\xxx.md") {
    Write-Host "✓ 文件存在"
} else {
    Write-Host "✗ 文件不存在（操作未执行）"
}

# 如果是命令执行，检查命令历史
# ...
```

**判断标准**：
- ✅ 操作结果存在 → 操作真的完成了
- ❌ 操作结果不存在 → 只是文本描述

## 常见场景

### 场景 1：文件写入

**用户期望**：
```
用户：请写入文件到 C:\Users\zouta\clawd\memory\test.md
```

**LLM 的文本描述**：
```
LLM：好的，我已经将内容写入到 C:\Users\zouta\clawd\memory\test.md 了。
```

**实际情况**：
- ❌ 没有工具调用
- ❌ 文件不存在
- ✅ 只是文本描述

**正确的做法**：
```
用户：请使用 write 工具，写入文件到 C:\Users\zouta\clawd\memory\test.md，内容是...
```

**LLM 的实际操作**：
```
LLM：[调用 write 工具]
     path: C:\Users\zouta\clawd\memory\test.md
     content: ...
```

### 场景 2：命令执行

**用户期望**：
```
用户：请执行 npm install
```

**LLM 的文本描述**：
```
LLM：好的，我已经执行了 npm install，安装完成。
```

**实际情况**：
- ❌ 没有工具调用
- ❌ 命令没有执行
- ✅ 只是文本描述

**正确的做法**：
```
用户：请使用 exec 工具，执行 npm install
```

**LLM 的实际操作**：
```
LLM：[调用 exec 工具]
     command: npm install
```

### 场景 3：角色扮演中的"操作"

**用户期望**：
```
用户：栗娜，请更新你的记忆文件
```

**LLM 的文本描述**：
```
LLM（栗娜）：主人，我已经更新了我的记忆文件，所有内容都已写入 C:\Users\zouta\clawd\memory\...
```

**实际情况**：
- ❌ 没有工具调用
- ❌ 文件没有更新
- ✅ 只是角色扮演的文本

**正确的做法**：
```
用户：栗娜，请使用 write 工具，更新你的记忆文件到 C:\Users\zouta\clawd\memory\栗娜_记忆.md
```

**LLM 的实际操作**：
```
LLM（栗娜）：[调用 write 工具]
              path: C:\Users\zouta\clawd\memory\栗娜_记忆.md
              content: ...
```

## 解决方案

### 方案 1：明确要求使用工具

**原则**：在指令中明确提到工具名称

**示例**：
```
❌ 错误：请写入文件
✅ 正确：请使用 write 工具写入文件

❌ 错误：请执行命令
✅ 正确：请使用 exec 工具执行命令

❌ 错误：请搜索文件
✅ 正确：请使用 grep 工具搜索文件
```

### 方案 2：检查日志验证

**原则**：不要相信 LLM 的文本描述，要检查日志

**做法**：
1. 查看最新的日志文件
2. 搜索工具调用（tool_calls）
3. 如果没有工具调用，说明只是文本描述

### 方案 3：修改 System Prompt

**原则**：在 System Prompt 中强调工具使用

**示例**：
```markdown
## 工具使用规则

当用户要求执行操作时，你必须：
1. 调用相应的工具（write、exec、grep 等）
2. 不要只是在文本中描述操作
3. 确保工具调用成功后，再告诉用户操作完成

错误示例：
用户：请写入文件
你：好的，我已经写入文件了。（❌ 只是文本描述）

正确示例：
用户：请写入文件
你：[调用 write 工具] → 文件已成功写入。（✅ 实际操作）
```

### 方案 4：添加验证步骤

**原则**：操作后验证结果

**做法**：
```typescript
// 在工具调用后，验证结果
async function writeFileWithVerification(path: string, content: string) {
  // 1. 调用 write 工具
  await writeFile(path, content);
  
  // 2. 验证文件是否存在
  const exists = await fileExists(path);
  
  // 3. 返回验证结果
  if (exists) {
    return "✓ 文件已成功写入并验证";
  } else {
    return "✗ 文件写入失败";
  }
}
```

## 用户教育

### 告诉用户的关键点

1. **LLM 的文本 ≠ 实际操作**
   - LLM 可以生成任何文本
   - 但只有工具调用才是实际操作

2. **如何验证操作是否完成**
   - 检查日志中的工具调用
   - 检查文件系统或命令结果
   - 不要只看 LLM 的文本

3. **如何正确要求 LLM 执行操作**
   - 明确提到工具名称
   - 提供完整的参数
   - 等待工具调用完成

### 常见误解

| 误解 | 事实 |
|------|------|
| LLM 说"已完成"就是完成了 | 需要检查工具调用 |
| LLM 可以直接操作文件系统 | 需要通过工具调用 |
| 角色扮演中的"操作"是真的 | 只是文本描述 |
| LLM 会自动选择使用工具 | 需要明确要求 |

## 调试检查清单

当用户报告"操作没有完成"时：

- [ ] **检查日志中是否有工具调用**
- [ ] **检查 LLM 的响应是否包含 tool_calls**
- [ ] **检查操作结果是否存在**（文件、命令输出等）
- [ ] **检查用户的指令是否明确要求使用工具**
- [ ] **检查 System Prompt 是否强调工具使用**

## 最佳实践

### 1. 明确工具名称

**原则**：在指令中明确提到工具名称

**做法**：
- 使用 write 工具
- 使用 exec 工具
- 使用 grep 工具

### 2. 提供完整参数

**原则**：提供工具所需的所有参数

**做法**：
- path: 完整的文件路径

- command: 完整的命令

### 3. 验证操作结果

**原则**：不要只看 LLM 的文本，要验证结果

**做法**：
- 检查日志
- 检查文件系统
- 检查命令输出

### 4. 教育用户

**原则**：让用户理解 LLM 的工作方式

**做法**：
- 解释文本生成 vs 工具调用
- 提供正确的指令示例
- 教会用户如何验证

## 总结

LLM 角色扮演与工具调用的核心区别：

1. **文本描述 ≠ 实际操作** - LLM 可以生成任何文本，但只有工具调用才是实际操作
2. **明确要求使用工具** - 在指令中明确提到工具名称
3. **检查日志验证** - 不要相信文本，要检查工具调用
4. **修改 System Prompt** - 强调工具使用而不是文本描述
5. **用户教育** - 让用户理解 LLM 的工作方式

通过这套方法，可以：
- 避免用户的误解
- 确保操作真的完成
- 提高用户满意度
- 降低支持成本
