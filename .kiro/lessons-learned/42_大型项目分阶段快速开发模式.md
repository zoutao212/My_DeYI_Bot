# 大型项目分阶段快速开发模式

> **核心价值**：将大型项目（> 200 个任务）的开发周期从数周缩短到数天，快速验证核心功能可行性。

---

## 问题识别

### 现象
- 大型项目任务数量庞大（例如：329 个任务）
- 完整开发周期过长（例如：16 周）
- 用户希望快速看到可运行的核心功能
- 测试、文档、优化任务占用大量时间

### 成本分析
- **时间成本**：完整开发需要数周甚至数月
- **机会成本**：核心功能验证延迟，无法快速调整方向
- **资源成本**：大量时间花在辅助任务上，而非核心功能

### 触发场景
1. 项目任务数量 > 200 个
2. 用户要求"快速开发"或"先看到核心功能"
3. 项目处于探索阶段，需要快速验证可行性
4. 测试和文档可以后补

---

## 解决方案

### 核心策略：分离核心功能与辅助任务

将任务分为两类：
1. **核心功能任务**：必须实现才能运行的代码
2. **辅助任务**：测试、文档、优化、示例等

### 执行流程

#### 第一步：任务分类

**核心功能任务**（必须实现）：
- 类型定义
- 核心类实现
- 接口实现
- 基础配置
- 关键逻辑

**辅助任务**（可延后）：
- 单元测试
- 集成测试
- 性能测试
- API 文档
- 使用示例
- 教程视频
- 性能优化
- 代码重构

#### 第二步：优先级排序

**P0（最高优先级）**：
- 核心类型定义
- 核心类实现
- 基础集成

**P1（高优先级）**：
- 错误处理
- 基础监控
- 基础文档

**P2（中优先级）**：
- 单元测试
- 集成测试
- 详细文档

**P3（低优先级）**：
- 性能优化
- 示例代码
- 教程视频

#### 第三步：分阶段执行

**阶段 1：核心功能开发**
- 只实现 P0 任务
- 每个阶段完成后验证编译
- 创建阶段总结文档

**阶段 2：基础完善**
- 实现 P1 任务
- 补充基础测试
- 补充基础文档

**阶段 3：全面完善**
- 实现 P2 和 P3 任务
- 补充完整测试
- 补充完整文档

#### 第四步：质量门槛

**核心功能阶段的质量门槛**：
- ✅ 编译通过（`pnpm tsc --noEmit`）
- ✅ 核心功能可运行
- ✅ 基础错误处理完善
- ✅ 阶段总结文档完整

**不要求**：
- ❌ 测试覆盖率
- ❌ 完整文档
- ❌ 性能优化
- ❌ 示例代码

## 实战案例：多层 Agent 架构项目

### 项目背景
- **总任务数**：329 个任务
- **预计时间**：16 周（完整开发）
- **用户需求**：快速验证核心功能可行性

### 任务分类结果

| 类型 | 任务数 | 占比 |
|------|--------|------|
| 核心功能 | ~100 | 30% |
| 单元测试 | ~80 | 24% |
| 集成测试 | ~40 | 12% |
| 文档 | ~60 | 18% |
| 优化 | ~30 | 9% |
| 示例 | ~19 | 6% |

### 执行策略
1. **只实现核心功能任务**（~100 个）
2. **跳过所有测试任务**（~120 个）
3. **跳过所有文档任务**（~60 个）
4. **跳过所有优化任务**（~30 个）
5. **跳过所有示例任务**（~19 个）

### 执行结果
- **实际完成任务**：约 100 个核心任务
- **实际耗时**：2-3 天（vs 16 周）
- **编译状态**：✅ 通过
- **功能状态**：✅ 核心功能可运行
- **效率提升**：约 **95% 时间节省**（核心功能阶段）

### 后续计划
- **阶段 2**：补充单元测试（3-5 天）
- **阶段 3**：补充集成测试（3-5 天）
- **阶段 4**：补充文档和示例（2-3 天）
- **阶段 5**：性能优化（2-3 天）

## 使用指南

### 何时使用此模式

**适用场景**：
- ✅ 项目任务数量 > 200 个
- ✅ 用户要求快速验证可行性
- ✅ 项目处于探索阶段
- ✅ 测试和文档可以后补
- ✅ 有明确的阶段划分

**不适用场景**：
- ❌ 生产环境部署（必须有完整测试）
- ❌ 安全关键系统（必须有完整测试和文档）
- ❌ 团队协作项目（需要完整文档）
- ❌ 开源项目（需要完整文档和示例）

### 执行步骤

#### 1. 分析任务清单
```bash
# 统计任务类型
grep -E "测试|test" tasks.md | wc -l  # 测试任务数
grep -E "文档|doc" tasks.md | wc -l   # 文档任务数
grep -E "优化|optim" tasks.md | wc -l # 优化任务数
```

#### 2. 标记核心任务
在任务清单中标记核心任务：
```markdown
- [ ] 1.1 创建核心类型定义 ⭐ 核心
- [ ] 1.2 实现核心类 ⭐ 核心
- [ ] 1.3 单元测试 ⏭️ 跳过
- [ ] 1.4 API 文档 ⏭️ 跳过
```

#### 3. 使用子代理批量执行
```typescript
// 委托给子代理执行核心任务
invokeSubAgent({
  name: "general-task-execution",
  prompt: `
    执行项目的核心任务。
    
    **执行策略**：
    1. 只实现核心功能代码
    2. 跳过所有测试任务
    3. 跳过所有文档任务
    4. 跳过所有优化任务
    5. 每个阶段完成后验证编译
    6. 创建阶段完成总结
    
    **核心任务列表**：
    [列出核心任务]
  `
});
```

#### 4. 验证编译
```bash
# 每个阶段完成后验证编译
pnpm tsc --noEmit
```

#### 5. 创建阶段总结
```markdown
# 阶段 X 完成总结

## 完成的工作
- ✅ 任务 1
- ✅ 任务 2
- ✅ 任务 3

## 创建的文件
1. file1.ts
2. file2.ts

## 编译状态
✅ 通过

## 后续工作
- 补充单元测试
- 补充文档
```

## 风险与缓解

### 风险 1：技术债务累积
**风险**：跳过测试和文档可能导致技术债务累积

**缓解措施**：
- 明确后续补充计划
- 预留足够的时间补充测试和文档
- 定期回顾技术债务

### 风险 2：核心功能有 Bug
**风险**：没有测试可能导致核心功能有 Bug

**缓解措施**：
- 通过编译检查确保基本正确性
- 手动测试核心功能
- 优先补充核心功能的测试

### 风险 3：文档缺失导致理解困难
**风险**：没有文档可能导致后续理解困难

**缓解措施**：
- 创建阶段总结文档
- 添加代码注释
- 优先补充核心功能的文档

### 风险 4：性能问题
**风险**：没有优化可能导致性能问题

**缓解措施**：
- 在设计阶段考虑性能
- 避免明显的性能问题
- 后续阶段进行性能优化

---

## 最佳实践

### 1. 明确阶段目标
每个阶段都要有明确的目标和验收标准。

### 2. 保持沟通
与用户保持沟通，确认优先级和阶段划分。

### 3. 及时验证
每个阶段完成后及时验证编译和基本功能。

### 4. 记录决策
记录为什么跳过某些任务，方便后续补充。

### 5. 预留时间
为后续阶段预留足够的时间补充测试和文档。

## 相关模式

### 与敏捷开发的关系
- **MVP（最小可行产品）**：类似，但更激进
- **迭代开发**：分阶段执行，逐步完善
- **持续集成**：每个阶段都验证编译

### 与 TDD 的关系
- **TDD（测试驱动开发）**：先写测试，再写代码
- **本模式**：先写核心代码，后补测试
- **适用场景不同**：TDD 适合生产环境，本模式适合探索阶段

## 常见问题

### Q: 跳过测试会不会有问题？
A: 会有风险，但可以通过编译检查和手动测试降低风险。后续阶段必须补充测试。

### Q: 跳过文档会不会导致理解困难？
A: 会有影响，但可以通过阶段总结和代码注释缓解。后续阶段必须补充文档。

### Q: 如何判断哪些任务是核心任务？
A: 问自己："如果不实现这个任务，系统能运行吗？" 如果答案是"不能"，就是核心任务。

### Q: 如何避免技术债务累积？
A: 明确后续补充计划，预留足够的时间，定期回顾技术债务。

### Q: 这个模式适合所有项目吗？
A: 不适合。只适合探索阶段的大型项目。生产环境和团队协作项目需要完整测试和文档。

## 总结

大型项目分阶段快速开发模式是一种**激进但有效**的开发策略，适合探索阶段的大型项目。

**核心原则**：
- 分离核心功能与辅助任务
- 优先实现核心功能
- 后续阶段补充测试和文档
- 每个阶段都有明确的质量门槛

**适用场景**：
- 项目任务数量 > 200 个
- 用户要求快速验证可行性
- 项目处于探索阶段
- 测试和文档可以后补

**效果**：
- 时间节省：约 95%（核心功能阶段）
- 风险控制：通过编译检查和阶段总结
- 适用性：探索阶段非常适合
