# UI 配置问题定位方法论

> **来源**：UI 干扰问题修复实战（2026-01-29）

---

## 核心问题

**用户反馈 UI 功能干扰体验，但不知道在哪里配置。**

常见表现：
- "这个弹窗怎么关不掉？"
- "这个审批提示一直出现"
- "这个卡片一直自动显示"
- "我改了配置但没有效果"

---

## 根本原因

### 1. UI 配置 vs 后端配置混淆 ⚠️ 最常见

**UI 配置**：
- 存储位置：`localStorage`（浏览器本地）
- 配置文件：`ui/src/ui/storage.ts`
- 作用范围：只影响当前浏览器
- 修改方式：修改默认值 + 重新构建 UI

**后端配置**：
- 存储位置：`~/.clawdbot/clawdbot.json`
- 配置文件：`src/config/zod-schema.*.ts`
- 作用范围：影响所有客户端
- 修改方式：修改配置文件 + 重启 Gateway

**关键教训**：
- 修改后端配置不会影响 UI 配置
- 修改 UI 配置不会影响后端配置
- 两者是完全独立的系统

### 2. UI 默认值设计不合理

**问题**：
- 干扰性功能默认开启（审批、调试面板）
- 技术细节默认显示（运行事件流、payload）
- 用户需要手动关闭

**正确做法**：
- 干扰性功能默认关闭
- 技术细节默认隐藏
- 关键信息默认显示

## 标准定位流程

### 第一步：识别问题类型

**问题是 UI 层面还是后端层面？**

| 特征 | UI 层面 | 后端层面 |
|------|---------|----------|
| 配置位置 | localStorage | clawdbot.json |
| 影响范围 | 当前浏览器 | 所有客户端 |
| 修改方式 | 修改 UI 代码 | 修改配置文件 |
| 重启要求 | 重新构建 UI | 重启 Gateway |

**判断方法**：
- 如果是"弹窗"、"卡片"、"按钮"等 UI 元素 → UI 层面
- 如果是"API 行为"、"数据处理"、"权限控制"等 → 后端层面

### 第二步：搜索相关关键词

**UI 层面问题**：
```powershell
# 搜索配置定义
grepSearch -query "关键词" -includePattern "ui/src/ui/storage.ts"

# 搜索 UI 组件
grepSearch -query "关键词" -includePattern "ui/src/ui/views/*.ts"

# 搜索 UI 开关
grepSearch -query "关键词" -includePattern "ui/src/ui/app-render.helpers.ts"
```

**后端层面问题**：
```powershell
# 搜索配置 schema
grepSearch -query "关键词" -includePattern "src/config/zod-schema.*.ts"

# 搜索配置类型
grepSearch -query "关键词" -includePattern "src/config/types.*.ts"

# 搜索配置使用
grepSearch -query "关键词" -includePattern "src/**/*.ts"
```

### 第三步：找到配置位置

**UI 配置位置**：

| 文件 | 作用 |
|------|------|
| `ui/src/ui/storage.ts` | 默认值定义 |
| `ui/src/ui/views/*.ts` | UI 组件渲染 |
| `ui/src/ui/app-render.helpers.ts` | UI 开关 |
| `ui/src/ui/app.ts` | 状态管理 |

**后端配置位置**：

| 文件 | 作用 |
|------|------|
| `src/config/zod-schema.*.ts` | 配置 schema |
| `src/config/types.*.ts` | 配置类型 |
| `src/gateway/server.impl.ts` | 配置使用 |
| `~/.clawdbot/clawdbot.json` | 用户配置 |

### 第四步：修改默认值

**UI 配置修改**：

1. 修改 `ui/src/ui/storage.ts` 中的默认值
2. 重新构建 UI：`cd ui && pnpm build`
3. 重新构建主项目：`pnpm build`
4. 验证 dist 文件：`Select-String -Path "dist/control-ui/assets/*.js" -Pattern "关键词"`
5. 重启 Gateway

**后端配置修改**：

1. 修改 `~/.clawdbot/clawdbot.json`
2. 重启 Gateway
3. 验证配置生效

### 第五步：验证效果

**UI 配置验证**：
```powershell
# 验证源码修改
Get-Content "ui/src/ui/storage.ts" -Encoding UTF8 | Select-String -Pattern "关键词" -Context 2,2

# 验证 dist 文件
Select-String -Path "dist/control-ui/assets/*.js" -Pattern "关键词" -Context 1,1
```

**后端配置验证**：
```powershell
# 验证配置文件
Get-Content "~/.clawdbot/clawdbot.json" -Encoding UTF8 | ConvertFrom-Json | Select-Object -ExpandProperty 关键路径

# 验证运行时行为
# 打开浏览器，测试功能
```

## 实战案例

### 案例 1：Chat 发送审批提示

**问题**：用户反馈 "Chat send approval needed" 审批提示仍然出现

**定位过程**：
1. 识别问题类型：UI 层面（弹窗提示）
2. 搜索关键词：`chatRequireSendApproval`
3. 找到配置位置：`ui/src/ui/storage.ts` 第 44 行
4. 发现默认值：`chatRequireSendApproval: true`
5. 修改默认值：`chatRequireSendApproval: false`
6. 重新构建并验证

**关键教训**：
- 用户以为关闭了 LLM 审批（后端配置），但实际上看到的是 Chat 发送审批（UI 配置）
- 两者是完全独立的功能

### 案例 2：运行事件流 UI 卡片

**问题**：用户反馈 "运行事件流 llm/tool/patch（不依赖最终回复）"UI 卡片一直自动出现

**定位过程**：
1. 识别问题类型：UI 层面（卡片显示）
2. 搜索关键词：`renderRun`
3. 找到配置位置：`ui/src/ui/views/run.ts`
4. 发现渲染逻辑：完整的运行面板渲染
5. 修改渲染逻辑：在函数开头添加 `return nothing;`
6. 重新构建并验证

**关键教训**：
- 运行面板默认显示，显示大量技术细节
- 应该默认隐藏，只在需要时显示

## UI 默认值设计原则

### 原则 1：干扰性功能默认关闭

**干扰性功能**：
- 审批提示（Chat 发送审批、LLM 审批、Exec 审批）
- 调试面板（运行事件流、Payload 显示）
- 技术细节（错误堆栈、内部状态）

**正确做法**：
- 默认关闭
- 提供 UI 开关
- 用户可以手动开启

### 原则 2：关键信息默认显示

**关键信息**：
- 错误提示（连接失败、请求失败）
- 状态提示（正在发送、正在加载）
- 结果提示（发送成功、保存成功）

**正确做法**：
- 默认显示
- 清晰明了
- 不干扰用户

### 原则 3：技术细节默认隐藏

**技术细节**：
- 运行事件流（llm/tool/patch）
- Payload 内容（请求体、响应体）
- 内部状态（session、context）

**正确做法**：
- 默认隐藏
- 提供"查看详情"按钮
- 只在需要时显示

## 常见错误

### 错误 1：混淆 UI 配置和后端配置

**错误做法**：
- 修改了后端配置，期望 UI 行为改变
- 修改了 UI 配置，期望后端行为改变

**正确做法**：
- 先识别问题类型（UI 层面 vs 后端层面）
- 再修改对应的配置

### 错误 2：只修改源码，不重新构建

**错误做法**：
- 修改了 `ui/src/ui/storage.ts`
- 没有运行 `pnpm build`
- 期望修改生效

**正确做法**：
1. 修改源码
2. 重新构建 UI：`cd ui && pnpm build`
3. 重新构建主项目：`pnpm build`
4. 验证 dist 文件
5. 重启 Gateway

### 错误 3：修改后不验证

**错误做法**：
- 修改了配置
- 没有验证是否生效
- 假设修改成功

**正确做法**：
1. 修改配置
2. 验证源码修改
3. 验证 dist 文件
4. 验证运行时行为

## 关键教训

1. **UI 配置和后端配置是独立的** - 不要混淆
2. **修改后必须重新构建** - 源码修改不会自动生效
3. **修改后必须验证** - 不要假设修改成功
4. **默认值设计很重要** - 干扰性功能默认关闭
5. **技术细节默认隐藏** - 只在需要时显示
