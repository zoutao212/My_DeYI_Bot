# 系统架构分析方法论

> **触发场景**：用户反馈"系统设计拉胯"、"功能不好用"、"经常出问题"，但不知道具体哪里有问题

---

## 核心原则

**不要只看表面症状，要追踪根本原因。**

系统问题往往是多个小问题的叠加：
- 单个模块可能没问题
- 但模块之间的协作有问题
- 或者缺少关键的降级/恢复机制

---

## 标准分析流程

### 第一步：收集现象

**从用户日志/错误信息中提取关键信息：**

1. **用户期望**：用户想做什么？
2. **实际结果**：系统实际做了什么？
3. **差异**：期望与实际的差距在哪里？

**示例：**
```
用户期望：写一个万字角色卡
实际结果：AI 反复询问"需要扩充什么主题"
差异：AI 忘记了任务目标
```

### 第二步：定位核心模块

**通过关键词搜索定位相关代码：**

```bash
# 搜索关键功能
grepSearch -query "关键词1|关键词2" -includePattern "src/**/*.ts"

# 搜索配置
grepSearch -query "config.*关键词" -includePattern "src/**/*.ts"

# 搜索工具
grepSearch -query "tool.*关键词" -includePattern "src/**/*.ts"
```

**示例：**
```bash
# 定位上下文管理
grepSearch -query "limitHistoryTurns|sanitizeSessionHistory"

# 定位记忆系统
grepSearch -query "memory_search|memory_get"

# 定位任务跟踪
grepSearch -query "task.*tracker|task.*state"
```

### 第三步：分析数据流

**追踪数据从哪来到哪去：**

1. **入口**：数据从哪里进入系统？
2. **处理**：数据经过了哪些处理步骤？
3. **存储**：数据存储在哪里？
4. **读取**：数据从哪里读取？
5. **出口**：数据最终去了哪里？

**绘制数据流图：**
```
用户输入
  ↓
解析 (parse)
  ↓
验证 (validate)
  ↓
处理 (process)
  ↓
存储 (store)
  ↓
读取 (load)
  ↓
裁剪 (limit)  ⚠️ 问题可能在这里
  ↓
发送给 LLM
```

### 第四步：识别设计缺陷

**对比最佳实践，找出设计缺陷：**

| 设计缺陷 | 表现 | 最佳实践 |
|---------|------|---------|
| **过度裁剪** | 直接丢弃旧数据 | 压缩成摘要 |
| **缺少降级** | 依赖外部 API，无降级 | 提供本地降级方案 |
| **缺少跟踪** | 无状态记录 | 显式跟踪任务状态 |
| **缺少恢复** | 失败后不重试 | 自动重试或切换方案 |

**常见设计缺陷模式：**

1. **单点依赖**：依赖单一外部服务，无降级方案
2. **状态丢失**：关键状态没有持久化
3. **错误传播**：错误没有被捕获和处理
4. **资源泄漏**：资源没有正确释放
5. **性能瓶颈**：关键路径上有阻塞操作

### 第五步：提出分级改进方案

**按优先级分级：**

| 优先级 | 定义 | 示例 |
|--------|------|------|
| **P0** | 立即修复，影响核心功能 | 任务目标丢失 |
| **P1** | 短期改进，影响用户体验 | 工具调用失败不重试 |
| **P2** | 长期优化，提升性能 | 智能上下文压缩 |

**每个方案必须包含：**

1. **问题描述**：具体是什么问题？
2. **根本原因**：为什么会有这个问题？
3. **改进方案**：如何修复？（带代码示例）
4. **预期效果**：修复后会有什么改善？
5. **工作量估算**：需要多少时间？

## 质量门槛

每次分析必须满足：

- ✅ 至少识别 3 个具体问题
- ✅ 每个问题都有明确的位置（文件:行号）
- ✅ 每个问题都有根本原因分析
- ✅ 每个问题都有改进方案（带代码示例）
- ✅ 改进方案按优先级分级（P0/P1/P2）
- ✅ 有验证方案（测试用例）

## 常见错误

### 错误 1：只看表面症状

❌ **错误做法：**
- "AI 不好用"
- "系统有问题"
- "功能不稳定"

✅ **正确做法：**
- "AI 在第 10 轮对话后忘记了任务目标，因为 `limitHistoryTurns` 直接丢弃了旧消息"
- "记忆检索失效，因为 API Key 配置错误且没有降级方案"

### 错误 2：只提问题不提方案

❌ **错误做法：**
- 列出一堆问题
- 但没有改进方案

✅ **正确做法：**
- 每个问题都有对应的改进方案
- 改进方案包含代码示例
- 改进方案按优先级分级

### 错误 3：方案不可执行

❌ **错误做法：**
- "应该优化上下文管理"（太抽象）
- "需要改进记忆系统"（不知道怎么改）

✅ **正确做法：**
- "在 `limitHistoryTurns` 中保留第一条用户消息（任务目标）"
- "在 `memory_search` 中添加关键词搜索降级方案"

## 实战案例

### 案例：Clawdbot 上下文管理问题

**问题现象：**
- 用户要求"写一个万字角色卡"
- AI 反复询问"需要扩充什么主题"
- 任务目标丢失

**根本原因分析：**

| 问题 | 位置 | 原因 | 影响 |
|------|------|------|------|
| 历史消息被过度裁剪 | `src/agents/pi-embedded-runner/history.ts:16` | 直接丢弃旧消息，没有摘要 | 任务目标丢失 |
| 记忆系统依赖外部 API | `src/agents/tools/memory-tool.ts:34` | API 不可用时直接返回空 | 记忆检索失效 |
| 缺少任务状态跟踪 | 无对应文件 | 没有显式跟踪任务状态 | 无法恢复任务 |

**改进方案：**

#### P0：保留任务目标

```typescript
export function limitHistoryTurns(
  messages: AgentMessage[],
  limit: number | undefined,
  options?: { preserveTaskGoal?: boolean }
): AgentMessage[] {
  if (!limit || limit <= 0) return messages;

  // 保留第一条用户消息（任务目标）
  const taskGoal = options?.preserveTaskGoal 
    ? messages.find(m => m.role === "user")
    : null;

  // 裁剪逻辑...
  const limited = messages.slice(lastUserIndex);

  // 将任务目标插入到开头
  return taskGoal ? [taskGoal, ...limited] : limited;
}
```

**预期效果：** AI 不会忘记任务目标

**工作量估算：** 1-2 天

## 总结

系统架构分析的核心是：

1. **从现象到本质**：不要只看表面症状
2. **从局部到全局**：追踪完整的数据流
3. **从问题到方案**：每个问题都有改进方案
4. **从方案到验证**：每个方案都有测试用例

通过系统化的分析方法，可以：
- 快速定位根本原因
- 提出可执行的改进方案
- 避免重复犯错
