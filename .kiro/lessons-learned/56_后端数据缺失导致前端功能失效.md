# 后端数据缺失导致前端功能失效

**日期**：2026-02-02  
**来源**：clientTools 显示问题调试实战

---

## 问题模式

### 现象

- ✅ 前端代码正确
- ✅ 前端逻辑正确
- ✅ 浏览器缓存已清除
- ❌ 功能仍然不生效

### 错误调试方向

1. 反复刷新浏览器（Ctrl+Shift+R）
2. 检查前端代码是否有问题
3. 怀疑浏览器缓存问题
4. 修改前端代码尝试修复

**结果**：浪费大量时间，问题仍然存在。

### 正确调试方向

1. **检查后端 API 返回的数据**
2. **追踪数据流：后端 → 前端**
3. **验证后端是否真的返回了数据**
4. **检查 Git 历史，确认是否一直就没有实现**

---

## 根本原因

**后端硬编码返回 `null` 或空数据。**

### 典型代码

```typescript
// ❌ 错误：硬编码返回 null
return {
  clientToolsStatus: "not_applicable",
  clientTools: null,  // 从来没有返回真实数据
};
```

**问题**：
- 前端无论如何修改都不会生效
- 因为后端根本没有发送数据
- 必须修复后端数据源

## 实战案例

### 案例：clientTools 预览面板不显示

**问题**：
- 前端代码正确（`chat-send-approval.ts`）
- 预览面板应该显示 Tool Definitions
- 但实际上什么都不显示

**错误调试**：
1. 反复刷新浏览器
2. 检查前端代码
3. 怀疑浏览器缓存

**正确调试**：
1. 检查后端 API 返回的数据
2. 发现 `clientTools: null`
3. 检查 Git 历史
4. 确认：**从来没有返回过真实数据**

**修复**：
```typescript
// ✅ 正确：返回真实数据
clientTools:
  tools.length > 0
    ? tools.map((t: any) => ({
        name: t.name,
        description: t.description,
        parameters: t.parameters,
      }))
    : null,
```

## 标准调试流程

### 第一步：验证后端数据

**不要相信"应该有数据"，必须验证！**

#### 方法 1：检查 API 响应

```powershell
# 查看最新的 trace 日志
$trace = Get-Content "C:\Users\zouta\.clawdbot\runtimelog\trace__*.jsonl" -Encoding UTF8 | ConvertFrom-Json
$preview = $trace | Where-Object { $_.event -eq "chat.send.preview" } | Select-Object -Last 1
$preview.payload.preview.clientTools
```

#### 方法 2：检查后端代码

```typescript
// 搜索返回 clientTools 的位置
// 检查是否硬编码返回 null
```

#### 方法 3：检查 Git 历史

```powershell
# 查看历史版本
git log --oneline -20 -- src/gateway/server-methods/chat.ts

# 查看特定版本的代码
git show <commit>:src/gateway/server-methods/chat.ts | Select-String -Pattern "clientTools"
```

### 第二步：追踪数据流

**从后端到前端，逐层验证。**

```
后端构建数据
  ↓ 验证：数据是否正确？
后端返回 API
  ↓ 验证：API 是否返回了数据？
前端接收数据
  ↓ 验证：前端是否收到了数据？
前端渲染
  ↓ 验证：前端是否正确渲染？
```

### 第三步：修复后端数据源

**在数据产生的地方修复，不是在前端打补丁。**

```typescript
// ❌ 错误：在前端打补丁
if (!preview.clientTools) {
  preview.clientTools = []; // 这不会解决根本问题
}

// ✅ 正确：修复后端数据源
clientTools:
  tools.length > 0
    ? tools.map((t: any) => ({
        name: t.name,
        description: t.description,
        parameters: t.parameters,
      }))
    : null,
```

## 关键教训

### 1. 数据流追踪优先

**前端功能失效时，先检查后端数据。**

- 不要假设后端"应该"返回数据
- 必须验证后端真的返回了数据
- 追踪数据流，从后端到前端

### 2. 不要相信"应该有数据"

**必须验证，不要猜测。**

- 检查 API 响应
- 检查后端代码
- 检查 Git 历史

### 3. Git 历史验证

**确认是否一直就没有实现。**

```powershell
# 查看历史版本
git log --oneline -20 -- <file>

# 查看特定版本的代码
git show <commit>:<file> | Select-String -Pattern "关键词"
```

### 4. 修复数据源，不是打补丁

**在数据产生的地方修复。**

- 不要在前端打补丁
- 不要在中间层打补丁
- 在后端数据源修复

### 5. 验证修复是否生效

**修复后必须验证。**

```powershell
# 重新检查 API 响应
$trace = Get-Content "C:\Users\zouta\.clawdbot\runtimelog\trace__*.jsonl" -Encoding UTF8 | ConvertFrom-Json
$preview = $trace | Where-Object { $_.event -eq "chat.send.preview" } | Select-Object -Last 1
$preview.payload.preview.clientTools

# 如果不是 null，说明修复成功
```

## 检查清单

前端功能失效时，按以下顺序检查：

- [ ] **验证后端数据**：检查 API 响应，确认后端是否返回了数据
- [ ] **追踪数据流**：从后端到前端，逐层验证
- [ ] **检查后端代码**：搜索返回数据的位置，检查是否硬编码返回 null
- [ ] **检查 Git 历史**：确认是否一直就没有实现
- [ ] **修复后端数据源**：在数据产生的地方修复
- [ ] **验证修复是否生效**：重新检查 API 响应
