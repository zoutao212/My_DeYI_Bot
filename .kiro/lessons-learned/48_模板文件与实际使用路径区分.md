# 模板文件与实际使用路径区分

**问题类型**: 路径混淆  
**严重程度**: ⚠️ 中等（容易导致文件创建到错误位置）  
**适用场景**: 当项目同时包含模板文件和用户实例时

---

## 问题描述

### 现象
AI 在创建/修改文件时，容易混淆两种路径：
1. **项目源码中的模板文件**（用于版本控制和分发）
2. **用户目录中的实际使用文件**（运行时真正读取的文件）

### 典型错误
```
❌ 错误：在项目源码目录创建文件
创建: D:\Git_GitHub\clawdbot\clawd\characters\lina\memory\core-memories.md

✅ 正确：在用户目录创建文件
创建: C:\Users\zouta\clawd\characters\lina\memory\core-memories.md
```

---

## 根本原因

### 1. 路径相似性
两个路径结构相似，容易混淆：
```
项目源码: D:\Git_GitHub\clawdbot\clawd\characters\lina\
用户目录: C:\Users\zouta\clawd\characters\lina\
```

### 2. 上下文切换
在同一对话中，可能需要：
- 查看项目源码（理解模板结构）
- 修改用户文件（实际使用）

### 3. 配置隐式性
配置文件中定义了实际使用路径，但不够显眼：
```json
{
  "agents": {
    "defaults": {
      "workspace": "C:\\Users\\zouta\\clawd"  // 容易被忽略
    }
  }
}
```

---

## 解决方案

### 方案 1：明确路径用途（推荐）

在对话开始时，明确两种路径的用途：

```markdown
## 路径说明

### 项目源码（模板文件）
- 路径: D:\Git_GitHub\clawdbot\clawd\
- 用途: 版本控制、代码审查、模板分发
- 操作: 只读（除非修改模板）

### 用户目录（实际使用）
- 路径: C:\Users\zouta\clawd\
- 用途: 运行时读取、用户数据存储
- 操作: 读写（实际使用的文件）
```

### 方案 2：验证路径解析逻辑

在创建/修改文件前，验证代码实际使用的路径：

```typescript
// 1. 读取配置
const workspace = cfg.agents?.defaults?.workspace;
// 结果: "C:\Users\zouta\clawd"

// 2. 代码解析
const effectiveWorkspaceDir = resolveAgentWorkspaceDir(cfg, agentId);
// 结果: "C:\Users\zouta\clawd"

// 3. 最终路径
const characterDir = join(effectiveWorkspaceDir, "characters", "lina");
// 结果: "C:\Users\zouta\clawd\characters\lina"
```

### 方案 3：使用配置验证命令

创建文件前，先验证配置：

```powershell
# 验证 workspace 配置
Get-Content "C:\Users\zouta\.clawdbot\clawdbot.json" -Encoding UTF8 | 
  ConvertFrom-Json | 
  Select-Object -ExpandProperty agents | 
  Select-Object -ExpandProperty defaults | 
  Select-Object workspace

# 输出: C:\Users\zouta\clawd ✅
```


## 代码中的路径解析

### 智能路径检测

代码中已经实现了智能路径检测，支持两种传入方式：

```typescript
// src/agents/lina/config/loader.ts
export async function loadCharacterProfile(
  characterName: string,
  basePath: string = process.cwd()
): Promise<CharacterProfile> {
  // 智能检测：如果 basePath 已经是 clawd 目录，直接使用
  const normalizedBasePath = basePath.endsWith("clawd") || 
                            basePath.endsWith("clawd\\") || 
                            basePath.endsWith("clawd/")
    ? basePath                    // 情况 1：直接使用
    : join(basePath, "clawd");    // 情况 2：追加 clawd
  
  const characterDir = join(normalizedBasePath, "characters", characterName);
  // ...
}
```

**优点**：
- ✅ 兼容两种路径传入方式
- ✅ 自动适配不同的调用场景
- ✅ 避免路径重复（如 `clawd/clawd/characters`）

### 配置驱动的路径解析

```typescript
// src/agents/agent-scope.ts
export function resolveAgentWorkspaceDir(cfg: ClawdbotConfig, agentId: string) {
  // 1. 优先使用 agent 特定配置
  const configured = resolveAgentConfig(cfg, id)?.workspace?.trim();
  if (configured) return resolveUserPath(configured);
  
  // 2. 使用默认配置
  const fallback = cfg.agents?.defaults?.workspace?.trim();
  if (fallback) return resolveUserPath(fallback);  // ✅ 返回用户目录
  
  // 3. 使用默认值
  return DEFAULT_AGENT_WORKSPACE_DIR;
}
```

**数据流**：
```
配置文件: C:\Users\zouta\.clawdbot\clawdbot.json
  ↓ workspace: "C:\Users\zouta\clawd"
代码解析: resolveAgentWorkspaceDir()
  ↓ 返回: "C:\Users\zouta\clawd"
智能检测: loadCharacterProfile()
  ↓ 识别为 clawd 目录，直接使用
最终路径: C:\Users\zouta\clawd\characters\lina\prompts\system.md
```

## 关键教训

### 1. 配置优先
- 始终以配置文件中的 `workspace` 为准
- 不要假设路径，要验证配置

### 2. 验证路径
- 创建文件前，先验证路径是否正确
- 使用 `Test-Path` 确认文件存在

### 3. 明确用途
- 模板文件：项目源码目录（版本控制）
- 实际使用：用户目录（运行时读取）

### 4. 追踪代码
- 不要只看文件结构，要追踪代码实际读取的路径
- 使用 `resolveAgentWorkspaceDir` 确认实际路径
