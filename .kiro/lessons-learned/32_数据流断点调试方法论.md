# 数据流断点调试方法论

**问题类型**：数据在处理流程中丢失或被修改

**触发场景**：
- 数据在某个环节丢失（如历史消息丢失、配置项不生效）
- 表面症状与实际问题不一致
- 需要追踪数据在多个环节的变化

**关键词**：数据流追踪、断点调试、日志追踪、数据丢失、环节定位

---

## 核心方法

### 第一步：确认数据源和终点

**目的**：确认问题确实存在，不是误判

**操作**：
1. **验证数据源**：原始数据是否完整？
   - 例如：Session 文件中是否有完整的历史消息？
   - 例如：配置文件中是否有正确的配置项？

2. **验证数据终点**：最终数据是否正确？
   - 例如：LLM payload 中是否包含历史消息？
   - 例如：运行时是否使用了正确的配置？

3. **对比差异**：数据源和终点的差异是什么？
   - 例如：Session 文件有 20 条消息，但 LLM payload 只有 3 条
   - 例如：配置文件有 `dmHistoryLimit: 0`，但运行时使用了默认值

**示例**：
```powershell
# 验证数据源（Session 文件）
$sessionFile = "$env:USERPROFILE\.clawdbot\agents\main\sessions\xxx.jsonl"
$lines = Get-Content $sessionFile -Encoding UTF8
$msgCount = ($lines | Where-Object { $_ -match '"type":"message"' }).Count
Write-Host "Session 文件中的消息数: $msgCount"

# 验证数据终点（LLM payload）
$traceFile = Get-ChildItem "$env:USERPROFILE\.clawdbot\runtimelog" -Filter "trace__*.jsonl" | Sort-Object LastWriteTime -Descending | Select-Object -First 1
$lines = Get-Content $traceFile.FullName -Encoding UTF8
foreach ($line in $lines) {
    if ($line -match '"event":"llm\.payload"') {
        $obj = $line | ConvertFrom-Json
        $payloadMsgCount = $obj.payload.payload.messages.Count
        Write-Host "LLM Payload 中的消息数: $payloadMsgCount"
        break
    }
}

# 对比差异
Write-Host "差异: $msgCount → $payloadMsgCount"
```

### 第二步：绘制数据流图

**目的**：理解数据在系统中的流动路径

**操作**：
1. **列出所有环节**：数据从源到终点经过哪些环节？
2. **标注关键函数**：每个环节对应的函数或模块是什么？
3. **标注数据格式**：每个环节的数据格式是什么？

**示例**：
```
Session 文件 (20 条消息)
  ↓ SessionManager.open()
SessionManager (? 条消息)
  ↓ prepareSessionManagerForRun()
SessionManager (? 条消息)
  ↓ createAgentSession()
activeSession.messages (? 条消息)
  ↓ sanitizeSessionHistory()
sanitized messages (? 条消息)
  ↓ limitHistoryTurns()
limited messages (? 条消息)
  ↓ replaceMessages()
activeSession.messages (? 条消息)
  ↓ LLM request
LLM payload (3 条消息)
```

### 第三步：在关键节点添加日志

**目的**：追踪数据在每个环节的变化

**操作**：
1. **在每个环节后添加日志**：记录数据的数量、格式、内容
2. **使用统一的日志格式**：便于搜索和对比
3. **记录关键信息**：不仅记录数量，还要记录内容预览

**示例**：
```typescript
// 在 SessionManager.open 后
const smContext = sessionManager.buildSessionContext();
log.info(`[attempt] SessionManager loaded ${smContext.messages.length} messages from file (sessionId: ${params.sessionId})`);
if (smContext.messages.length > 0) {
  const roles = smContext.messages.map(m => m.role).join(' → ');
  log.info(`[attempt] Message roles: ${roles}`);
}

// 在 createAgentSession 后
log.info(`[attempt] createAgentSession returned ${activeSession.messages.length} messages (sessionId: ${params.sessionId})`);
if (activeSession.messages.length > 0) {
  const roles = activeSession.messages.map(m => m.role).join(' → ');
  log.info(`[attempt] Active session message roles: ${roles}`);
}

// 在 sanitizeSessionHistory 后
log.info(`[attempt] sanitizeSessionHistory returned ${prior.length} messages`);

// 在 limitHistoryTurns 后
log.info(`[attempt] limitHistoryTurns returned ${limited.length} messages`);
```

### 第四步：运行测试并分析日志

**目的**：找到数据丢失的具体环节

**操作**：
1. **重新构建**：`pnpm build`
2. **重启服务**：确保使用最新代码
3. **触发问题**：执行会导致问题的操作
4. **查看日志**：搜索添加的日志，对比数据变化

**示例**：
```powershell
# 构建
pnpm build

# 重启 Gateway
Get-Process | Where-Object { $_.ProcessName -like "*clawdbot*" } | Stop-Process -Force
clawdbot gateway run --bind loopback --port 18789

# 触发问题（发送测试消息）
# ...

# 查看日志
$traceFile = Get-ChildItem "$env:USERPROFILE\.clawdbot\runtimelog" -Filter "trace__*.jsonl" | Sort-Object LastWriteTime -Descending | Select-Object -First 1
Select-String -Path $traceFile.FullName -Pattern "SessionManager loaded|createAgentSession returned|sanitizeSessionHistory|limitHistoryTurns" -Context 0,1 -Encoding UTF8
```

### 第五步：定位问题并修复

**目的**：根据日志确认问题，应用修复方案

**操作**：
1. **分析日志**：找到数据丢失的环节
2. **确认原因**：为什么在这个环节丢失？
3. **应用修复**：修复代码
4. **验证修复**：重新测试，确认问题解决

**示例**：

**场景 A：SessionManager 加载时就丢失了**
```
SessionManager loaded 3 messages  ← 问题在这里
createAgentSession returned 3 messages
```

**原因**：`prepareSessionManagerForRun` 重置了 session 文件

**修复**：
```typescript
// 在 session-manager-init.ts 中
if (params.hadSessionFile && header && !hasAssistant) {
  // ❌ 不要重置文件
  // await fs.writeFile(params.sessionFile, "", "utf-8");
  
  // ✅ 只标记为 unflushed
  sm.flushed = false;
}
```

**场景 B：createAgentSession 时丢失了**
```
SessionManager loaded 20 messages
createAgentSession returned 3 messages  ← 问题在这里
```

**原因**：`createAgentSession` 没有使用 sessionManager 的历史消息

**修复**：
```typescript
// 在 createAgentSession 后
const smContext = sessionManager.buildSessionContext();
if (smContext.messages.length > activeSession.messages.length) {
  log.warn(`[attempt] createAgentSession lost messages: ${smContext.messages.length} → ${activeSession.messages.length}`);
  
  // 手动恢复历史消息
  activeSession.agent.replaceMessages(smContext.messages);
  
  log.info(`[attempt] Restored ${smContext.messages.length} messages from sessionManager`);
}
```

**场景 C：limitHistoryTurns 时丢失了**
```
SessionManager loaded 20 messages
createAgentSession returned 20 messages
sanitizeSessionHistory returned 20 messages
limitHistoryTurns returned 3 messages  ← 问题在这里
```

**原因**：`dmHistoryLimit` 配置限制了历史消息

**修复**：
```json
{
  "channels": {
    "telegram": {
      "dmHistoryLimit": 0  // 禁用限制
    }
  }
}
```

## 关键原则

### 1. 不要猜测，要验证

❌ **错误做法**：
- "可能是 dmHistoryLimit 的问题"（没有验证配置文件）
- "可能是 limitHistoryTurns 的问题"（没有验证是否真的调用了）

✅ **正确做法**：
- 查看配置文件，确认 dmHistoryLimit 的值
- 添加日志，确认 limitHistoryTurns 是否被调用，以及输入输出

### 2. 从两端向中间追踪

❌ **错误做法**：
- 从头到尾逐个检查每个环节（效率低）

✅ **正确做法**：
- 先确认数据源（Session 文件）和数据终点（LLM payload）
- 然后在中间关键节点添加日志
- 二分查找：如果中间节点数据正确，问题在后半段；否则在前半段

### 3. 日志要详细但不冗余

❌ **错误做法**：
- 只记录数量：`log.info("Messages: 20")`（不知道是哪个环节）
- 记录全部内容：`log.info(JSON.stringify(messages))`（日志太大）

✅ **正确做法**：
- 记录环节、数量、角色序列：
  ```typescript
  log.info(`[attempt] SessionManager loaded ${count} messages`);
  log.info(`[attempt] Message roles: ${roles.join(' → ')}`);
  ```

### 4. 修复后要验证

❌ **错误做法**：
- 修复代码后，假设问题解决了

✅ **正确做法**：
- 重新构建、重启、测试
- 查看日志，确认数据流正常
- 测试多种场景，确保没有引入新问题

## 实战案例：多轮对话遗忘问题

### 问题描述

用户反馈：AI 在多轮对话中不停遗忘，连上一轮的对话内容都没有记住。

### 调试过程

#### 第一步：确认数据源和终点

```powershell
# 验证 Session 文件
$sessionFile = "$env:USERPROFILE\.clawdbot\agents\main\sessions\71cdcb5c-915f-4e4c-9f22-e6fbc947d6cd.jsonl"
$lines = Get-Content $sessionFile -Encoding UTF8
$msgCount = ($lines | Where-Object { $_ -match '"type":"message"' }).Count
Write-Host "Session 文件中的消息数: $msgCount"
# 输出：20+ 条消息

# 验证 LLM payload
$traceFile = Get-ChildItem "$env:USERPROFILE\.clawdbot\runtimelog" -Filter "trace__*.jsonl" | Sort-Object LastWriteTime -Descending | Select-Object -First 1
# 输出：只有 3 条消息（bootstrap + 上一次回复 + 当前消息）
```

**结论**：数据源完整，数据终点丢失，问题确实存在。

#### 第二步：绘制数据流图

```
Session 文件 (20 条消息) ✅
  ↓ SessionManager.open()
SessionManager (? 条消息)
  ↓ prepareSessionManagerForRun()
SessionManager (? 条消息)
  ↓ createAgentSession()
activeSession.messages (? 条消息)
  ↓ sanitizeSessionHistory()
sanitized messages (? 条消息)
  ↓ limitHistoryTurns()
limited messages (? 条消息)
  ↓ LLM request
LLM payload (3 条消息) ❌
```

#### 第三步：添加日志

```typescript
// 在 SessionManager.open 后
const smContext = sessionManager.buildSessionContext();
log.info(`[attempt] SessionManager loaded ${smContext.messages.length} messages from file`);

// 在 createAgentSession 后
log.info(`[attempt] createAgentSession returned ${activeSession.messages.length} messages`);
```

#### 第四步：运行测试

```powershell
pnpm build
# 重启 Gateway
# 发送测试消息
# 查看日志
```

#### 第五步：定位问题

**预期结果**：
- 如果 `SessionManager loaded 3 messages`：问题在 `prepareSessionManagerForRun`
- 如果 `SessionManager loaded 20 messages` 但 `createAgentSession returned 3 messages`：问题在 `createAgentSession`

**修复方案**：根据日志结果应用对应的修复。

## 总结

**数据流断点调试方法论**：
1. 确认数据源和终点（验证问题存在）
2. 绘制数据流图（理解数据路径）
3. 在关键节点添加日志（追踪数据变化）
4. 运行测试并分析日志（找到丢失环节）
5. 定位问题并修复（应用修复方案）

**关键原则**：
- 不要猜测，要验证
- 从两端向中间追踪
- 日志要详细但不冗余
- 修复后要验证

**适用场景**：
- 数据在处理流程中丢失
- 配置项不生效
- 表面症状与实际问题不一致
- 需要追踪数据在多个环节的变化
