# LLM 驱动 vs 硬编码规则设计原则

## 模式识别

### 问题
系统使用硬编码规则（如字数 > 5000、数据量 > 10 万字）来判断任务是否需要分解，这种方式不够灵活，无法适应各种复杂场景。

### 原因
1. **硬编码规则无法理解任务的真实复杂度**：不同任务的复杂度标准不同（生成内容 vs 数据处理 vs 多步骤流程）
2. **硬编码规则难以扩展和维护**：每次新增场景都需要修改代码
3. **硬编码规则缺乏灵活性**：无法根据实际情况动态调整

### 解法
改为 LLM 驱动：
1. **通过系统提示词引导 LLM 分析任务内容**：让 LLM 理解任务的真实复杂度
2. **系统只提供工具和基础设施**：不做任务分析和分解
3. **通过系统提示词提供示例和规则**：引导 LLM 的行为

## 适用场景

当你需要实现以下功能时，应该优先考虑 LLM 驱动而不是硬编码规则：

1. **智能判断**：任务分解、内容分类、质量评估、复杂度分析等
2. **灵活适应**：不同用户、不同场景、不同需求
3. **动态调整**：根据实际情况调整行为
4. **可扩展性**：需要支持新的场景和需求

## 实施步骤

### 第一步：识别硬编码规则

检查代码中是否有以下模式：
- 固定的阈值判断（如 `if (wordCount > 5000)`）
- 固定的分类逻辑（如 `if (type === 'A') { ... } else if (type === 'B') { ... }`）
- 固定的策略选择（如 `switch (taskType) { case 'continuous': ... }`）

### 第二步：设计系统提示词

将硬编码规则转换为系统提示词：

**设计原则：**
1. **简单直接**：使用简单的语言，避免复杂的概念和条件判断
2. **负面规则**：使用"不要做"而不是"要做"
3. **可选步骤**：让 LLM 根据需要决定，不强制执行某些步骤
4. **具体示例**：提供具体的示例，而不是抽象的描述
5. **短句子**：使用短句子，避免复杂的从句

**示例：**
```markdown
## 任务分解

当你收到一个复杂的任务时，你可以将它分解成多个子任务。

### 什么时候需要分解任务？

你可以根据以下情况判断是否需要分解任务：
- 任务涉及大量内容生成（如生成 10000 字的文章）
- 任务涉及大量数据处理（如总结 100 万字的电子书）
- 任务涉及多个步骤（如先读取文件，再分析内容，最后生成报告）

### 示例

**用户请求**：请帮我生成一个 10000 字的科幻小说

**你的分解**：
1. 生成第 1-2000 字，包括开头和人物介绍
2. 生成第 2001-4000 字，继续故事发展
3. ...
```

### 第三步：提供工具和基础设施

系统只提供工具和基础设施，不做任务分析和分解：

1. **工具**：提供 LLM 可以调用的工具（如 `enqueue_task`）
2. **基础设施**：提供队列管理、循环检测、Hook 防护等基础设施
3. **反馈机制**：提供错误信息和正确做法的反馈

### 第四步：验证效果

验证 LLM 驱动的方式是否比硬编码规则更好：

- ✅ **灵活性**：LLM 是否能适应各种复杂场景？
- ✅ **准确性**：LLM 的判断是否比硬编码规则更准确？
- ✅ **可扩展性**：是否容易支持新的场景和需求？
- ✅ **可维护性**：是否减少了代码修改的频率？

## 对比表

| 维度 | 硬编码规则 | LLM 驱动 |
|------|-----------|----------|
| **灵活性** | ❌ 固定规则，难以适应新场景 | ✅ 动态判断，灵活适应 |
| **准确性** | ⚠️ 依赖阈值，可能误判 | ✅ 理解任务内容，更准确 |
| **可扩展性** | ❌ 每次新增场景都需要修改代码 | ✅ 只需修改系统提示词 |
| **可维护性** | ❌ 代码复杂，难以维护 | ✅ 系统提示词简单，易于维护 |
| **性能** | ✅ 快速判断 | ⚠️ 需要 LLM 调用（稍慢） |
| **成本** | ✅ 无额外成本 | ⚠️ 需要 LLM token 消耗 |

## 注意事项

1. **不是所有场景都适合 LLM 驱动**：
   - 如果规则非常简单且固定（如验证邮箱格式），硬编码规则更合适
   - 如果性能要求极高（如实时系统），硬编码规则更合适
   - 如果成本敏感（如高频调用），硬编码规则更合适

2. **系统提示词需要精心设计**：
   - 遵循"简单直接、负面规则、可选步骤、具体示例、短句子"的原则
   - 提供足够的示例和规则
   - 避免复杂的概念和条件判断

3. **需要提供反馈机制**：
   - 当 LLM 判断错误时，提供明确的错误信息
   - 当 LLM 行为不符合预期时，提供正确做法
   - 记录日志供调试

## 实战案例

### 案例：智能任务分解系统

**原设计（硬编码规则）：**
```typescript
function isLargeTask(task: string): boolean {
  const wordCount = task.split(' ').length;
  if (wordCount > 5000) return true;
  
  const dataSize = estimateDataSize(task);
  if (dataSize > 100000) return true;
  
  const stepCount = countSteps(task);
  if (stepCount > 3) return true;
  
  return false;
}
```

**问题：**
- 字数阈值（5000）是否合理？不同任务的复杂度标准不同
- 如何估算数据大小？`estimateDataSize` 的实现可能不准确
- 如何计数步骤？`countSteps` 的实现可能不准确
- 无法适应新的场景（如并行处理多个文件）

**新设计（LLM 驱动）：**
```markdown
## 任务分解

当你收到一个复杂的任务时，你可以将它分解成多个子任务。

### 什么时候需要分解任务？

你可以根据以下情况判断是否需要分解任务：
- 任务涉及大量内容生成（如生成 10000 字的文章）
- 任务涉及大量数据处理（如总结 100 万字的电子书）
- 任务涉及多个步骤（如先读取文件，再分析内容，最后生成报告）
- 任务需要并行处理多个文件或数据

### 如何分解任务？

1. **分析任务**：理解用户的需求，识别任务的关键步骤
2. **创建子任务**：为每个步骤创建一个子任务，使用 `enqueue_task` 工具
3. **提供清晰的 prompt**：每个子任务的 prompt 应该清晰、具体、可执行
4. **提供简短的 summary**：每个子任务的 summary 应该简短地描述任务的目标

### 示例

**用户请求**：请帮我生成一个 10000 字的科幻小说

**你的分解**：
1. 调用 `enqueue_task`，prompt: "请生成科幻小说的第 1-2000 字，包括开头和人物介绍"，summary: "生成小说第 1-2000 字"
2. 调用 `enqueue_task`，prompt: "请生成科幻小说的第 2001-4000 字，继续故事发展"，summary: "生成小说第 2001-4000 字"
3. ...
```

**优势：**
- ✅ LLM 可以根据任务内容动态判断是否需要分解
- ✅ LLM 可以根据实际情况调整分解策略
- ✅ 只需修改系统提示词，无需修改代码
- ✅ 容易支持新的场景和需求

## 总结

**核心原则：**
- 当需要"智能判断"时，优先考虑 LLM 驱动
- 当需要"灵活适应"时，优先考虑 LLM 驱动
- 当需要"可扩展性"时，优先考虑 LLM 驱动

**实施步骤：**
1. 识别硬编码规则
2. 设计系统提示词
3. 提供工具和基础设施
4. 验证效果

**注意事项：**
- 不是所有场景都适合 LLM 驱动
- 系统提示词需要精心设计
- 需要提供反馈机制
