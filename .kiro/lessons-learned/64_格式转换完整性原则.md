# 格式转换完整性原则

**问题类型**：格式转换、API 兼容性  
**关键词**：格式转换、消息类型、toolResult、混合 API、provider 字段、条件判断失败

---

## 问题现象

**场景 1：格式转换代码不执行**
- 代码有正确的格式转换逻辑
- 但条件判断失败，代码分支不执行
- 日志中没有格式转换的输出

**场景 2：部分消息类型未转换**
- 格式转换函数只处理部分消息类型
- 导致某些消息格式错误
- 例如：`functionResponse.name` 变成 `"unknown"`

---

## 根本原因

### 原因 1：条件判断依赖的字段为空

**示例**：
```typescript
if (base.provider && base.provider.toLowerCase().includes("vectorengine")) {
  // 格式转换代码
}
```

**问题**：
- `base.provider` 是空字符串
- 条件判断失败，代码不执行

**根因**：
- provider 信息可能在不同字段中（`provider` 或 `modelId`）
- 代码只检查了一个字段

### 原因 2：格式转换不完整

**示例**：
```typescript
// 只处理 role: "tool"
if (role === "tool") {
  // 转换逻辑
}

// 没有处理 role: "toolResult"
```

**问题**：
- Session 中保存的是 `role: "toolResult"`（Clawdbot 内部格式）
- 格式转换函数只处理 `role: "tool"`（OpenAI 格式）
- 导致 `role: "toolResult"` 消息没有被转换

## 调试方法

### 步骤 1：确认代码已生效

```powershell
# 检查源文件修改时间
Get-Item "src/agents/xxx.ts" | Select LastWriteTime

# 检查 dist 文件修改时间
Get-Item "dist/agents/xxx.js" | Select LastWriteTime

# 检查 dist 文件内容
Select-String -Path "dist/agents/xxx.js" -Pattern "关键代码" -Context 2,2
```

### 步骤 2：确认代码被执行

```powershell
# 检查日志中是否有相关输出
$trace = Get-Content "C:\Users\zouta\.clawdbot\runtimelog\trace__*.jsonl" -Encoding UTF8 | 
  ForEach-Object { $_ | ConvertFrom-Json }

$logs = $trace | Where-Object { 
  $_.payload -and $_.payload.message -and $_.payload.message -like "*关键词*" 
}

Write-Host "找到 $($logs.Count) 个相关日志"
```

### 步骤 3：如果没有日志，检查条件判断

**提取 trace 日志，查看字段实际值**：
```powershell
$payloads = $trace | Where-Object { $_.event -eq "llm.payload" }
$seq1 = $payloads[0].payload

Write-Host "provider: $($seq1.provider)"
Write-Host "model: $($seq1.model)"
Write-Host "api: $($seq1.api)"
```

**对比代码中的条件判断**：
```typescript
// 代码检查的字段
if (base.provider && base.provider.toLowerCase().includes("vectorengine")) {
  // ...
}

// 实际的字段值
// provider: ""  ❌ 空字符串
// model: "vectorengine/gemini-3-flash-preview"  ✅ 包含 provider 信息
```

### 步骤 4：修复条件判断

**方案 1：从其他字段提取信息**
```typescript
// 从 modelId 中提取 provider
const modelStr = typeof base.modelId === "string" ? base.modelId : "";
const providerFromModel = modelStr.includes("/") ? modelStr.split("/")[0] : "";
const effectiveProvider = (base.provider || providerFromModel).toLowerCase();

if (effectiveProvider.includes("vectorengine")) {
  // 格式转换代码
}
```

**方案 2：检查多个字段**
```typescript
const provider = base.provider || "";
const model = base.modelId || "";
const isVectorengine = provider.includes("vectorengine") || model.includes("vectorengine");

if (isVectorengine) {
  // 格式转换代码
}
```

### 步骤 5：检查格式转换完整性

**提取 session 日志，查看保存的消息格式**：
```powershell
$sessionFile = Get-ChildItem "C:\Users\zouta\.clawdbot\agents\main\sessions\*.jsonl" | 
  Sort-Object LastWriteTime -Descending | Select-Object -First 1

$messages = Get-Content $sessionFile.FullName -Encoding UTF8 | 
  ForEach-Object { $_ | ConvertFrom-Json }

# 查看工具结果消息
$toolResults = $messages | Where-Object { 
  $_.message -and $_.message.role -eq "toolResult" 
}

$toolResults[0].message | ConvertTo-Json -Depth 10
```

**对比格式转换函数**：
```typescript
// 格式转换函数只处理 role: "tool"
if (role === "tool") {
  // 转换逻辑
}

// 但 session 中保存的是 role: "toolResult"
// 需要添加对 role: "toolResult" 的处理
if (role === "toolresult") {
  // 转换逻辑
}
```

## 修复原则

### 原则 1：字段 fallback

**不要假设字段一定有值**：
```typescript
// ❌ 错误：假设 provider 字段一定有值
if (base.provider.includes("vectorengine")) {
  // ...
}

// ✅ 正确：使用 fallback
const effectiveProvider = (base.provider || providerFromModel).toLowerCase();
if (effectiveProvider.includes("vectorengine")) {
  // ...
}
```

### 原则 2：格式转换完整性

**处理所有可能的消息类型**：
```typescript
// ❌ 错误：只处理部分消息类型
if (role === "tool") {
  // 转换逻辑
}

// ✅ 正确：处理所有可能的消息类型
if (role === "tool") {
  // OpenAI 格式的工具结果
}

if (role === "toolresult") {
  // Clawdbot 内部格式的工具结果
}
```

### 原则 3：验证条件判断

**修复后验证条件判断是否生效**：
1. 添加日志输出
2. 检查日志中是否有输出
3. 如果没有输出，说明条件判断仍然失败

## 实战案例

### 案例：vectorengine 无限循环问题

**问题**：
- LLM 不断重复调用同一个工具
- `functionResponse.name` 是 `"unknown"`

**调试过程**：
1. **第一次修复**：添加 `role: "toolResult"` 的格式转换逻辑
2. **验证**：构建成功，但问题仍然存在
3. **检查日志**：没有格式转换的日志输出
4. **检查条件判断**：`base.provider` 是空字符串
5. **第二次修复**：从 `modelId` 中提取 provider
6. **验证**：构建成功，等待用户测试

**关键教训**：
- 不要假设字段一定有值
- 修复后必须验证条件判断是否生效
- 如果没有日志输出，说明条件判断失败

## 检查清单

修复格式转换问题时，必须检查：

- [ ] **确认代码已生效**：检查 dist 文件和时间戳
- [ ] **确认代码被执行**：检查日志中是否有相关输出
- [ ] **如果没有日志**：检查条件判断依赖的字段
- [ ] **提取 trace 日志**：查看字段实际值
- [ ] **修复条件判断**：使用 fallback 或检查多个字段
- [ ] **检查格式转换完整性**：处理所有可能的消息类型
- [ ] **验证修复效果**：重启后检查日志输出
