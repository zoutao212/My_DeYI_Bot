# 全局拦截器错误处理规范

> **核心原则**：所有全局拦截器必须有完整的错误处理，避免错误传播到顶层导致系统崩溃。

---

## 问题场景

**触发条件**：
1. 全局拦截器（如 `globalThis.fetch`）没有 try-catch
2. 特殊错误（如 AbortError）在拦截器中抛出
3. 错误传播到顶层，导致 unhandledRejection
4. 系统崩溃或重启

**典型案例**：
- `llm-gated-fetch.ts` 的 `wrapped` 函数缺少 try-catch
- API 返回错误后，`runAbortController.abort()` 被调用
- AbortError 传播到顶层，导致 Gateway 崩溃

---

## 根本原因

### 1. 全局拦截器的特殊性

**特点**：
- 全局拦截器是"基础设施代码"
- 所有调用都会经过拦截器
- 拦截器中的错误会影响所有调用

**风险**：
- 如果拦截器没有错误处理，任何错误都会传播到顶层
- 导致 unhandledRejection 和系统崩溃

### 2. 特殊错误的传播性

**AbortError 的特点**：
- 由 `AbortController.abort()` 触发
- 会传播到所有使用 `signal` 的异步操作
- 必须被正确处理，否则会导致 unhandledRejection

**其他特殊错误**：
- TimeoutError
- NetworkError
- PermissionError

---

## 标准规范

### 规范 1：所有全局拦截器必须有 try-catch

**错误做法**：
```typescript
// ❌ 没有 try-catch
globalThis.fetch = async (input, init) => {
  // 业务逻辑...
  return await original(input, init);
};
```

**正确做法**：
```typescript
// ✅ 有 try-catch
globalThis.fetch = async (input, init) => {
  try {
    // 业务逻辑...
    return await original(input, init);
  } catch (error) {
    // 捕获并记录错误
    console.warn("[fetch-interceptor] Error:", error);
    // 重新抛出，让上层处理
    throw error;
  }
};
```

### 规范 2：特殊错误必须单独处理

**原因**：
- 特殊错误（如 AbortError）有特定的语义
- 需要单独记录和处理

**示例**：
```typescript
globalThis.fetch = async (input, init) => {
  try {
    return await original(input, init);
  } catch (error) {
    // 特殊错误单独处理
    if (error && typeof error === "object" && "name" in error) {
      if (error.name === "AbortError") {
        console.warn("[fetch-interceptor] Request aborted:", error);
        throw error;
      }
      if (error.name === "TimeoutError") {
        console.warn("[fetch-interceptor] Request timeout:", error);
        throw error;
      }
    }
    // 其他错误
    console.warn("[fetch-interceptor] Error:", error);
    throw error;
  }
};
```

### 规范 3：错误必须重新抛出

**原因**：
- 拦截器不应该"吞掉"错误
- 上层需要知道错误发生了
- 保持错误传播链完整

**错误做法**：
```typescript
// ❌ 吞掉错误
try {
  return await original(input, init);
} catch (error) {
  console.error("Error:", error);
  return null; // 错误：吞掉了错误
}
```

**正确做法**：
```typescript
// ✅ 重新抛出错误
try {
  return await original(input, init);
} catch (error) {
  console.warn("Error:", error);
  throw error; // 正确：重新抛出
}
```

## 验证方法

### 1. 代码审查

**检查清单**：
- [ ] 所有全局拦截器都有 try-catch
- [ ] 特殊错误（AbortError、TimeoutError）单独处理
- [ ] 错误被重新抛出，不被吞掉
- [ ] 错误日志包含足够的上下文信息

### 2. 测试验证

**测试场景**：
- 模拟 AbortError
- 模拟 TimeoutError
- 模拟网络错误
- 验证错误是否被正确捕获和记录

**示例**：
```typescript
test("fetch interceptor handles AbortError", async () => {
  const controller = new AbortController();
  controller.abort();
  
  await expect(
    fetch("https://example.com", { signal: controller.signal })
  ).rejects.toThrow("AbortError");
  
  // 验证日志
  expect(console.warn).toHaveBeenCalledWith(
    expect.stringContaining("Request aborted")
  );
});
```

### 3. 运行时监控

**监控指标**：
- unhandledRejection 事件数量
- 系统崩溃次数
- 错误日志中的拦截器错误

## 实战案例

### 案例 1：llm-gated-fetch.ts

**问题**：
- `wrapped` 函数没有 try-catch
- AbortError 传播到顶层
- Gateway 崩溃

**修复**：
```typescript
const wrapped: typeof fetch = async (input, init) => {
  try {
    // 业务逻辑...
    return await executeRequestWithRetry(attemptKey, input, init);
  } catch (error) {
    // 捕获 AbortError
    if (error && typeof error === "object" && "name" in error && error.name === "AbortError") {
      console.warn("[llm-gated-fetch] Request aborted:", error);
      throw error;
    }
    throw error;
  }
};
```

**效果**：
- AbortError 被正确捕获和记录
- 错误被重新抛出，由上层处理
- Gateway 不再崩溃

## 关键教训

### 1. 全局拦截器是"基础设施代码"

**特点**：
- 影响范围广
- 错误影响大
- 容易被忽视

**要求**：
- 必须有完整的错误处理
- 必须有详细的日志
- 必须有充分的测试

### 2. 特殊错误需要特殊处理

**原因**：
- 特殊错误有特定的语义
- 需要单独记录和处理
- 不能一概而论

**示例**：
- AbortError：请求被取消
- TimeoutError：请求超时
- NetworkError：网络错误

### 3. 错误传播链必须完整

**原则**：
- 拦截器不应该"吞掉"错误
- 错误必须重新抛出
- 保持错误传播链完整

**数据流**：
```
错误产生
  ↓ 捕获并记录
拦截器
  ↓ 重新抛出
上层处理
  ↓ 处理或忽略
```

---

## 检查清单

在编写或审查全局拦截器时，使用以下检查清单：

- [ ] **有 try-catch**：所有异步操作都在 try-catch 中
- [ ] **特殊错误单独处理**：AbortError、TimeoutError 等单独处理
- [ ] **错误重新抛出**：不吞掉错误，重新抛出让上层处理
- [ ] **日志完整**：错误日志包含足够的上下文信息
- [ ] **测试覆盖**：有测试覆盖错误场景
- [ ] **监控到位**：有监控指标跟踪拦截器错误
