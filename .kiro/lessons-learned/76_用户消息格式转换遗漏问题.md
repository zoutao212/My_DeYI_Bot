# 用户消息格式转换遗漏问题

**问题类型**：格式转换不完整  
**影响**：LLM 无限循环、对话历史格式混乱  
**严重程度**：🔴 严重

---

## 问题描述

在实现 OpenAI 格式到 Gemini 格式的转换时，只转换了 assistant 消息和 tool 消息，**遗漏了用户消息的转换**，导致对话历史中混合了两种格式，LLM 无法正确理解历史消息，出现无限循环。

---

## 问题现象

### 症状

- LLM 在每次工具调用后都重新读取 SOUL.md, USER.md
- 无限循环执行相同的工具调用
- 用户无法正常对话

### 日志证据

```
→ seq=2 model=vectorengine/gemini-3-flash-preview
→ seq=3 model=vectorengine/gemini-3-flash-preview
...
→ seq=23 model=vectorengine/gemini-3-flash-preview
```

---

## 根本原因

### 格式转换不完整

**问题**：`convertOpenAIToGeminiFormat` 函数只转换了：
- `role: assistant` → `role: model`
- `role: tool` → `role: user` (with `functionResponse`)
- `role: toolresult` → `role: user` (with `functionResponse`)

**遗漏**：没有转换 `role: user` 的消息！

### 对话历史格式混乱

**OpenAI 格式（用户消息）**：
```json
{
  "role": "user",
  "content": [
    {
      "type": "text",
      "text": "..."
    }
  ]
}
```

**Gemini 格式（LLM 返回）**：
```json
{
  "role": "model",
  "parts": [
    {
      "functionCall": {
        "name": "read",
        "args": { "path": "SOUL.md" }
      }
    }
  ]
}
```

**结果**：对话历史中混合了两种格式，导致 LLM 无法正确理解历史消息。

## 修复方案

### 修复位置

**文件**：`src/agents/gemini-payload-thought-signature.ts`  
**函数**：`convertOpenAIToGeminiFormat`

### 修复内容

在 `convertOpenAIToGeminiFormat` 函数中，添加用户消息的格式转换逻辑：

```typescript
// 🆕 Fix: Convert user messages from OpenAI format to Gemini format
// OpenAI format: { role: "user", content: [{ type: "text", text: "..." }] }
// Gemini format: { role: "user", parts: [{ text: "..." }] }
if (role === "user") {
  const content = msgRec.content;
  
  // If content is an array (OpenAI format), convert to parts (Gemini format)
  if (Array.isArray(content)) {
    const parts = content
      .map((item) => {
        if (!item || typeof item !== "object") return null;
        const itemRec = item as Record<string, unknown>;
        
        // Extract text from { type: "text", text: "..." }
        if (itemRec.type === "text" && typeof itemRec.text === "string") {
          return { text: itemRec.text };
        }
        
        // Keep other types as-is (e.g., image_url)
        return item;
      })
      .filter(Boolean);
    
    if (parts.length > 0) {
      log.debug(`[format] Converted user message from OpenAI format to Gemini format (${parts.length} parts)`);
      return {
        role: "user",
        parts: parts,
      };
    }
  }
  
  // If content is a string (simple format), convert to parts
  if (typeof content === "string" && content.length > 0) {
    log.debug(`[format] Converted user message from string to Gemini format`);
    return {
      role: "user",
      parts: [{ text: content }],
    };
  }
  
  // Return as-is if already in Gemini format or empty
  return msg;
}
```

### 修复逻辑

1. **检查 `role: user` 的消息**
2. **如果 `content` 是数组（OpenAI 格式）**：
   - 提取每个元素的 `text` 字段
   - 转换为 `parts` 数组（Gemini 格式）
3. **如果 `content` 是字符串**：
   - 转换为 `parts: [{ text: content }]`
4. **如果已经是 Gemini 格式**：
   - 保持不变

## 关键教训

### 1. 格式转换必须完整

**问题**：只转换了部分消息类型，遗漏了用户消息

**教训**：
- 转换所有类型的消息（user, assistant, tool, toolresult, system）
- 确保没有遗漏
- 列出所有消息类型的清单，逐一检查

### 2. 对话历史格式必须统一

**问题**：混合使用 OpenAI 格式和 Gemini 格式会导致 LLM 行为异常

**教训**：
- 所有消息都使用同一种格式
- 在发送给 API 前统一转换
- 不要让不同格式的消息混合在一起

### 3. 调试要看完整数据流

**问题**：只看日志级别（debug/info），没有看 trace 事件

**教训**：
- 检查 `llm.payload` 事件，看实际发送的数据
- 检查 `patch.thought_signature.scan` 事件，看 patcher 的执行结果
- 不要只依赖日志级别

### 4. 验证要多维度

**问题**：只看"修复成功"，没有看"不再需要修复"

**教训**：
- 验证修复代码是否被执行（日志）
- 验证修复是否生效（payload 格式）
- 验证问题是否解决（LLM 行为）

### 5. 格式转换的完整性检查清单

**必须转换的消息类型**：
- [ ] `role: user` - 用户消息
- [ ] `role: assistant` - LLM 返回
- [ ] `role: tool` - 工具结果（OpenAI 格式）
- [ ] `role: toolresult` - 工具结果（Clawdbot 内部格式）
- [ ] `role: system` - 系统提示词（如果需要）

**转换内容**：
- [ ] `content` → `parts`（用户消息）
- [ ] `role: assistant` → `role: model`（LLM 返回）
- [ ] `tool_calls` → `functionCall`（工具调用）
- [ ] `tool` → `functionResponse`（工具结果）
