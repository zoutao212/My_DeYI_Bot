# 全局上下文数据过期问题

**问题类型**：数据流 / 全局状态管理  
**严重程度**：⚠️ 中等（可能导致逻辑错误）  
**发现日期**：2026-02-04

---

## 问题模式

### 现象

全局上下文在多次调用之间共享，可能包含过期的数据（来自上一次调用），导致读取到错误的值。

**典型场景**：
- 函数 A 设置全局上下文 `context.value = "A"`
- 函数 B 读取全局上下文，期望得到 "B"，但实际得到 "A"（过期数据）
- 导致逻辑错误或行为异常

### 根本原因

1. **全局上下文的生命周期跨越多次函数调用**
   - 全局变量在进程生命周期内持续存在
   - 不同的函数调用可能读取到上一次调用设置的值

2. **没有明确的"最新数据"优先级**
   - 多个数据源（全局上下文、函数参数、session 等）
   - 不清楚应该优先使用哪个数据源

3. **缺少数据过期检测机制**
   - 没有时间戳或版本号
   - 无法判断数据是否过期

## 实战案例

### 案例：Hook 副作用导致的队列任务 Prompt 错误

**背景**：
- 用户发送消息："帮我生成 5 段内容..."
- LLM 调用 5 次 `enqueue_task`，创建 5 个队列任务
- 执行第一个队列任务时，Hook 读取的 prompt 是原始用户消息，而不是队列任务的 prompt

**数据流**：
```
用户消息处理:
  setCurrentFollowupRunContext({ prompt: "帮我生成 5 段内容...", isQueueTask: false })
  ↓
LLM 调用 enqueue_task:
  创建队列任务 { prompt: "请生成第 1 段内容..." }
  ↓
队列任务执行:
  setCurrentFollowupRunContext({ prompt: "请生成第 1 段内容...", isQueueTask: true })
  ↓
Hook 调用:
  读取 params.prompt → "帮我生成 5 段内容..." ❌（过期数据）
  应该读取 followupContext.prompt → "请生成第 1 段内容..." ✅
```

**问题**：
- Hook 调用时，全局上下文已经被更新为队列任务的上下文
- 但 Hook 使用的是 `params.prompt`（函数参数），而不是 `followupContext.prompt`（全局上下文）
- 导致读取到过期的数据

**修复**：
```typescript
// ❌ 错误：只使用函数参数
const hookResult = await hookRunner.runBeforeAgentStart({
  prompt: params.prompt,  // 可能是过期数据
});

// ✅ 正确：优先使用全局上下文，回退到函数参数
const actualPrompt = followupContext?.prompt ?? params.prompt;
const hookResult = await hookRunner.runBeforeAgentStart({
  prompt: actualPrompt,  // 使用最新数据
});
```

---

## 解决方案

### 方案 1：优先使用全局上下文（推荐）

**原则**：优先使用全局上下文中的最新数据，提供回退方案。

**模式**：
```typescript
const actual = globalContext?.value ?? fallbackValue;
```

**示例**：
```typescript
// 从全局上下文获取最新数据
const followupContext = getCurrentFollowupRunContext();

// 优先使用全局上下文，回退到函数参数
const actualPrompt = followupContext?.prompt ?? params.prompt;
const actualSessionKey = followupContext?.run.sessionKey ?? params.sessionKey;
```

**优点**：
- 简单直接
- 总是使用最新数据
- 有回退方案（如果全局上下文为空）

**缺点**：
- 依赖全局状态
- 需要确保全局上下文被正确设置

### 方案 2：添加数据版本号

**原则**：为全局上下文添加版本号或时间戳，判断数据是否过期。

**模式**：
```typescript
type ContextWithVersion = {
  value: string;
  version: number;
  timestamp: number;
};

// 设置时增加版本号
setGlobalContext({ value: "new", version: currentVersion + 1, timestamp: Date.now() });

// 读取时检查版本号
const context = getGlobalContext();
if (context.version < expectedVersion) {
  // 数据过期，使用回退方案
}
```

**优点**：
- 可以明确判断数据是否过期
- 更安全

**缺点**：
- 实现复杂
- 需要维护版本号

### 方案 3：避免使用全局上下文

**原则**：通过函数参数显式传递所有需要的数据，避免使用全局状态。

**模式**：
```typescript
// ❌ 使用全局上下文
function processData() {
  const value = globalContext.value;
}

// ✅ 通过参数传递
function processData(value: string) {
  // 使用参数
}
```

**优点**：
- 最安全
- 数据流清晰
- 易于测试

**缺点**：
- 需要修改函数签名
- 参数可能很多

## 最佳实践

### 1. 明确数据优先级

当有多个数据源时，明确优先级：

```typescript
// 优先级：全局上下文 > 函数参数 > 默认值
const value = globalContext?.value ?? params.value ?? defaultValue;
```

### 2. 在关键节点验证数据

在使用数据前，验证数据是否正确：

```typescript
const actualPrompt = followupContext?.prompt ?? params.prompt;

// 验证数据
if (!actualPrompt) {
  throw new Error("Prompt is required");
}

// 添加日志
log.debug(`Using prompt: ${actualPrompt.slice(0, 50)}...`);
```

### 3. 清理全局上下文

在函数执行完成后，清理全局上下文：

```typescript
try {
  setGlobalContext({ value: "new" });
  // 执行逻辑
} finally {
  setGlobalContext(null);  // 清理
}
```

### 4. 使用 TypeScript 类型保护

使用 TypeScript 类型保护确保数据存在：

```typescript
const context = getGlobalContext();

if (context?.value) {
  // TypeScript 知道 context.value 存在
  processValue(context.value);
}
```

## 调试方法

### 1. 添加日志追踪数据流

```typescript
log.debug(`[Context] Setting: ${JSON.stringify(context)}`);
log.debug(`[Context] Reading: ${JSON.stringify(getGlobalContext())}`);
log.debug(`[Context] Using: ${actualValue}`);
```

### 2. 检查全局上下文的设置和读取

```typescript
// 设置时
console.log(`[Set] context.value = ${value}`);

// 读取时
const context = getGlobalContext();
console.log(`[Get] context.value = ${context?.value}`);
console.log(`[Get] params.value = ${params.value}`);
console.log(`[Get] actual.value = ${actualValue}`);
```

### 3. 使用断点调试

在关键位置设置断点：
- 全局上下文设置处
- 全局上下文读取处
- 数据使用处

检查：
- 全局上下文的值是否正确
- 是否被意外覆盖
- 是否在正确的时机读取

## 检查清单

当遇到全局上下文相关问题时，检查：

- [ ] **全局上下文是否被正确设置？**
  - 检查 `setGlobalContext` 的调用位置
  - 检查设置的值是否正确

- [ ] **全局上下文是否被意外覆盖？**
  - 检查是否有多处设置全局上下文
  - 检查设置的顺序是否正确

- [ ] **读取全局上下文的时机是否正确？**
  - 检查读取时全局上下文是否已经被设置
  - 检查是否读取到过期的数据

- [ ] **是否有回退方案？**
  - 检查如果全局上下文为空，是否有默认值
  - 检查回退方案是否正确

- [ ] **是否清理了全局上下文？**
  - 检查函数执行完成后是否清理
  - 检查是否有内存泄漏

## 关键教训

1. **全局上下文是共享的**
   - 在多次调用之间持续存在
   - 可能包含过期的数据

2. **优先使用最新数据**
   - 优先使用全局上下文中的最新数据
   - 提供回退方案（函数参数、默认值）

3. **明确数据优先级**
   - 当有多个数据源时，明确优先级
   - 使用 `??` 运算符实现回退

4. **在关键节点验证数据**
   - 验证数据是否存在
   - 验证数据是否正确
   - 添加日志追踪数据流

5. **清理全局上下文**
   - 函数执行完成后清理
   - 避免内存泄漏
