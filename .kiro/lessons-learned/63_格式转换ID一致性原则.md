# 格式转换 ID 一致性原则

**日期**：2026-02-03  
**问题类型**：格式转换、ID 匹配、无限循环

---

## 问题现象

当系统进行格式转换时（如 Gemini ↔ OpenAI），如果 ID 不一致，会导致严重问题：
- LLM 陷入无限循环
- 不断重复调用同一个工具
- 无法生成最终回复

**典型症状**：
- seq 不断递增（seq=2, 3, 4, 5...）
- 每次调用的内容完全相同
- payload 大小递增（历史消息越来越多）
- LLM 没有生成最终回复

---

## 问题根因

**格式转换时使用了随机 ID，导致 ID 不匹配。**

### 数据流分析

```
❌ 错误的数据流：

seq=2: LLM 调用工具
  → functionCall { name: "write", args: {...} }
  → 转换为 OpenAI 格式时生成随机 ID: "call_abc123"
  → 保存到 session

seq=3: 工具结果返回
  → functionResponse { name: "write", response: {...} }
  → 转换为 OpenAI 格式时生成新的随机 ID: "call_xyz789"
  → 保存到 session

seq=4: LLM 读取历史
  → 看到 tool_call_id="call_abc123"
  → 但工具结果的 tool_call_id="call_xyz789"
  → ❌ ID 不匹配！
  → LLM 认为工具还没有返回结果
  → 继续调用工具
```

### 根本原因

1. **源格式中没有 ID**
   - Gemini 格式的 `functionCall` 没有 `id` 字段
   - Gemini 格式的 `functionResponse` 也没有 `id` 字段

2. **转换时随机生成 ID**
   - 每次转换都生成新的随机 ID
   - 导致工具调用和结果的 ID 不匹配

3. **LLM 依赖 ID 匹配**
   - OpenAI 格式中，`tool_call_id` 用于匹配工具调用和结果
   - 如果 ID 不匹配，LLM 认为工具还没有返回结果

---

## 解决方案

### 方案 1：生成稳定的 ID

**思路**：基于工具名称和参数生成稳定的 ID

**实现**：
```typescript
function generateStableToolCallId(name: string, args: unknown): string {
  const argsStr = typeof args === "object" ? JSON.stringify(args) : String(args);
  const seed = `${name}:${argsStr}`;
  // 使用 hash 生成稳定的 ID
  let hash = 0;
  for (let i = 0; i < seed.length; i++) {
    const char = seed.charCodeAt(i);
    hash = ((hash << 5) - hash) + char;
    hash = hash & hash;
  }
  return `call_${Math.abs(hash).toString(36)}`;
}
```

**优点**：
- 同样的工具名称和参数，生成同样的 ID
- 工具调用和结果的 ID 一致

**缺点**：
- 工具结果中可能没有参数信息
- 需要从历史消息中查找对应的工具调用
- 实现复杂

### 方案 2：保持原始格式（推荐）

**思路**：针对特殊 provider，session 保存原始格式

**实现**：
```typescript
// 针对 vectorengine，保持 Gemini 格式
// 不需要转换为 OpenAI 格式
// 避免 ID 不匹配问题
```

**优点**：
- 简单、可靠
- 避免格式转换
- 避免 ID 不匹配

**缺点**：
- 违反了"Session 格式统一"原则
- 但对于特殊 provider，这是必要的

### 方案 3：在转换时传递 ID

**思路**：在格式转换时，保留原始 ID

**实现**：
```typescript
// 在 functionCall 中添加 ID
const id = generateId();
functionCall.id = id;

// 在 functionResponse 中使用同样的 ID
functionResponse.id = id;
```

**优点**：
- 保持 ID 一致性
- 格式转换后仍然可以匹配

**缺点**：
- 需要修改源格式
- 可能不被 API 接受

## 格式转换 ID 一致性原则

### 核心原则

**格式转换时，必须保持 ID 一致性。**

### 实现规则

1. **如果源格式有 ID**
   - 必须保留原始 ID
   - 不要生成新的 ID

2. **如果源格式没有 ID**
   - 使用稳定的 ID 生成算法（基于内容的 hash）
   - 或者保持原始格式，避免转换

3. **如果无法保持 ID 一致性**
   - 不要进行格式转换
   - 保持原始格式

### 验证方法

**检查点**：
- ✅ 工具调用的 ID 是否稳定？
- ✅ 工具结果的 ID 是否与调用的 ID 一致？
- ✅ LLM 是否能正确匹配工具调用和结果？
- ✅ 是否会出现无限循环？

**验证步骤**：
1. 提取 trace 日志
2. 检查工具调用的 ID
3. 检查工具结果的 ID
4. 验证 ID 是否一致
5. 验证 LLM 是否生成了最终回复

## 常见错误

### 错误 1：使用随机 ID

**问题**：每次转换都生成新的随机 ID

```typescript
// ❌ 错误
tool_call_id: `call_${Math.random().toString(36).substring(2, 15)}`
```

**后果**：ID 不匹配，LLM 陷入无限循环

**解决**：使用稳定的 ID 生成算法

### 错误 2：忽略 ID 匹配

**问题**：只关注格式转换，忽略 ID 匹配

**后果**：格式正确，但 ID 不匹配，LLM 无法正常工作

**解决**：验证 ID 是否一致

### 错误 3：强制格式统一

**问题**：为了"格式统一"，强行转换所有格式

**后果**：某些 provider 的格式转换会导致 ID 不匹配

**解决**：针对特殊 provider，保持原始格式

---

## 适用场景

**当系统需要进行格式转换时**：
- OpenAI ↔ Gemini
- OpenAI ↔ Claude
- OpenAI ↔ 其他自定义格式

**关键判断**：
- 如果格式中有 ID 字段，必须保留
- 如果格式中没有 ID 字段，必须使用稳定的 ID 生成算法
- 如果无法保持 ID 一致性，不要转换

## 调试方法

### 1. 观察现象

**症状**：
- 无限循环
- 重复调用同一个工具
- payload 大小递增

### 2. 检查 ID

**步骤**：
```powershell
# 提取 trace 日志
$trace = Get-Content "C:\Users\zouta\.clawdbot\runtimelog\trace__*.jsonl" -Encoding UTF8 | ConvertFrom-Json

# 检查工具调用的 ID
$payloads = $trace | Where-Object { $_.event -eq "llm.payload" }
$payloads[0].payload.payload.messages | Where-Object { $_.role -eq "assistant" } | Select tool_calls

# 检查工具结果的 ID
$payloads[1].payload.payload.messages | Where-Object { $_.role -eq "tool" } | Select tool_call_id
```

### 3. 验证 ID 是否一致

**检查点**：
- 工具调用的 `tool_calls[].id`
- 工具结果的 `tool_call_id`
- 两者是否一致？

### 4. 找到 ID 生成的位置

**搜索**：

## 关键教训

1. **格式转换必须保持 ID 一致性**
   - 不要使用随机 ID
   - 必须使用稳定的 ID 生成算法

2. **验证工具调用的完整流程**
   - 不要只看"工具调用成功"
   - 要验证"LLM 是否看到了工具结果"
   - 要验证"LLM 是否生成了最终回复"

3. **无限循环的常见原因**
   - tool_call_id 不匹配
   - 工具结果格式错误
   - LLM 没有看到工具结果

4. **不是所有格式都适合转换**
   - 如果格式中没有 ID，转换会很复杂
   - 保持原始格式可能是更好的选择

5. **格式统一不是绝对的**
   - 对于特殊 provider，可以保存原始格式
   - 重要的是避免 ID 不匹配，而不是强制统一
