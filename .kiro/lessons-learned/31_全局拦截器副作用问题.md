# 全局拦截器副作用问题

**问题类型**：全局拦截器、副作用、性能问题

**关键词**：全局拦截、fetch、副作用、上下文检查、性能优化

---

## 问题描述

全局拦截器（如 `llm-gated-fetch`）会拦截**所有**请求，包括不应该被拦截的请求（如 Embeddings API、内部服务等），导致：

1. **性能问题**：所有请求都被延迟（请求间隔控制）
2. **功能异常**：非目标请求被错误处理
3. **系统崩溃**：重试机制触发，导致级联失败

---

## 实战案例

### 场景

`llm-gated-fetch.ts` 拦截所有 `fetch` 请求，用于 LLM 请求审批。

### 问题代码

```typescript
const wrapped: typeof fetch = async (input: RequestInfo | URL, init?: RequestInit) => {
  // ❌ 错误：先执行拦截逻辑，再检查上下文
  const now = Date.now();
  const timeSinceLastRequest = now - lastRequestTime;
  if (timeSinceLastRequest < MIN_REQUEST_INTERVAL_MS) {
    await new Promise(resolve => setTimeout(resolve, waitTime));
  }
  lastRequestTime = Date.now();
  
  const ctx = getLlmRequestContext();
  if (!ctx) {
    return await original(input, init); // 已经被延迟了
  }
  // ...
};
```

**问题**：
- Embeddings API 请求没有 LLM 请求上下文（`ctx === null`）
- 但仍然被请求间隔控制延迟
- 导致并发 Embeddings 请求被串行化，性能下降
- 重试机制触发，最终崩溃

### 修复代码

```typescript
const wrapped: typeof fetch = async (input: RequestInfo | URL, init?: RequestInit) => {
  const ctx = getLlmRequestContext();
  
  // ✅ 正确：先检查上下文，再决定是否拦截
  if (!ctx) {
    return await original(input, init); // 直接放行，不执行任何拦截逻辑
  }
  
  // 只对 LLM 请求执行拦截逻辑
  const now = Date.now();
  const timeSinceLastRequest = now - lastRequestTime;
  if (timeSinceLastRequest < MIN_REQUEST_INTERVAL_MS) {
    await new Promise(resolve => setTimeout(resolve, waitTime));
  }
  lastRequestTime = Date.now();
  // ...
};
```

**修复原理**：
- 先检查上下文，如果没有上下文，直接调用原始函数
- 只对有上下文的请求执行拦截逻辑
- 避免影响无关请求

## 方法论

### 1. 全局拦截器设计原则

#### 原则 1：最小影响范围

**规则**：只拦截需要拦截的请求，其他请求直接放行

**实现**：
```typescript
function intercept(request) {
  // 先检查是否需要拦截
  if (!shouldIntercept(request)) {
    return original(request); // 直接放行
  }
  
  // 执行拦截逻辑
  // ...
}
```

#### 原则 2：上下文优先检查

**规则**：在执行任何拦截逻辑之前，先检查上下文

**实现**：
```typescript
function intercept(request) {
  const ctx = getContext();
  
  // 没有上下文，直接放行
  if (!ctx) {
    return original(request);
  }
  
  // 有上下文，执行拦截逻辑
  // ...
}
```

#### 原则 3：避免副作用

**规则**：拦截逻辑不应该影响无关请求

**错误示例**：
```typescript
// ❌ 错误：全局变量影响所有请求
let lastRequestTime = 0;

function intercept(request) {
  const now = Date.now();
  if (now - lastRequestTime < 1000) {
    await sleep(1000); // 所有请求都被延迟
  }
  lastRequestTime = now;
  
  const ctx = getContext();
  if (!ctx) {
    return original(request); // 已经被延迟了
  }
  // ...
}
```

**正确示例**：
```typescript
// ✅ 正确：只对目标请求使用全局变量
let lastLlmRequestTime = 0;

function intercept(request) {
  const ctx = getContext();
  if (!ctx) {
    return original(request); // 直接放行，不影响全局变量
  }
  
  // 只对 LLM 请求执行延迟
  const now = Date.now();
  if (now - lastLlmRequestTime < 1000) {
    await sleep(1000);
  }
  lastLlmRequestTime = now;
  // ...
}
```

### 2. 调试全局拦截器问题

#### 步骤 1：识别拦截器

**方法**：搜索全局对象的修改

```bash

```

#### 步骤 2：检查拦截逻辑

**检查点**：
- [ ] 是否先检查上下文？
- [ ] 是否有全局变量影响所有请求？
- [ ] 是否有副作用（延迟、重试、日志等）？

#### 步骤 3：验证影响范围

**方法**：添加日志，观察哪些请求被拦截

```typescript
const wrapped: typeof fetch = async (input, init) => {
  console.log('[intercept] request:', input);
  
  const ctx = getContext();
  console.log('[intercept] context:', ctx ? 'yes' : 'no');
  
  if (!ctx) {
    console.log('[intercept] bypass');
    return await original(input, init);
  }
  
  console.log('[intercept] intercept');
  // ...
};
```

#### 步骤 4：修复并验证

**验证方法**：
1. 测试目标请求（应该被拦截）
2. 测试无关请求（不应该被拦截）
3. 测试并发请求（不应该相互影响）

## 关键教训

1. **全局拦截器是双刃剑**：功能强大，但容易产生副作用
2. **上下文优先检查**：在执行任何拦截逻辑之前，先检查上下文
3. **最小影响范围**：只拦截需要拦截的请求，其他请求直接放行
4. **避免全局变量污染**：全局变量只应该影响目标请求
5. **充分测试**：测试目标请求、无关请求、并发请求
