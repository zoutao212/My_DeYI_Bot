# 配置生效但执行失败的调试方法论

> **场景**：用户报告"配置不生效"，但实际上配置已生效，只是后续执行出错。

---

## 问题识别

### 典型症状

- 用户说："配置了 XXX，但没有效果"
- 用户说："XXX 功能不工作"
- 用户说："配置根本不会被保存和使用"

### 常见误判

❌ **错误判断**：配置没有被保存  
✅ **正确判断**：配置已保存，但执行时出错

---

## 调试流程（5 步验证法）

### 步骤 1：验证配置文件

**目标**：确认配置是否写入文件

```powershell
# 检查配置文件
$config = Get-Content "path/to/config.json" -Encoding UTF8 | ConvertFrom-Json

if ($config.xxx.yyy) {
    Write-Host "✅ 配置存在"
} else {
    Write-Host "❌ 配置不存在"
}
```

**结果**：
- ✅ 配置存在 → 进入步骤 2
- ❌ 配置不存在 → 检查保存逻辑

### 步骤 2：验证配置加载

**目标**：确认配置是否被系统加载

```powershell
# 检查日志
Select-String -Path "path/to/log.log" -Pattern "config.*reload|config.*load" -Context 1,1
```

**关键日志**：
- `config hot reload applied (xxx.yyy)`
- `config loaded: xxx.yyy`

**结果**：
- ✅ 配置被加载 → 进入步骤 3
- ❌ 配置未加载 → 检查加载逻辑或重启服务

### 步骤 3：验证配置使用

**目标**：确认功能是否被调用

```powershell
# 检查日志
Select-String -Path "path/to/log.log" -Pattern "功能名称|tool.*start|api.*call" -Context 1,1
```

**关键日志**：
- `embedded run tool start: tool=xxx`
- `xxx: query start`
- `xxx: batch start`

**结果**：
- ✅ 功能被调用 → 进入步骤 4
- ❌ 功能未调用 → 检查触发条件或配置项

### 步骤 4：验证执行结果

**目标**：确认执行是否成功

```powershell
# 检查错误日志
Select-String -Path "path/to/log.log" -Pattern "error|Error|ERROR|failed|Failed|FAILED" -Context 2,2
```

**关键日志**：
- `Unhandled promise rejection: TypeError: fetch failed`
- `xxx failed: 500 Internal Server Error`
- `Command exited with code 1`

**结果**：
- ✅ 执行成功 → 问题可能在其他地方
- ❌ 执行失败 → **这才是真正的问题！**

### 步骤 5：区分问题类型

**配置问题 vs 执行问题**

| 问题类型 | 特征 | 解决方向 |
|---------|------|---------|
| **配置问题** | 配置未保存、未加载、未使用 | 检查配置保存/加载/使用逻辑 |
| **执行问题** | 配置已生效，但执行时出错 | 检查执行逻辑、网络、依赖 |

## 实战案例：Embeddings 配置

### 用户报告

> "Embeddings 配置根本不会被正确保存和使用"

### 调试过程

#### 步骤 1：验证配置文件 ✅

```powershell
$config = Get-Content "C:\Users\zouta\.clawdbot\clawdbot.json" -Encoding UTF8 | ConvertFrom-Json
$config.agents.defaults.memorySearch
# 输出：配置存在
```

#### 步骤 2：验证配置加载 ✅

```
[reload] config hot reload applied (agents.defaults.memorySearch)
```

#### 步骤 3：验证配置使用 ✅

```
[agent/embedded] embedded run tool start: tool=memory_search
[memory] memory embeddings: query start
[memory] memory embeddings: batch start
```

#### 步骤 4：验证执行结果 ❌

```
Unhandled promise rejection: TypeError: fetch failed
at node:internal/deps/undici/undici:13510:13
```

#### 步骤 5：区分问题类型

**结论**：这是**执行问题**，不是配置问题！

- 配置已正确保存 ✅
- 配置已正确加载 ✅
- 配置已正确使用 ✅
- 执行时网络请求失败 ❌

## 关键教训

### 1. 不要被用户的描述误导

用户说"配置不生效"，不代表配置真的没有生效。

**可能的情况**：
- 配置生效了，但执行失败
- 配置生效了，但结果不符合预期
- 配置生效了，但有其他错误

### 2. 必须追踪完整的数据流

**数据流**：保存 → 加载 → 使用 → 执行

**验证点**：
- 保存：配置文件是否包含配置
- 加载：日志是否显示配置被加载
- 使用：日志是否显示功能被调用
- 执行：日志是否显示执行成功

### 3. 区分配置问题和执行问题

**配置问题**：
- 配置未保存
- 配置未加载
- 配置未使用

**执行问题**：
- 网络请求失败
- 依赖缺失
- 权限不足
- 资源不足

### 4. 执行问题需要不同的调试方法

**配置问题**：检查配置保存/加载/使用逻辑  
**执行问题**：检查执行逻辑、网络、依赖、资源

---

## 快速检查清单

调试"配置不生效"问题时，按以下顺序检查：

- [ ] **配置文件**：配置是否写入文件？
- [ ] **配置加载**：日志是否显示配置被加载？
- [ ] **配置使用**：日志是否显示功能被调用？
- [ ] **执行结果**：日志是否显示执行成功？
- [ ] **问题类型**：配置问题 or 执行问题？

**关键**：不要在步骤 1 就下结论，必须完成所有 5 步验证！
