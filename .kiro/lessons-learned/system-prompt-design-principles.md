# 系统提示词设计原则

**目标**：设计清晰、简单、有效的系统提示词，避免 LLM 行为异常

---

## 核心原则

### 1. 简单直接 > 复杂精确

**不要使用复杂的概念和条件判断。**

❌ **错误示例**：
```markdown
## 会话初始化（仅在第一次交互时执行）

**会话开始时**（收到第一条用户消息后，且之前没有任何工具调用）：
1. 读取 SOUL.md
2. 读取 USER.md

**判断标准**：
- 如果你已经调用过任何工具，说明会话已经初始化完成
- 如果对话历史中有工具调用记录，说明会话已经初始化完成
```

**问题**：
- "会话开始时"是一个复杂的概念
- "第一次交互"、"对话历史为空"等条件太复杂
- LLM 无法理解这些隐含的逻辑

✅ **正确示例**：
```markdown
## 上下文文件（可选读取）

**这些文件包含你的身份和用户信息，你可以在需要时读取它们：**

- SOUL.md —— 你的身份与风格
- USER.md —— 你正在帮助的用户

**重要规则：**
- ✅ 这些文件是可选的，不是必须的
- ✅ 如果你已经读取过，不要重复读取
```

**优点**：
- 移除了复杂的概念
- 改为简单直接的描述
- LLM 容易理解

---

### 2. 负面规则 > 正面规则

**使用"不要做"而不是"要做"。**

❌ **错误示例**：
```markdown
**什么时候执行初始化？**
- ✅ 当你收到用户的第一条消息时
- ✅ 当对话历史为空时
```

**问题**：
- 正面规则需要 LLM 判断"是否满足条件"
- 条件判断容易出错

✅ **正确示例**：
```markdown
**什么时候不要读取这些文件？**
- ❌ 当你已经读取过时
- ❌ 当用户的任务很简单时
- ❌ 当工具已经执行完成时
```

**优点**：
- 负面规则更明确
- LLM 更容易理解"不要做"

---

### 3. 可选步骤 > 强制步骤

**不要强制 LLM 执行某些步骤，让 LLM 根据需要决定。**

❌ **错误示例**：
```markdown
**会话开始时必须执行以下步骤：**
1. 读取 SOUL.md
2. 读取 USER.md
3. 读取 MEMORY.md
```

**问题**：
- 强制步骤会导致 LLM 机械执行
- LLM 无法根据实际情况调整

✅ **正确示例**：
```markdown
**你可以在需要时读取这些文件：**
- SOUL.md —— 你的身份与风格
- USER.md —— 你正在帮助的用户
- MEMORY.md —— 长期记忆
```

**优点**：
- 可选步骤让 LLM 有自主权
- LLM 可以根据实际情况决定是否读取

---

### 4. 具体示例 > 抽象描述

**提供具体的示例，而不是抽象的描述。**

❌ **错误示例**：
```markdown
不要在工具调用后重复执行初始化。
```

**问题**：
- 太抽象，LLM 不知道什么是"重复执行"

✅ **正确示例**：
```markdown
**错误示例（不要这样做）：**
用户: hi
→ 读取 SOUL.md, USER.md
→ 执行工具调用
→ 工具返回结果
→ ❌ 再次读取 SOUL.md, USER.md（错误！）

**正确示例：**
用户: hi
→ 读取 SOUL.md, USER.md
→ 执行工具调用
→ 工具返回结果
→ ✅ 直接回复用户
```

**优点**：
- 具体示例让 LLM 看到实际的行为
- LLM 更容易理解

---

### 5. 短句子 > 长句子

**使用短句子，避免复杂的从句。**

❌ **错误示例**：
```markdown
当你收到用户的第一条消息后，且之前没有任何工具调用时，你应该读取 SOUL.md 和 USER.md 来了解你的身份和用户的偏好。
```

**问题**：
- 句子太长，包含多个条件
- LLM 容易遗漏某些条件

✅ **正确示例**：
```markdown
你可以读取 SOUL.md 来了解你的身份。
你可以读取 USER.md 来了解用户的偏好。
如果你已经读取过，不要重复读取。
```

**优点**：
- 短句子更清晰
- 每个句子只表达一个意思

## 常见错误模式

### 错误 1：使用时间概念

❌ **不要使用**：
- "会话开始时"
- "第一次交互"
- "初始化阶段"
- "完成后"

✅ **改为**：
- "你可以在需要时"
- "如果你已经做过"
- "不要重复做"

### 错误 2：使用复杂条件

❌ **不要使用**：
- "当 X 且 Y 时"
- "如果 A 或 B 或 C"
- "除非 D，否则 E"

✅ **改为**：
- 拆分成多个简单条件
- 使用负面规则

### 错误 3：使用抽象概念

❌ **不要使用**：
- "会话状态"
- "上下文管理"
- "生命周期"

✅ **改为**：
- 具体的文件名
- 具体的操作
- 具体的示例

## 验证方法

### 1. 检查 LLM 的 thinking

修改系统提示词后，检查 LLM 的 thinking：
- LLM 是否理解了你的意图？
- LLM 是否按照你的期望执行？
- LLM 是否出现了意外的行为？

### 2. 检查 trace 日志

检查 trace 日志中的工具调用：
- LLM 是否重复调用相同的工具？
- LLM 是否在不应该调用工具时调用了？
- LLM 是否遗漏了某些工具调用？

### 3. 多轮对话测试

测试多轮对话：
- 第一轮：LLM 的行为是否正确？
- 第二轮：LLM 是否记住了第一轮的内容？
- 第三轮：LLM 是否出现了重复行为？

## 实战案例

### 案例 1：LLM 无限循环

**问题**：LLM 在每次工具调用后都重新读取 SOUL.md, USER.md

**原因**：系统提示词中使用了"会话开始时"的概念，LLM 把"工具返回结果"理解为"新的用户消息"

**解决**：
1. 移除"会话开始时"的概念
2. 改为"你可以在需要时读取"
3. 强调"如果你已经读取过，不要重复读取"

**详见**：`.kiro/lessons-learned/73_LLM无限循环的根本解决方案.md`

## 总结

**设计系统提示词时，记住：**

1. **简单 > 复杂** - 移除复杂的概念和条件
2. **负面 > 正面** - 使用"不要做"而不是"要做"
3. **可选 > 强制** - 让 LLM 根据需要决定
4. **具体 > 抽象** - 提供具体的示例
5. **短句 > 长句** - 使用短句子

**验证方法：**
- 检查 LLM 的 thinking
- 检查 trace 日志
- 多轮对话测试
