# 历史消息格式处理策略

**问题类型**：多轮对话中，历史消息的格式导致 API 报错

**触发场景**：
- 中转 API 格式不完全兼容
- LLM 对历史消息格式敏感
- 多轮对话中出现格式错误

---

## 核心原则

**最后一轮对话 vs 历史对话**

| 类型 | 处理策略 | 原因 |
|------|---------|------|
| 最后一轮对话 | 保留完整格式 | LLM 需要完整信息继续执行 |
| 历史对话 | 简化格式 | 避免格式冲突，减少 token |

---

## 实战案例

### 案例 1：yinli provider 的 thought_signature

**问题**：
- LLM 返回的响应中包含 `thoughtSignature`
- 历史消息中的 `thoughtSignature` 导致 API 报错

**解决方案**：
- 最后一轮：保留 `thoughtSignature`（如果 API 支持）
- 历史对话：移除 `thoughtSignature`

**详见**：`.kiro/lessons-learned/49_Provider特定API兼容性问题处理.md`

### 案例 2：vectorengine 的 tool_calls

**问题**：
- 最后一轮：需要 Gemini 格式（`functionCall` + `functionResponse`）
- 历史对话：OpenAI 格式导致 API 报错

**解决方案**：
- 最后一轮：转换为 Gemini 格式
- 历史对话：内联到 `content`（纯文本）


**详见**：`.kiro/lessons-learned/77_OpenAI兼容API格式差异调试方法论.md`

## 通用处理策略

### 策略 A：内联到 content（推荐）

**适用场景**：历史对话的工具调用和结果

**实现**：
```typescript
// 历史 tool_calls：内联到 content
const toolCallsText = msg.tool_calls
  .map((tc: any) => `[调用工具: ${tc.function?.name || "unknown"}]`)
  .join("\n");

msg.content = (msg.content || "") + `\n\n${toolCallsText}`;
delete msg.tool_calls;

// 历史 tool 结果：内联到前一条 assistant 消息
const toolResult = `\n\n[工具执行结果]\n${msg.content || "(no output)"}`;
assistantMsg.content += toolResult;
msg._shouldDelete = true;
```

**优点**：
- 完全避免格式冲突
- 减少消息数量
- LLM 仍然能看到历史信息

**缺点**：
- 丢失结构化信息
- 无法精确重现历史对话

### 策略 B：移除特殊字段

**适用场景**：API 不支持某些扩展字段

**实现**：
```typescript
// 移除不支持的字段
if (Array.isArray(msg.content)) {
  for (const block of msg.content) {
    if (block && typeof block === "object") {
      delete block.thoughtSignature;
      delete block.thought_signature;
    }
  }
}
```

**优点**：
- 保留结构化信息
- 最小化修改

**缺点**：
- 可能丢失重要信息
- 需要知道哪些字段不支持

### 策略 C：转换为统一格式

**适用场景**：API 支持多种格式，但需要统一

**实现**：
```typescript
// 统一转换为 OpenAI 格式
if (msg.role === "function") {
  msg.role = "tool";
  msg.tool_call_id = extractToolCallId(msg);
  msg.content = extractContent(msg);
}
```

**优点**：
- 保留完整信息
- 格式统一

**缺点**：
- 实现复杂
- 需要双向转换

## 选择策略的决策树

```
历史消息格式问题
  ├─ API 完全不支持某种格式？
  │   └─ 是 → 策略 A（内联到 content）
  │
  ├─ API 不支持某些扩展字段？
  │   └─ 是 → 策略 B（移除特殊字段）
  │
  └─ API 支持多种格式？
      └─ 是 → 策略 C（转换为统一格式）
```

## 实现检查清单

- [ ] **识别最后一轮对话**：找到最后一对 `assistant + tool` 消息
- [ ] **保留最后一轮格式**：确保 LLM 能继续执行
- [ ] **简化历史对话**：选择合适的策略（A/B/C）
- [ ] **验证格式转换**：检查转换后的 payload
- [ ] **测试多轮对话**：至少测试 3 轮对话

## 常见错误

### 错误 1：所有消息都简化

```typescript
// ❌ 错误：最后一轮也简化了
for (const msg of messages) {
  if (msg.tool_calls) {
    msg.content = "[调用工具]";
    delete msg.tool_calls;
  }
}
```

**问题**：LLM 无法继续执行工具调用

### 错误 2：没有识别"最后一轮"

```typescript
// ❌ 错误：从后往前找，但没有配对
let lastToolIndex = -1;
for (let i = messages.length - 1; i >= 0; i--) {
  if (messages[i].role === "tool") {
    lastToolIndex = i;
    break;
  }
}
```

**问题**：可能找到的不是最后一对

**正确做法**：
```typescript
// ✅ 正确：找到所有配对，取最后一对
const pairs = [];
for (let i = 0; i < messages.length; i++) {
  if (messages[i].role === "assistant" && messages[i].tool_calls) {
    for (let j = i + 1; j < messages.length; j++) {
      if (messages[j].role === "tool") {
        pairs.push({ assistantIndex: i, toolIndex: j });
        break;
      }
    }
  }
}
const lastPair = pairs[pairs.length - 1];
```

## 关键教训

1. **最后一轮 ≠ 最后一条消息**
   - 最后一轮是"最后一对 assistant + tool"
   - 不是"最后一条 tool 消息"

2. **简化历史对话是必要的**
   - 避免格式冲突
   - 减少 token 消耗
   - 提高兼容性

3. **内联是最安全的策略**
   - 完全避免格式问题
   - 适用于大多数场景
   - 实现简单

4. **验证要测试多轮对话**
   - 第一轮可能成功（没有历史消息）
   - 第二轮可能失败（有历史消息）
   - 至少测试 3 轮
