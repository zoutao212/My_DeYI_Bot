# 设计文档：栗娜（Lina）人格化系统

## 概述

栗娜（Lina）是 Butler Agent 的人格化定义。她不是独立的系统，而是 Butler 的一个"人格"或"角色"。所有能力都复用 Butler 现有的功能（TaskDelegator、MemoryService、TaskBoard），只添加独有的提醒管理功能。

### 核心定位

**Lina = Butler 的人格化定义**

- **人格定义**：在配置文件中定义角色设定、对话风格、工作原则
- **能力实现**：完全复用 Butler 的能力（任务管理、记忆管理、任务委托）
- **独有功能**：提醒管理器（ReminderManager）

### 设计目标

1. **人格化**：Lina 有独特的性格和对话风格
2. **复用能力**：不重复实现 Butler 已有的功能
3. **易于扩展**：可以轻松添加更多角色（如丽丝、小艾等）
4. **配置驱动**：通过配置文件定义角色，无需修改代码

2. **Lina 人格层**：

   - **提醒管理器**：Lina 的独有功能
3. **Butler 能力层**：复用 Butler 的所有能力
   - **TaskDelegator**：任务管理和委托
   - **MemoryService**：记忆存储和检索
   - **TaskBoard**：任务看板和进度跟踪



## 组件和接口

### 1. 角色配置系统

负责加载和管理角色定义。

#### 配置文件结构

```
clawd/characters/lina/
├── profile.md       # 角色设定（性格、能力、对话风格）
└── config.json      # 角色配置（功能开关、参数）
```

#### 配置接口

```typescript
interface CharacterConfig {
  name: string;                    // 角色名称（如 "lina"）
  displayName: string;             // 显示名称（如 "栗娜"）
  version: string;                 // 版本号
  enabled: boolean;                // 是否启用
  features: {                      // 功能开关
    reminders: boolean;            // 提醒功能
    taskManagement: boolean;       // 任务管理
    memoryManagement: boolean;     // 记忆管理
    taskDelegation: boolean;       // 任务委托
  };
  systemPrompt: {                  // System Prompt 配置
    role: string;                  // 角色定位
    personality: string[];         // 性格特征
    addressUser: string;           // 称呼用户
    addressSelf: string;           // 自称
  };
  reminders?: {                    // 提醒配置（可选）
    enabled: boolean;              // 是否启用
    checkInterval: number;         // 检查间隔（毫秒）
    advanceNotice: number;         // 提前通知时间（分钟）
  };
}
```

### 2. 提醒管理器（ReminderManager）

Lina 的独有功能，负责管理用户的提醒事项。

#### 接口

```typescript
interface ReminderManager {
  // 创建提醒
  createReminder(reminder: Reminder): Promise<string>;
  
  // 获取提醒列表
  getReminders(filter: ReminderFilter): Promise<Reminder[]>;
  
  // 更新提醒
  updateReminder(reminderId: string, updates: Partial<Reminder>): Promise<void>;
  
  // 删除提醒
  deleteReminder(reminderId: string): Promise<void>;
  
  // 暂停提醒
  pauseReminder(reminderId: string, until: Date): Promise<void>;
  
  // 恢复提醒
  resumeReminder(reminderId: string): Promise<void>;
  
  // 检查到期提醒
  checkDueReminders(): Promise<Reminder[]>;
}
```

#### 实现要点

- 使用定时器或调度器检查到期提醒
- 支持一次性提醒和重复提醒
- 支持提前提醒（如提前 10 分钟）
- 支持提醒的暂停和恢复
- 数据持久化到 `clawd/characters/lina/reminders.json`

### 3. Butler 能力复用

Lina 直接使用 Butler 的能力，不重复实现：

#### 任务管理（TaskDelegator）

```typescript
// Lina 直接调用 Butler 的 TaskDelegator
const taskId = await butler.taskDelegator.delegateTask({
  type: "code",
  description: "实现用户登录功能",
  parameters: { ... }

```

#### 记忆管理（MemoryService）

```typescript
const memories = await butler.memoryService.retrieveMemories({
  query: "上次和张三讨论了什么",
  limit: 10
});
```

#### 进度跟踪（TaskBoard）

```typescript
// Lina 直接调用 Butler 的 TaskBoard
const progress = await butler.taskBoard.getProgress(taskId);
```

```typescript
interface Reminder {
  id: string;                    // 提醒 ID
```

## 正确性属性

属性是系统应该在所有有效执行中保持为真的特征或行为。属性是人类可读规范和机器可验证正确性保证之间的桥梁。

### 任务管理属性

**属性 1：任务创建往返一致性**
*对于任何*有效的任务信息，创建任务后应该能够从存储中读取相同的任务信息
**验证需求：1.1, 8.1**

**属性 2：任务状态更新持久化**
*对于任何*任务，更新状态后应该能够从存储中读取更新后的状态
**验证需求：1.3, 4.2**

**属性 3：任务删除一致性**
*对于任何*任务，删除后应该无法从存储中读取该任务
**验证需求：1.7, 8.4**

**属性 4：任务更新往返一致性**
*对于任何*任务和任何有效的更新内容，更新后应该能够从存储中读取更新后的信息
**验证需求：1.6, 8.5**

**属性 5：任务查询过滤正确性**
*对于任何*任务列表和任何过滤条件，查询结果应该只包含满足过滤条件的任务
**验证需求：1.2, 1.5**

### 记忆管理属性

**属性 6：记忆归档往返一致性**
*对于任何*对话内容，归档后应该能够从存储中检索相同的内容
**验证需求：2.2, 8.2**

**属性 7：记忆检索相关性**
*对于任何*查询，检索结果应该只包含与查询相关的记忆
**验证需求：2.1**

**属性 8：记忆时间范围过滤正确性**
*对于任何*时间范围，检索结果应该只包含该时间范围内的记忆
**验证需求：2.4**

**属性 9：记忆删除一致性**
*对于任何*记忆，删除后应该无法从存储中检索该记忆
**验证需求：2.6**

**属性 10：记忆总结完整性**
*对于任何*话题，总结应该包含所有相关记忆的关键信息
**验证需求：2.3**

### 任务委托属性

**属性 11：任务委托状态一致性**
*对于任何*技术任务，委托后应该能够查询到任务状态，并且状态应该从 pending 开始
**验证需求：3.1, 3.2**

**属性 12：任务取消有效性**
*对于任何*正在执行的任务，取消后任务状态应该变为 cancelled
**验证需求：3.6**

**属性 13：任务完成通知及时性**
*对于任何*完成的任务，系统应该在任务完成后立即通知用户
**验证需求：3.5, 4.5**

**属性 14：任务失败错误报告**
*对于任何*失败的任务，系统应该记录失败原因并通知用户
**验证需求：3.3, 9.1**

### 进度跟踪属性

**属性 15：任务时间记录完整性**
*对于任何*任务，应该记录开始时间，完成时应该记录完成时间
**验证需求：4.1, 4.5**

**属性 16：任务进度查询一致性**
*对于任何*任务，查询进度应该返回当前的实际进度
**验证需求：3.4, 4.3**

**属性 17：长时间运行任务检测**
*对于任何*执行时间超过阈值的任务，系统应该检测并主动询问用户
**验证需求：4.4**

**属性 18：历史任务查询完整性**
*对于任何*时间范围，历史任务查询应该返回该范围内的所有任务
**验证需求：4.6**

### 智能提醒属性

**属性 19：定时提醒准确性**
*对于任何*提醒，系统应该在指定时间（误差 ±1 分钟内）触发提醒
**验证需求：1.4, 5.1**

**属性 20：提前提醒正确性**

## 错误处理

### 错误分类

系统将错误分为以下几类：

1. **用户输入错误**：用户输入无效或不完整
2. **系统组件错误**：系统组件不可用或返回错误
3. **存储错误**：数据存储或读取失败
4. **网络错误**：网络连接失败或超时
5. **未知错误**：无法分类的错误

### 错误处理策略

#### 用户输入错误

- 友好地提示用户正确的输入格式
- 提供示例帮助用户理解
- 不记录为系统错误

#### 系统组件错误

- 记录错误日志
- 告知用户哪个组件不可用
- 提供降级方案（如果可能）
- 提供重试选项

#### 存储错误

- 记录错误日志
- 告知用户存储失败
- 提供重试选项
- 如果多次重试失败，建议用户检查系统状态

#### 网络错误

- 记录错误日志
- 告知用户网络问题
- 自动重试（最多 3 次）
- 如果重试失败，建议用户检查网络连接

#### 未知错误

- 记录详细的错误日志（包括堆栈跟踪）
- 告知用户发生了未知错误
- 提供错误 ID 供用户报告
- 尝试恢复到安全状态

### 错误恢复

系统应该能够从错误中恢复，不应该因为单个错误而完全崩溃：

- 使用 try-catch 包裹所有可能出错的操作
- 提供默认值或降级方案
- 保持系统状态的一致性
- 记录错误但继续运行

### 错误通知

错误通知应该：

- 用友好的语言描述错误
- 避免技术术语（除非用户是技术人员）
- 提供可能的解决方案
- 提供重试或取消选项

## 测试策略

### 双重测试方法

系统将使用单元测试和属性测试相结合的方法：

- **单元测试**：验证特定示例、边界情况和错误条件
- **属性测试**：验证所有输入的通用属性

两者是互补的，都是必需的，以实现全面覆盖。

### 单元测试

单元测试应该关注：

- **特定示例**：演示正确行为的具体示例
- **边界情况**：空输入、极大值、极小值等
- **错误条件**：无效输入、系统组件不可用等
- **集成点**：组件之间的交互

单元测试不应该过多，应该专注于关键场景。

### 属性测试

属性测试应该关注：

- **通用属性**：对所有输入都成立的规则
- **往返属性**：创建-读取-更新-删除的一致性
- **不变量**：系统状态的不变性
- **幂等性**：重复操作的结果一致性

每个属性测试应该：

- 运行至少 100 次迭代（由于随机化）
- 引用设计文档中的属性
- 使用标签格式：**Feature: lina-daily-management, Property {number}: {property_text}**

### 测试覆盖

测试应该覆盖：

- 所有核心功能
- 所有错误处理路径
- 所有系统集成点
- 所有数据持久化操作
- 所有性能关键路径

### 测试工具

- **单元测试框架**：Vitest
