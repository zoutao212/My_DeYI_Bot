# 需求文档：多层 Agent 系统架构

## 引言

当前 Clawdbot 的 Agent 架构存在一个关键问题：**角色扮演型 Agent（如栗娜）只会"描述"操作，不会实际调用工具执行操作**。这导致用户体验不佳，用户期望的操作没有真正执行。

本需求文档定义了一个**多层 Agent 系统架构**，通过分层设计，将角色扮演、任务理解、任务调度、工具执行等职责清晰分离，确保每一层专注于自己的职责，从而解决角色扮演与工具调用冲突的问题。

## 术语表

- **虚拟世界层（Virtual World Layer）**：人格化的侍女角色（丽丝、艾莉等），完全不知道底层技术细节，生活在虚拟文字世界中
- **管家层（Butler Layer）**：栗娜（Agent）- 行政总监/管家，理解用户意图，分解任务，调度底层系统
- **任务调度层（Task Orchestration Layer）**：任务分解器、任务执行器、进度跟踪器，智能调度，自动分解复杂任务
- **执行层（Execution Layer）**：工具调用执行器，只负责执行具体的工具调用
- **任务委托（Task Delegation）**：管家层将任务委托给任务调度层的机制
- **工具调用（Tool Call）**：执行层调用具体的工具（read, write, edit, exec, grep 等）
- **技能（Skill）**：封装的可复用功能模块

## 需求

### 需求 1：虚拟世界层设计

**用户故事**：作为用户，我希望与人格化的侍女角色（丽丝、艾莉等）互动，享受沉浸式的角色扮演体验，而不需要她们了解任何技术细节。

#### 验收标准

1. WHEN 用户与虚拟世界层的角色互动 THEN 系统应该提供纯粹的角色扮演体验，不涉及任何工具调用
2. WHEN 虚拟世界层的角色收到用户消息 THEN 系统应该只返回角色扮演的对话内容，不包含任何技术操作描述
3. THE 虚拟世界层 SHALL 完全不知道底层技术细节（工具、API、文件系统等）
4. THE 虚拟世界层 SHALL 只能进行对话、角色扮演、情感交互
5. WHEN 用户要求虚拟世界层执行技术操作 THEN 系统应该将请求转发给管家层处理

### 需求 2：管家层设计

**用户故事**：作为用户，我希望栗娜（管家）能够理解我的意图，并将任务委托给底层系统执行，而不是只"描述"操作。

#### 验收标准

1. WHEN 用户向栗娜发出任务请求 THEN 栗娜应该理解用户意图并分解任务
2. WHEN 栗娜分解任务后 THEN 栗娜应该将任务委托给任务调度层执行
3. WHEN 任务调度层返回执行结果 THEN 栗娜应该将结果以友好的方式反馈给用户
4. THE 管家层 SHALL 不直接执行工具调用，而是委托给任务调度层
5. THE 管家层 SHALL 可以调用独立的系统任务或技能（如记忆检索、知识查询等）
6. WHEN 栗娜收到复杂任务 THEN 栗娜应该将任务分解为多个子任务并逐一委托执行

### 需求 3：任务调度层设计

**用户故事**：作为系统开发者，我希望有一个智能的任务调度层，能够自动分解复杂任务，调度执行层完成任务，并跟踪进度。

#### 验收标准

1. WHEN 任务调度层接收到管家层的任务委托 THEN 系统应该自动判断任务是否需要分解
2. WHEN 任务需要分解 THEN 系统应该将任务分解为多个子任务
3. WHEN 任务分解完成 THEN 系统应该调度执行层完成每个子任务
4. WHEN 子任务执行失败 THEN 系统应该自动处理失败和重试
5. WHEN 所有子任务完成 THEN 系统应该将结果汇总并返回给管家层
6. THE 任务调度层 SHALL 深度融合现有的任务分解系统（`.kiro/specs/task-decomposition-and-tracking/`）
7. THE 任务调度层 SHALL 复用 TaskBoard、Orchestrator、Executor 等组件
8. THE 任务调度层 SHALL 支持串行和并行执行策略

### 需求 4：执行层设计

**用户故事**：作为系统开发者，我希望有一个专门的执行层，只负责执行具体的工具调用，不做任何决策。

#### 验收标准

1. WHEN 执行层接收到工具调用请求 THEN 系统应该执行对应的工具调用
2. WHEN 工具调用成功 THEN 系统应该返回执行结果
3. WHEN 工具调用失败 THEN 系统应该返回错误信息
4. THE 执行层 SHALL 只负责执行工具调用，不做任何决策
5. THE 执行层 SHALL 支持所有现有的工具（read, write, edit, exec, grep 等）
6. THE 执行层 SHALL 支持技能（Skills）调用

### 需求 5：任务委托机制

**用户故事**：作为系统开发者，我希望管家层可以通过标准的任务委托机制将任务委托给任务调度层，确保任务能够被正确执行。

#### 验收标准

1. WHEN 管家层需要委托任务 THEN 系统应该提供标准的任务委托接口
2. WHEN 任务委托接口被调用 THEN 系统应该将任务传递给任务调度层
3. WHEN 任务调度层完成任务 THEN 系统应该将结果返回给管家层
4. THE 任务委托机制 SHALL 支持同步和异步两种模式
5. THE 任务委托机制 SHALL 支持进度回调
6. THE 任务委托机制 SHALL 支持取消任务

### 需求 6：层次通信协议

**用户故事**：作为系统开发者，我希望各层之间有清晰的通信协议，确保数据传递的准确性和一致性。

#### 验收标准

1. WHEN 层与层之间通信 THEN 系统应该使用标准的通信协议
2. THE 通信协议 SHALL 定义清晰的消息格式
3. THE 通信协议 SHALL 支持错误处理和重试
4. THE 通信协议 SHALL 支持进度通知
5. THE 通信协议 SHALL 支持取消操作

### 需求 7：角色扮演与工具调用分离

**用户故事**：作为用户，我希望角色扮演和工具调用完全分离，角色扮演时不涉及技术细节，工具调用时不涉及角色扮演。

#### 验收标准

1. WHEN 虚拟世界层进行角色扮演 THEN 系统不应该涉及任何工具调用
2. WHEN 执行层执行工具调用 THEN 系统不应该涉及任何角色扮演
3. WHEN 管家层委托任务 THEN 系统应该清晰区分角色扮演和工具调用
4. THE 系统 SHALL 确保角色扮演和工具调用在不同的层次进行
5. THE 系统 SHALL 确保角色扮演不会干扰工具调用的执行

### 需求 8：智能调度

**用户故事**：作为系统开发者，我希望任务调度层能够智能地判断任务是否需要分解，自动选择合适的执行策略。

#### 验收标准

1. WHEN 任务调度层接收到简单任务 THEN 系统应该直接调用执行层完成任务
2. WHEN 任务调度层接收到复杂任务 THEN 系统应该自动分解任务
3. WHEN 任务可以并行执行 THEN 系统应该选择并行执行策略
4. WHEN 任务必须串行执行 THEN 系统应该选择串行执行策略
5. WHEN 任务执行失败 THEN 系统应该自动处理失败和重试

### 需求 9：兼容性

**用户故事**：作为系统开发者，我希望新的多层架构能够兼容现有代码，不破坏现有功能。

#### 验收标准

1. THE 新架构 SHALL 兼容现有的 Agent 系统
2. THE 新架构 SHALL 复用现有的任务分解系统（`src/agents/task-board/`）
3. THE 新架构 SHALL 复用现有的 System Prompt 构建器（`src/agents/system-prompt.ts`）
4. THE 新架构 SHALL 复用现有的 Session Manager（Pi Agent Core）
5. WHEN 新架构部署后 THEN 现有功能应该继续正常工作

### 需求 10：性能

**用户故事**：作为用户，我希望新的多层架构不会显著增加系统延迟，并且通过 System Prompt 分层优化能够显著降低 token 消耗。

#### 验收标准

1. WHEN 新架构处理任务 THEN 系统延迟不应该显著增加（增加不超过 20%）
2. WHEN 新架构使用 System Prompt 分层优化 THEN token 消耗应该显著降低（预期降低 30-50%）
3. THE 新架构 SHALL 优化层次通信，减少不必要的数据传递
4. THE 新架构 SHALL 优化任务分解，避免过度分解
5. THE 新架构 SHALL 通过 System Prompt 分层，避免在角色扮演层携带工具使用提示词

### 需求 11：可观测性

**用户故事**：作为系统开发者，我希望能够观测到任务在各层之间的流转过程，便于调试和优化。

#### 验收标准

1. WHEN 任务在各层之间流转 THEN 系统应该记录详细的日志
2. THE 日志 SHALL 包含任务 ID、层次、时间戳、状态等信息
3. THE 日志 SHALL 支持按任务 ID 查询完整的流转过程
4. THE 系统 SHALL 提供可视化的任务流转图
5. THE 系统 SHALL 提供性能监控指标（延迟、token 消耗等）

### 需求 12：渐进式迁移

**用户故事**：作为系统开发者,我希望能够渐进式地迁移到新架构，而不是推倒重来。

#### 验收标准

1. THE 新架构 SHALL 支持渐进式迁移
2. WHEN 迁移过程中 THEN 旧架构和新架构应该能够共存
3. WHEN 迁移完成后 THEN 旧架构应该能够被安全移除
4. THE 迁移过程 SHALL 不影响现有功能的正常使用
5. THE 迁移过程 SHALL 提供详细的迁移指南

### 需求 13：System Prompt 分层优化

**用户故事**：作为系统开发者，我希望不同层次的 Agent 使用不同的 System Prompt，避免不必要的 token 消耗。

#### 验收标准

1. WHEN 虚拟世界层进行对话 THEN 系统不应该在 System Prompt 中包含工具使用提示词
2. WHEN 虚拟世界层进行对话 THEN 系统不应该在 System Prompt 中包含底层系统提示词
3. WHEN 管家层处理任务 THEN 系统应该在 System Prompt 中包含任务委托相关的提示词
4. WHEN 执行层执行工具调用 THEN 系统应该在 System Prompt 中包含工具使用提示词
5. THE System Prompt 分层 SHALL 大幅节省 token 消耗（预期节省 30-50%）
6. THE System Prompt 分层 SHALL 提高角色扮演内容的准确性

### 需求 14：对话前后的任务调度

**用户故事**：作为系统开发者，我希望在虚拟世界层对话前后，管家层和任务调度层能够自动执行记忆填充、总结归档等任务。

#### 验收标准

1. WHEN 虚拟世界层开始对话前 THEN 系统应该自动执行记忆填充任务
2. WHEN 记忆填充任务完成 THEN 系统应该将相关记忆注入到虚拟世界层的上下文中
3. WHEN 虚拟世界层对话结束后 THEN 系统应该自动执行总结归档任务
4. WHEN 总结归档任务完成 THEN 系统应该将对话内容归档到长期记忆中
5. THE 对话前后的任务调度 SHALL 对用户透明（用户无感知）
6. THE 对话前后的任务调度 SHALL 支持自定义任务（如情感分析、关键词提取等）
7. WHEN 对话前后的任务执行失败 THEN 系统应该记录错误但不影响对话流程

## 非功能性需求

### 可维护性

- 代码应该清晰、简洁、易于理解
- 每一层的职责应该单一、明确
- 接口应该稳定、易于扩展

### 可测试性

- 每一层应该能够独立测试
- 应该提供完整的单元测试和集成测试
- 应该提供端到端测试

### 可扩展性

- 应该支持添加新的层次
- 应该支持添加新的工具
- 应该支持添加新的技能

## 约束条件

1. **必须兼容现有代码**：不破坏现有的 Agent 系统
2. **必须复用现有组件**：任务分解系统、System Prompt 构建器、Session Manager
3. **必须保持性能**：不增加显著的延迟和 token 消耗
4. **必须使用 TypeScript**：保持代码风格一致
5. **必须遵循现有的代码规范**：Oxlint、Oxfmt
