# 需求文档：批量任务执行优化

## 概述

优化递归任务系统的执行效率，通过批量创建和批量执行任务，显著降低 LLM 请求成本（节省 26-60% tokens）和提高执行效率（减少 40-75% 请求次数）。

## 背景

### 当前问题

1. **系统提示词开销大**：每次 LLM 请求都携带完整的系统提示词（5000+ tokens）
2. **任务粒度过细**：任务树分解非常细腻，导致大量小任务
3. **请求次数过多**：每个小任务单独请求 LLM
4. **LLM 能力未充分利用**：LLM 单次可输出 3000-6000 字，但每次只完成一个小任务

### 优化目标

1. 批量创建任务（一次性创建多个相关任务）
2. 批量执行任务（一次 LLM 请求完成多个小任务）
3. 智能分组（根据任务特征自动分组）
4. 保持灵活性（支持单任务和批量两种模式）

## 功能需求

### FR1: batch_enqueue_tasks 工具

**优先级**：P0（核心功能）

**描述**：提供批量创建任务的工具，允许 LLM 一次性创建多个相关任务

**验收标准**：
- FR1.1: 工具接受任务列表作为参数
- FR1.2: 每个任务包含 prompt、summary 和可选的元数据
- FR1.3: 支持 batchMode 参数（auto/force/disable）
- FR1.4: 返回创建的任务 ID 列表
- FR1.5: 自动添加元数据（estimatedTokens、canBatch）

### FR2: TaskGrouper（任务分组器）

**优先级**：P0（核心功能）

**描述**：智能分组任务，决定哪些任务可以批量执行

**验收标准**：
- FR2.1: 根据相似度分组（相似的任务优先批量执行）
- FR2.2: 根据大小分组（小任务优先批量执行）
- FR2.3: 根据依赖关系分组（无依赖关系的任务可以批量执行）
- FR2.4: 单个批次最多 3-5 个任务
- FR2.5: 单个批次预估输出 < 6000 tokens
- FR2.6: 批次内任务必须无依赖关系

### FR3: BatchExecutor（批量执行器）

**优先级**：P0（核心功能）

**描述**：将多个任务合并为一次 LLM 请求

**验收标准**：
- FR3.1: 合并多个任务的 prompt
- FR3.2: 添加输出格式要求（使用分隔符）
- FR3.3: 单次 LLM 请求
- FR3.4: 解析输出并拆分到各个任务
- FR3.5: 保存结果到各个任务
- FR3.6: 如果拆分失败，回退到单任务执行

### FR4: 智能预估

**优先级**：P1（重要功能）

**描述**：智能预估任务输出 tokens，优化分组策略

**验收标准**：
- FR4.1: 从 prompt 中提取字数要求
- FR4.2: 从 summary 中提取字数要求
- FR4.3: 根据复杂度估算
- FR4.4: 预估误差 < 30%

### FR5: 动态调整

**优先级**：P2（增强功能）

**描述**：根据实际执行情况动态调整分组策略

**验收标准**：
- FR5.1: 记录执行历史（最近 20 次）
- FR5.2: 计算平均误差
- FR5.3: 根据误差调整批次大小
- FR5.4: 如果误差 > 30%，减小批次大小
- FR5.5: 如果误差 < 10%，增大批次大小

## 非功能需求

### NFR1: 性能

- NFR1.1: 批量执行应节省 26-60% tokens
- NFR1.2: 批量执行应减少 40-75% 请求次数
- NFR1.3: 分组延迟 < 100ms
- NFR1.4: 输出拆分延迟 < 200ms

### NFR2: 可靠性

- NFR2.1: 输出拆分成功率 > 95%
- NFR2.2: 如果拆分失败，自动回退到单任务执行
- NFR2.3: 批次执行失败不影响其他批次

### NFR3: 可维护性

- NFR3.1: 代码模块化，易于扩展
- NFR3.2: 完整的单元测试覆盖
- NFR3.3: 详细的日志记录

### NFR4: 兼容性

- NFR4.1: 向后兼容现有的 enqueue_task 工具
- NFR4.2: 支持单任务和批量两种模式
- NFR4.3: 不破坏现有的任务树结构

## 用户故事

### US1: 生成大量内容

**作为**：用户  
**我想要**：生成 10000 字的文章  
**以便**：快速获得完整的文章

**验收标准**：
- LLM 自动将任务分解为 5 个子任务
- 系统自动分组为 2-3 个批次
- 2-3 次 LLM 请求完成所有任务
- 节省 26% tokens

### US2: 处理大量数据

**作为**：用户  
**我想要**：总结 100 个文件  
**以便**：快速了解所有文件的内容

**验收标准**：
- LLM 自动将任务分解为 100 个子任务
- 系统自动分组为 25 个批次
- 25 次 LLM 请求完成所有任务
- 节省 60% tokens

### US3: 执行多个相似任务

**作为**：用户  
**我想要**：生成 20 个章节  
**以便**：快速完成整本书的创作

**验收标准**：
- LLM 自动将任务分解为 20 个子任务
- 系统自动分组为 7 个批次
- 7 次 LLM 请求完成所有任务
- 节省 50% tokens

## 约束条件

### C1: 技术约束

- C1.1: 必须基于现有的任务树系统
- C1.2: 必须兼容现有的 enqueue_task 工具
- C1.3: 必须支持所有 LLM 提供商（Gemini、Claude、OpenAI）

### C2: 业务约束

- C2.1: 不能破坏现有功能
- C2.2: 必须保持向后兼容
- C2.3: 必须提供降级方案（如果批量执行失败）

### C3: 安全约束

- C3.1: 单个批次最多 5 个任务
- C3.2: 单个批次最多 8000 tokens
- C3.3: 批次执行超时时间 2 分钟

## 风险与缓解

### R1: 输出拆分失败

**风险**：LLM 可能不按照格式输出，导致拆分失败

**缓解**：
- 提供清晰的输出格式要求
- 使用明显的分隔符
- 如果拆分失败，回退到单任务执行

### R2: 批量执行质量下降

**风险**：批量执行可能导致单个任务的质量下降

**缓解**：
- 限制批次大小（最多 5 个任务）
- 限制批次 tokens（最多 8000 tokens）
- 提供 batchMode="disable" 选项

### R3: 兼容性问题

**风险**：新功能可能破坏现有功能

**缓解**：
- 保持向后兼容
- 提供降级方案
- 充分测试

## 成功指标

### KPI1: 成本节省

- 目标：节省 26-60% tokens
- 测量：对比批量执行和单任务执行的 tokens 消耗

### KPI2: 效率提升

- 目标：减少 40-75% 请求次数
- 测量：对比批量执行和单任务执行的请求次数

### KPI3: 质量保持

- 目标：批量执行的质量与单任务执行相当
- 测量：用户满意度调查

### KPI4: 可靠性

- 目标：输出拆分成功率 > 95%
- 测量：拆分成功次数 / 总次数
