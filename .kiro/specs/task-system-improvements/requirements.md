# 递归任务系统改进需求文档

**日期**: 2026-02-06  
**状态**: 草稿

---

## 1. 概述

本文档定义了对现有递归任务系统的三个关键改进需求,旨在提升系统的精确性、用户体验和智能化水平。

---

## 2. 背景

当前递归任务系统已经实现了基本的任务分解、并行执行和交付报告功能,但在实际使用中发现了以下问题:

1. **并行执行匹配不精确**: 使用 `prompt` 字符串匹配队列项与任务树,当两个子任务的 prompt 相同时会产生误匹配
2. **交付报告格式单一**: 只支持纯文本 Markdown 格式,缺乏富文本展示能力
3. **深度控制过于简单**: 仅基于子任务数量的简单分段函数,未考虑任务类型、复杂度等因素

---

## 3. 用户故事

### 3.1 精确的并行任务匹配

**作为** 系统开发者  
**我希望** 并行执行时能精确匹配队列项与任务树中的任务  
**以便** 避免因 prompt 相同导致的误匹配,确保正确的任务被并发执行

**验收标准**:
- 队列项携带 `subTaskId` 字段
- 匹配逻辑使用 ID 而非 prompt 字符串
- 即使两个任务的 prompt 完全相同,也能正确区分和匹配
- 向后兼容:没有 `subTaskId` 的旧队列项仍能正常执行(回退到 prompt 匹配)

---

### 3.2 富文本交付报告

**作为** 用户  
**我希望** 收到格式丰富的任务交付报告  
**以便** 更直观地查看任务完成情况、统计数据和详细信息

**验收标准**:
- 支持 HTML 格式的交付报告
- HTML 报告包含:
  - 彩色的状态标识(成功/失败/进行中)
  - 可折叠的任务详情
  - 图表化的统计数据(成功率、耗时等)
  - 超链接(如果支持,链接到任务文件)
- 根据频道能力自动选择格式:
  - Telegram: 支持 HTML 格式
  - Web: 支持完整 HTML + CSS
  - 其他频道: 回退到 Markdown
- 保持 Markdown 格式作为备选方案

---

### 3.3 智能的自适应深度控制

**作为** 系统  
**我希望** 根据任务的实际复杂度智能决定分解深度  
**以便** 避免过度分解简单任务或分解不足复杂任务

**验收标准**:
- 引入任务复杂度评分模型,综合考虑:
  - Prompt 长度(字符数)
  - 任务类型(写作 vs 编码 vs 数据处理等)
  - 工具依赖数量
  - 历史同类任务的平均深度
  - 预估的任务时长
- 复杂度评分范围: 0-100
- 深度计算公式:
  ```
  maxDepth = 1 + floor(complexityScore / 25)
  最小值: 1
  最大值: 4
  ```
- 支持手动覆盖(用户可以在 enqueue_task 时指定 maxDepth)
- 记录每个任务的复杂度评分,用于后续优化

---

## 4. 非功能需求

### 4.1 性能

- 复杂度评分计算应在 100ms 内完成
- ID 匹配的性能应优于或等于字符串匹配
- HTML 报告生成不应显著增加响应时间(< 200ms)

### 4.2 兼容性

- 所有改进必须向后兼容现有系统
- 旧的队列项(没有 subTaskId)仍能正常执行
- 不支持 HTML 的频道自动回退到 Markdown

### 4.3 可维护性

- 复杂度评分模型应易于调整和优化
- HTML 模板应与业务逻辑分离
- 新增的字段和逻辑应有清晰的文档

---

## 5. 技术约束

### 5.1 数据结构

- `FollowupRun` 接口需要扩展,添加 `subTaskId?: string` 字段
- `SubTask` 接口已有 `id` 字段,无需修改
- 复杂度评分结果存储在 `SubTask.metadata.complexityScore`

### 5.2 依赖

- HTML 报告生成可能需要引入模板引擎(如 `handlebars` 或使用原生模板字符串)
- 复杂度评分可能需要 NLP 库来分析 prompt(可选,初期可用简单规则)

### 5.3 测试

- 每个改进都需要单元测试
- 并行匹配需要集成测试验证正确性
- HTML 报告需要快照测试

---

## 6. 优先级

1. **高优先级**: 精确的并行任务匹配(影响系统正确性)
2. **中优先级**: 智能的自适应深度控制(提升系统智能化)
3. **低优先级**: 富文本交付报告(提升用户体验,但不影响核心功能)

---

## 7. 风险与缓解

### 7.1 风险

- **向后兼容性风险**: 修改 `FollowupRun` 接口可能影响现有代码
- **复杂度评分准确性**: 初期模型可能不够准确
- **HTML 注入风险**: 生成 HTML 报告时需要防止 XSS 攻击

### 7.2 缓解措施

- 所有新字段使用可选类型(`?`)
- 复杂度模型设计为可配置和可迭代
- 使用模板引擎自动转义 HTML,或手动转义所有用户输入

---

## 8. 成功指标

- 并行任务匹配准确率达到 100%(无误匹配)
- 复杂度评分与实际任务深度的相关性 > 0.7
- HTML 报告在支持的频道上的采用率 > 80%
- 用户对交付报告的满意度提升(通过反馈收集)
